Rewind: Course-Correct Without Starting Over

Every prompt you send checkpoints your session. The rewind menu turns those checkpoints into surgical undo — conversation, code, or both — which changes how boldly you can let an agent work.

M11 · Steering · 2 of 2 Bloom: Apply Assessed by: Assignment A2

Objective: Use plan mode to scope work before execution and the rewind menu to course-correct a session. (Lesson 1: plan mode. This lesson: rewind.)

1 · Checkpoints: the safety net you already have

You don't set checkpoints up — from the Checkpointing docs: “checkpointing automatically captures the state of your code before each user prompt. This safety net lets you pursue ambitious, wide-scale tasks knowing you can always return to a prior code state.” Every prompt creates one; snapshots are kept for the 100 most recent checkpoints; and they're saved with the conversation, so a session you resume tomorrow can still rewind.

To open the menu: run /rewind, or press Esc Esc when the prompt input is empty (if you've typed something, double-Esc clears the text instead — press ↑ to recall it). The menu lists each prompt you sent; select one, then choose what to do from there.

2 · The menu: two restores, two summarizes

OptionWhat it doesReach for it when…
Restore code and conversationReverts both to that pointThe whole direction since then was wrong
Restore conversationRewinds messages, keeps current codeThe code is fine but the discussion went sideways
Restore codeReverts files, keeps the conversationKeep everything learned, discard the edits it produced
Summarize from hereCompresses that message onward into a summaryDiscard a verbose side-quest, keep early context intact
Summarize up to hereCompresses everything before that messageSquash early setup, keep recent work in full detail

The restore/summarize distinction is the one to internalize: restore reverts state; summarize compresses history without touching files on disk. The summarize pair is targeted compaction — /compact aimed at the half of the conversation you choose, with optional inline instructions to steer what the summary keeps. After a conversation restore, your original prompt from that point is placed back in the input field, ready to edit and re-send — which is precisely the M08 move: the failed attempt polluted the context; rewinding un-pollutes it and hands you the prompt to improve.

Why this changes your behavior

From the best-practices guide: “Instead of carefully planning every move, you can tell Claude to try something risky. If it doesn't work, rewind and try a different approach.” Checkpoints convert failed experiments from damage into information. With plan mode scoping the work up front and rewind un-doing what goes wrong anyway, you have this objective's full steering toolkit.

3 · What rewind does not cover

The limitations are as examinable as the features — and they matter for trusting the tool correctly:

  • Bash-command changes aren't tracked. Only edits made through Claude's file-editing tools are captured. If Claude ran rm file.txt or mv old.txt new.txt in the shell, rewind cannot undo it.
  • External changes aren't tracked — edits you made in your editor, or another concurrent session's work.
  • It's not version control. The docs' framing: “Think of checkpoints as ‘local undo’ and Git as ‘permanent history.’” Checkpoints expire with session cleanup (30 days by default); commits don't. Week 1's commit discipline is not replaced by any of this.

One recovery detail worth knowing before A2's experiments: if you ran /clear earlier in the same Claude Code process, the rewind menu's top entry lets you resume the pre-clear session — so even a hasty clear is survivable.

Hands-on checklist

Run this drill in your Week 1 repo — deliberately break things so the recovery is practiced, not theoretical (A2 asks you to practice rewind):

Self-check

1. Claude's last three turns produced a refactor you dislike, but the debugging insights from earlier in the session are valuable. You want the code back as it was and to keep the conversation's knowledge. Which option?

B. Restore code reverts the files while the conversation — including everything learned — stays. A discards the valuable discussion; C is backwards (keeps the unwanted code, loses the talk); D destroys exactly what you wanted to keep and re-typing it is lossy.

2. During the session, Claude ran mv config.yaml config.old.yaml in Bash and also edited main.py with its editing tools. You restore code to an earlier checkpoint. What state are you in?

C. This is the limitation in action: checkpoints track file-editing-tool changes only, so the tool-made edit reverts and the shell-made rename survives. Knowing this asymmetry is what keeps rewind from giving you false confidence — for shell-side damage, git is the answer.

3. Early in a long session you did messy environment setup; the recent thirty turns are focused, valuable implementation. Context is filling. The targeted move is:

A. “Up to here” compresses everything before the selected point — squashing the setup noise while the recent work stays verbatim, exactly matching the docs' “compress early setup discussion” use case. B compresses the wrong side (the valuable work); C deletes the recent thirty turns entirely; D overapplies M08 — mid-task with rich unexternalized state is the acknowledged good case for targeted summarization.

4. Why doesn't checkpointing replace git in your workflow?

D. “Local undo” vs. “permanent history” is the docs' own framing: checkpoints expire with session cleanup, miss bash and external changes, and live only on your machine. B is false (they're automatic and instant), C inverts how they work — automatic capture before every prompt is the entire point.
