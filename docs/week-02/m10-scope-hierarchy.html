CLAUDE.md Scopes — and When to Use a Skill Instead

Instructions can live at four levels, load in a defined order, and can be scoped to paths — or moved out of the always-loaded tier entirely. This is the map of where a given instruction belongs.

M10 · CLAUDE.md Bloom: Understand Assessed by: Quiz 2

Objective: Describe the CLAUDE.md scope hierarchy (user, project, path-scoped) and when to move content into a skill instead.

1 · The four scopes

The Memory docs define four locations, each answering a different question — who should this instruction apply to?

ScopeLocationPurposeShared with
Managed policySystem path (e.g. C:\Program Files\ClaudeCode\CLAUDE.md on Windows)Organization-wide standards, deployed by ITAll users on the machine
User~/.claude/CLAUDE.mdYour personal preferences, every projectJust you (all projects)
Project./CLAUDE.md or ./.claude/CLAUDE.mdTeam conventions: build commands, architecture, standardsTeam, via version control
Local./CLAUDE.local.md (gitignored)Personal project-specific notes: sandbox URLs, test dataJust you (this project)

The sorting rule falls out of the “shared with” column: a preference that follows you everywhere (“explain before large refactors”) is user scope; a fact about the project that a teammate's Claude should also know is project scope and belongs in git; a fact about the project true only for your machine is local scope. Your Week 1 /init file was project scope — the one graders and teammates get for free.

2 · How files load: concatenate, never override

All discovered files are concatenated into context rather than overriding each other — every applicable scope is present at once, ordered from broadest to most specific, so project instructions appear after user instructions, and within a directory CLAUDE.local.md is appended after CLAUDE.md. Claude Code finds them by walking up the directory tree from where you launched: run in foo/bar/, and foo/CLAUDE.md loads too — the monorepo case. If two files give conflicting guidance, nothing arbitrates; “Claude may pick one arbitrarily,” so consistency across scopes is on you.

One more direction matters: files in subdirectories below your working directory are not loaded at launch — they're pulled in on demand when Claude reads files in those directories. Which is the transition to the real insight of this objective:

3 · Path-scoped rules: pay only when relevant

Everything in section 1 loads into every session, paying the rent M09 warned about. The first escape is .claude/rules/: split instructions into topic files (testing.md, api-design.md), and give a rule YAML frontmatter naming the paths it governs:

---
paths:
  - "src/api/**/*.ts"
---
# API Development Rules
- All API endpoints must include input validation

A rule with a paths field loads only when Claude reads a matching file — an API rule costs nothing in a session that never touches the API. (Rules without paths load at launch, same as CLAUDE.md; user-level rules in ~/.claude/rules/ load before project rules.) This is the third tier the objective names: user-wide, project-wide, path-scoped.

4 · When to move content into a skill

Path-scoping helps when relevance is where; skills help when relevance is what you're doing. The Memory docs draw the line in one sentence: “If an entry is a multi-step procedure or only matters for one part of the codebase, move it to a skill or a path-scoped rule instead.” And the rules docs add the load-behavior contrast: rules load every session or when matching files are opened; skills “only load when you invoke them or when Claude determines they're relevant to your prompt.”

The content is…It belongs in…Because it loads…
A fact/rule needed in every session (“use pnpm”)CLAUDE.md (right scope)Always — and should
A rule for one region of the codebasePath-scoped ruleWhen matching files are read
A multi-step procedure used sometimes (release process, review checklist)SkillOn demand, when invoked or relevant
A step that must run every time, enforcedHookNot context at all — it executes as code

The hook row completes the picture: CLAUDE.md, rules, and skills are all advisory context; when “usually follows it” isn't good enough — a check that must gate every commit — the docs point you to hooks, which run deterministically regardless of what Claude decides. Week 7 has you write skills; Week 8 covers hooks. Today you only need to route content correctly.

Also in the memory family

Auto memory is the complement to everything above: notes Claude writes itself — build commands it learned, corrections you made — stored per-repository and loaded each session (the first 200 lines or 25KB of its MEMORY.md index). You write CLAUDE.md; Claude writes auto memory. Browse or edit both with /memory, and verify what actually loaded with /context.

Self-check

1. “Always explain your changes before large refactors” — you want this in every project you ever open, but your teammates shouldn't inherit it. Where does it go?

B. “Just you + all projects” is the user-scope cell of the table. A shares it with every teammate via git; C works but must be recreated in every repo forever; D reinvents always-on loading with extra machinery — a rule matching everything is just a costlier CLAUDE.md.

2. Your project CLAUDE.md has grown a 30-line section of rules that only apply to files under src/api/. The best refactor is:

C. “Only matters for one part of the codebase” is the literal trigger for path-scoping: full adherence benefit in API sessions, zero token rent everywhere else. A pays for importance with dilution (M09's bloat problem); B changes who gets the rules (your teammates lose them) rather than when they load; D misroutes — these are rules about a region, not a procedure you perform.

3. A 25-step “how we cut a release” runbook is sitting in CLAUDE.md. It's used about twice a month. Per the docs' dividing line, it should become:

D. “If an entry is a multi-step procedure… move it to a skill.” Twenty-five steps loaded into every session to be used twice a month is the exact waste skills eliminate. A widens the audience without fixing the cost; B hides a team procedure from the team; C is a hack — the procedure isn't about editing that file, and it wouldn't load when you actually need it.

4. Your user CLAUDE.md says “prefer verbose commit messages”; the project CLAUDE.md says “one-line commit messages only.” What actually happens?

A. Loading is concatenation, not resolution — with contradictory instructions “Claude may pick one arbitrarily.” Broadest-to-most-specific ordering means the project line appears later, which helps in practice but is not an override mechanism (B) — the fix is to remove the contradiction. No error exists (C), and D gets the ordering backwards.

5. “Every commit must pass lint — no exceptions, ever, even if Claude decides otherwise.” Why is CLAUDE.md the wrong home for the enforcement of this rule?

C. The docs draw this line explicitly: CLAUDE.md is context Claude “treats as context, not enforced configuration” — “to block an action regardless of what Claude decides, use a PreToolUse hook instead.” Stating the norm in CLAUDE.md is fine; guaranteeing it is a hook's job. A and B are invented mechanics, and D claims an enforcement layer that doesn't exist.
