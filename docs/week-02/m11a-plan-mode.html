Plan Mode: Scope Before You Execute

A mode where Claude can read everything and change nothing — so the two of you agree on the approach before a single file is edited. The cheapest place to catch a misunderstanding is before the code exists.

M11 · Steering · 1 of 2 Bloom: Apply Assessed by: Assignment A2

Objective: Use plan mode to scope work before execution and the rewind menu to course-correct a session. (This lesson: plan mode. Lesson 2: rewind.)

1 · What plan mode is

The official definition: “Plan mode tells Claude to research and propose changes without making them. Claude reads files, runs shell commands to explore, and writes a plan, but does not edit your source.” It's one of Claude Code's permission modes — a sibling of the default ask-before-acting mode you've used since Week 1 — with edits structurally blocked until you approve a plan.

Getting in and out:

Cycle modes in a sessionShift+Tab — default → accept-edits → plan; the status bar shows ⏸ plan mode on

Start a session in plan modeclaude --permission-mode plan
Leave without approving anythingShift+Tab again

2 · Why: the misunderstanding tax

Anthropic's best-practices guide names the risk plan mode exists to remove: “Letting Claude jump straight to coding can produce code that solves the wrong problem.” Its recommended shape for non-trivial work is four phases — Explore → Plan → Implement → Commit — with the first two inside plan mode: have Claude read the relevant code and answer questions, then ask for a concrete plan (“What files need to change? What's the session flow? Create a plan.”).

Pocock's introduction to plan mode adds the human half of the argument: “You don't know what you want until you see it.” Writing no code forces the conversation you'd otherwise have through code — requirements get articulated, wrong assumptions surface, and you veto them while a veto costs nothing. Meanwhile the agent is quietly doing something equally valuable: building a context window full of the right files before any code gets written. Which is why his workflow note matters: “You usually don't clear the context window between planning and execution” — the plan's research is precisely the context you want the implementation to run on. Plan-then-execute is one session, not two; it's the memento principle's complement, not a repeat of it.

Steering, not paperwork

Connect this to Week 1's M05: planning decisions are the ones humans dominate. Plan mode is that division of labor built into the tool — it holds the “what and how, roughly” conversation in a place where you still have full veto power, then hands execution to the agent with the decisions already made.

3 · Reviewing and approving a plan

When Claude's plan is ready, it presents it and asks how to proceed. Your options, per the docs:

  • Approve and let it run — “Yes, auto-accept edits” (or auto mode where available): Claude implements without per-edit prompts. Use when the plan is precise enough that you'd approve each edit anyway.
  • Approve but watch — “Yes, manually approve edits”: implementation proceeds, you still confirm each change. Good default while you're calibrating trust in Week 2.
  • Keep planning — “No, keep planning”: stay in plan mode and tell Claude what to change. Iterating here is the point of the mode — a plan revision costs sentences; a code revision costs a rewind.
  • Edit the plan yourself — press Ctrl+G to open the plan in your text editor and change it directly before Claude proceeds.

Approving exits plan mode and implementation starts. To plan the next chunk, cycle back with Shift+Tab or prefix with /plan.

4 · When to plan — and when to skip it

Plan mode is not a virtue ritual; the official guidance is candid that it “adds overhead.” The dividing line:

Plan firstSkip the plan
You're uncertain about the approachScope is clear and the fix is small
The change touches multiple filesA typo fix, a log line, a rename
You're unfamiliar with the code being modifiedYou've made this exact kind of change before
The one-sentence test

From the best-practices guide: “If you could describe the diff in one sentence, skip the plan.” Cheap, memorable, and it will be on the quiz in application form.

Hands-on checklist

In your Week 1 repo, run one full plan-first cycle (A2 requires demonstrating this):

Self-check

1. In plan mode, Claude can ______ but cannot ______.

C. Research and propose, don't modify: reads and exploration are the mode's whole job; edits stay blocked until you approve the plan. A inverts it, B mistakes restriction for paralysis (exploration is active work), and D invents a new-vs-existing distinction that doesn't exist.

2. Which task most clearly deserves plan mode first?

B. It scores on all three plan-first criteria: uncertain approach, many files, unfamiliar code — and you could not describe its diff in one sentence. A, C, and D each fit in a sentence; planning them is pure overhead.

3. Claude presents a plan that's right except it targets the wrong config file. The best move is:

D. A plan revision costs a sentence — that's the entire economics of the mode; approving a known-wrong plan (A, C) voluntarily pays the misunderstanding tax the mode exists to avoid, and B throws away all the exploration context over a one-line fix.

4. Why does Pocock advise against clearing context between an approved plan and its execution?

A. Plan-then-execute is one continuous unit of work: the exploration is the well-contexted state you want at execution time, so clearing between them would throw away signal, not pollution. No command is disabled (B); the conversation isn't disk (C); and D misreads the principle — memento says clear between units of work after externalizing state, which is fully consistent with not clearing inside one.
