Agentic Coding with Claude Week 2 · Objective M07

The Smart Zone and the Dumb Zone

Why the same model gets visibly dumber as a session grows, what that degradation looks like from your chair, and how to judge when a session has stopped paying for itself.

M07 · Context Bloom: Understand Assessed by: Quiz 2

Objective: Explain why context occupancy degrades model output (“smart zone” vs. “dumb zone”) and estimate when a session should be reset.

1 · What actually fills the context window

Everything Claude knows about your session lives in one place: the context window — in Matt Pocock's dictionary definition, “everything the model sees on each model provider request. Finite, model-specific, and the only surface through which the model perceives anything.” There is no side channel. If it isn't in the window, Claude doesn't know it; if it is, Claude is paying attention to it whether it helps or not.

And far more enters the window than you see in your terminal. Before you type your first prompt, the system prompt, your CLAUDE.md files, auto memory, and the list of available skills and tools have already loaded. Then, as Claude works, the docs' interactive walkthrough shows where the real weight comes from: file reads dominate context usage. A single file read can cost more tokens than your entire prompt; every command output, every diff, and every response accumulates on top. Anthropic's best-practices guide is direct about the scale: “A single debugging session or codebase exploration might generate and consume tens of thousands of tokens.”

See it live

Run /context in any session for a live breakdown of your window by category — how much is system overhead, memory files, messages, and free space. It's the instrument panel for everything in this lesson.

2 · Occupancy degrades output — the mechanism

The core fact this week is built on, from the best-practices guide: “LLM performance degrades as context fills. When the context window is getting full, Claude may start ‘forgetting’ earlier instructions or making more mistakes.” The window doesn't fail like a disk filling up — nothing errors. Output quality just slides.

Pocock's AI Coding Dictionary names the mechanism attention degradation: “As a session grows, each token's attention budget is spread across more competitors.” The attention the model can devote to any one relationship — an instruction you gave, a constraint in CLAUDE.md, the actual bug — shrinks as thousands of tokens of stale file reads and old command output compete for it. Signal thins; noise crowds in.

Hence the two zones, in the dictionary's words:

  • Smart zone: “Early in a session the agent is in a ‘smart zone’ — sharp, focused, recall is good.”
  • Dumb zone: “As the session grows it drifts into a ‘dumb zone’: sloppier, forgetful, more mistakes — and more faithfulness hallucinations.”

A faithfulness hallucination is the nastiest symptom: the model confidently asserts something its own context contradicts — quoting a function signature that the file it read an hour of tokens ago plainly doesn't have. The information is in the window; there's just not enough attention left to retrieve it faithfully.

3 · Where's the boundary?

Nobody can give you an exact line, and Pocock doesn't try — his materials put the onset of the dumb zone well before the advertised context limit (figures around 125K–150K tokens on frontier models circulate in his workshop and dictionary discussions), while noting that where the boundary sits is debated; that it exists is not. The practical takeaway does not depend on the number:

The rule that matters

The advertised window is not the useful window. A model that accepts 200K tokens does not think equally well at token 180,000 as at token 20,000. Plan your sessions around the smart zone, not around the spec sheet.

This is also why “more context always helps” — the beginner instinct — is backwards. Irrelevant context isn't neutral filler; it actively competes for attention with the context that matters. The best-practices guide: long sessions full of irrelevant content “can reduce performance and sometimes distract Claude.”

4 · Estimating when to reset

“Estimate when a session should be reset” is the second half of this objective. You have two kinds of evidence: the gauge and the symptoms.

SignalWhat it looks likeVerdict
Task boundaryYou finished one task and are about to start an unrelated oneReset — always, regardless of occupancy
Two failed correctionsYou've corrected the same mistake twice and it's still wrongReset — the guide's explicit rule: after two failed corrections, /clear and write a better prompt incorporating what you learned
ForgettingClaude drops an instruction you gave earlier, or re-makes a mistake it already fixedReset — you're watching attention degradation happen
Confident wrongnessClaude asserts things the session's own files contradictReset — faithfulness hallucination, deep dumb zone
The gauge/context shows the window largely consumed by old explorationReset before starting anything new that matters

Notice that the strongest signals are behavioral, not numerical. The best-practices guide's summary of the judgment: “A clean session with a better prompt almost always outperforms a long session with accumulated corrections.” The kitchen-sink session — task, unrelated question, back to the task — is the canonical way students wander into the dumb zone without noticing.

Where this week goes next

Knowing that you should reset raises the practical question: reset how, without losing the state you care about? That's M08, the memento principle — and M12 has you generate the evidence for all of this yourself.

Self-check

1. Ninety minutes into a debugging session, Claude confidently describes a helper function's arguments — but the file it read earlier in the session shows a different signature. What is the most likely explanation?

C. This is a faithfulness hallucination, the signature symptom of the dumb zone: the information is present in the window, but each token's attention budget is spread across so many competitors that retrieval fails. A and D are possible in other scenarios, but the question stipulates the file was read this session; B misunderstands where session knowledge comes from — the context window, not training.

2. A classmate says: “My model supports 200K tokens, so I keep sessions going until the window is nearly full — otherwise I'm wasting capacity.” What's wrong with this reasoning?

B. Performance slides long before capacity runs out — the dumb zone begins far below the advertised limit, and exactly where is debated but its existence isn't. C is invented; D is backwards — file reads and tool output, not your prompts, dominate the window.

3. You fixed a failing test, then spent twenty minutes asking Claude unrelated questions about Docker, and now want to start renaming a module. The session feels fine so far. Per this lesson, you should:

D. The reset-at-task-boundaries rule doesn't wait for symptoms — irrelevant context (the test fix, the Docker chat) competes for attention with the renaming task and can only hurt it. This is the kitchen-sink session forming in real time. C doesn't work: a degraded model is not a reliable reporter of its own degradation.

4. Why does a big irrelevant file read make output worse, rather than just using up spare room?

A. That's attention degradation: “each token's attention budget is spread across more competitors.” Occupancy itself carries a quality cost, which is why B — the “it's free until it's full” model — is the specific misconception this objective exists to remove. Weights never change during a session (C), and D is invented.

5. You've told Claude twice that the project uses pnpm, not npm, and it just ran npm install again. The best next move is:

B. Two failed corrections is the guide's explicit reset threshold: the context is polluted with failed approaches, and a clean session with the constraint stated up front “almost always outperforms” a third correction shouted into a noisy window. A bigger window (C) just gives you a longer runway to the same dumb zone. (A recurring correction is also a textbook candidate for CLAUDE.md — that's M09.)
