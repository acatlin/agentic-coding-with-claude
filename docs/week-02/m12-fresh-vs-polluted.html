Agentic Coding with Claude Week 2 · Objective M12

The Session Experiment: Fresh vs. Polluted

Everything this week has claimed that context quality shapes output quality. Now you stop taking the course's word for it: run the same task twice — once in the smart zone, once out of it — and weigh the evidence yourself.

M12 · Evidence Bloom: Evaluate Assessed by: Assignment A2

Objective: Run the same task in a well-contexted fresh session vs. a polluted long session and document the observed quality difference.

1 · The claim under test

Stated falsifiably: holding the task, prompt, and starting code constant, a fresh session seeded with good externalized state will produce observably better output than a long session whose window is full of unrelated material. The mechanism (M07: attention degradation) predicts the polluted run will show more forgotten constraints, more corrections needed, and more confident errors. Your job is to observe honestly — including the possibility that the difference is smaller than the course implies. That, too, is a finding worth documenting.

2 · Designing a fair comparison

An Evaluate objective is graded on the quality of your judgment, and judgment starts with method. Three variables to control:

  1. Same task, same prompt. Write the prompt once, in a file, and paste it verbatim into both runs. A “moderate” task is one that touches 2–3 files and has checkable results — e.g. “add a --top N option to wordcount that prints the N most frequent words, with tests” in your Week 1 repo.
  2. Same starting code. Run A from a clean commit; then git reset --hard (or a fresh branch from the same commit) before run B. Otherwise run B is building on run A's output and the comparison is void.
  3. Same model and settings. Change only the variable under test: the state of the context window.

Then set up the two conditions:

Run A — fresh, well-contextedRun B — polluted, long
Setup/clear, your lean CLAUDE.md (M09) in place, paste the promptAt the end of a long working session (see below), paste the same prompt without clearing
Window stateNear-empty; only durable state loadedTens of thousands of tokens of unrelated history
CheckRun /context before each run and record the occupancy — these two numbers anchor your write-up
Pollute honestly

The polluted session must be realistic, not sabotage. Recreate the best-practices guide's kitchen-sink pattern with real work: explore a few large files for an unrelated question, debug something with a couple of failed approaches, ask a few off-topic questions. Do not feed Claude false information or instructions to do badly — that tests gullibility, not context degradation, and your write-up should say how you polluted so a reader can judge the fairness themselves.

3 · What to measure

Decide the rubric before run A, so the outcome can't tempt you into scoring what favors the story:

  • Correctness: does the feature work? Do the tests pass — and actually test the behavior?
  • Adherence: did it follow CLAUDE.md conventions and every constraint in the prompt? Count violations.
  • Corrections needed: how many times did you have to redirect? (Week 1's M05 transcript-reading skill is how you count.)
  • Memory failures: forgotten constraints, re-made mistakes, assertions contradicted by files it read — M07's symptom list, tallied.
  • Turns/time to done, as a secondary signal.

4 · Documenting the difference

A2 asks for a half-page comparison. The structure graders look for: setup (task, how B was polluted, occupancy numbers from /context) → observations (the rubric, scored, with at least one verbatim example per claim) → verdict (what the evidence supports, stated no more strongly than the evidence allows). The most common failure is a write-up that narrates the doctrine instead of the data — asserting “the polluted session was worse, as expected” with nothing a reader could check. One quoted exchange where Claude forgets a constraint outweighs a paragraph of vibes.

5 · Judge two write-ups

Before writing yours, practice the evaluation on someone else's. Two condensed A2 excerpts, same assignment:

Write-up 1 — “Jordan”

“The fresh session was way better, just like the readings said it would be. The polluted session felt slower and dumber and made more mistakes. This confirms the smart zone/dumb zone theory. Context management clearly matters and I will use /clear more.”

Write-up 2 — “Sam”

“Both runs produced a working --top flag. Fresh run (18% occupancy): 2 turns, one correction (misread my tie-breaking rule). Polluted run (81% after ~50 turns of unrelated debugging): 5 turns, three corrections — it named the flag --limit despite the prompt, used unittest although CLAUDE.md and the existing suite use pytest, and claimed ‘all tests pass’ when 2 failed; pasting the failure output fixed it. Difference was smaller than I expected on correctness (both got there) but large on adherence and supervision cost. Caveat: n=1, and my pollution recipe (mostly debugging another file) may matter.”

Which write-up better meets the objective, and why?

C — and the reason matters. Jordan's is doctrine-narration: every sentence could have been written before the experiment, and none of it can be checked. Sam's earns its verdict — numbers a reader can weigh, one example per claim (naming, convention, false success), an honest surprise (correctness gap smaller than expected), and a stated limitation. B praises Sam's for the wrong reason: the write-up's value is its method, not its direction — Sam's would be equally good if the runs had tied. D overcorrects: n=1 can't prove the general claim, but the objective says document the observed difference, and a well-controlled anecdote with stated limits is exactly what good engineering judgment produces from one trial.

Self-check

1. A student runs the fresh session first, commits its output, then runs the polluted session on top of that commit. What's broken?

B. Run B now faces a different (mostly solved) task — any result is uninterpretable. The fix is resetting to the same commit before each run. Order (D) doesn't matter as long as the start state is identical; commits (C) are fine, and useful, as reset points.

2. To make the polluted condition “really show the effect,” a student ends the long session by telling Claude several false facts about the repo before pasting the task prompt. Why does this invalidate the experiment?

D. The claim is about occupancy and irrelevance degrading attention, and the condition must resemble how sessions actually bloat — real detours, real debugging, real off-topic questions. Deliberate misinformation would degrade a fresh session too, so observing failure tells you nothing about context occupancy. An experiment's pollution recipe should be disclosed precisely so readers can catch this.

3. Your two runs come out nearly identical in quality. The best response in your A2 write-up is:

A. The objective grades your documentation and judgment, not the direction of the outcome — and a null result honestly analyzed is stronger evaluative work than a confirmation vaguely narrated. B is running the experiment until it agrees with you (mention the extra run if you do one — as a second data point, not a replacement); C overclaims off n=1 exactly as badly as Jordan's write-up does in the other direction.

4. Which detail does the fresh condition require — not just an empty window?

C. The experiment tests the whole Week 2 stack: fresh means deterministic empty conversation + durable state you wrote (M08 + M09) — an empty window with no CLAUDE.md would handicap run A and muddy the comparison. B produces a summary, not a fresh state; D changes a controlled variable and voids the whole design.
