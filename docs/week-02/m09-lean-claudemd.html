Agentic Coding with Claude Week 2 · Objective M09

Writing a Lean CLAUDE.md

The most important file you can create — and the one most students ruin by overfilling. This lesson teaches the craft: what earns a line, what to cut, and how to defend every exclusion.

M09 · CLAUDE.md Bloom: Create Assessed by: Assignment A2

Objective: Write a lean CLAUDE.md (build commands, conventions, layout, non-negotiable rules) and justify what was deliberately left out.

1 · What CLAUDE.md is — and what it costs

CLAUDE.md is a plain markdown file Claude reads at the start of every session: persistent instructions that survive every /clear. In memento-principle terms (M08), it's your most durable externalized state — the conventions layer of your project's memory.

But it isn't configuration, and it isn't free. The Memory docs are explicit on both points: CLAUDE.md is context, “not enforced configuration” — Claude reads it and tries to follow it — and it's “loaded into the context window at the start of every session, consuming tokens alongside your conversation.” Every line you write occupies smart-zone budget in every session forever, whether or not that session needs it. That's why this objective says lean: the file's cost structure, not aesthetics, demands it.

The failure mode

From the best-practices guide: “Bloated CLAUDE.md files cause Claude to ignore your actual instructions!” Past a point, adding rules makes all rules less followed — important ones get lost in the noise. The docs target under 200 lines per file; A2 holds you to under 100. Length is not thoroughness; it's dilution.

2 · What earns a line

The Memory docs give a precise trigger list. Add to CLAUDE.md when:

  • Claude makes the same mistake a second time
  • A code review catches something Claude should have known about this codebase
  • You type the same correction into chat that you typed last session
  • A new teammate would need the same context to be productive

Notice what these have in common: each line is a proven, recurring need — CLAUDE.md is grown from evidence, not drafted from imagination. The categories that recur, matching this objective's own list:

✅ Include❌ Exclude
Build/test commands Claude can't guess (pnpm test:unit, required env vars)Anything Claude can figure out by reading code
Code style rules that differ from defaultsStandard language conventions Claude already knows
Project layout facts that aren't obvious (“API handlers live in src/api/handlers/”)File-by-file descriptions of the codebase
Repository etiquette (branch naming, commit format)Information that changes frequently
Non-negotiable rules and gotchas (“never edit generated/ by hand”)Self-evident practices like “write clean code”
Architectural decisions specific to this projectLong explanations, tutorials, API documentation (link instead)

And write each surviving line concretely enough to verify — the docs' own examples: “Use 2-space indentation” not “Format code properly”; “Run npm test before committing” not “Test your changes.” A vague rule costs the same tokens as a precise one and buys you nothing.

3 · The pruning test — and justifying exclusions

The single sharpest tool in the official guidance is a question. For each line, ask:

The test

“Would removing this cause Claude to make mistakes?” If not, cut it.

A2 makes the cutting visible: you must note, in comments, two things you deliberately excluded and why. That's the Create-level skill this objective assesses — a lean file is defined by its refusals. A good exclusion note names the candidate, the reason it fails the test, and where the content lives instead if it lives anywhere. (HTML comments <!-- like this --> are ideal: the docs note block-level comments are stripped before injection, so your justifications cost zero tokens.) For example:

<!-- Excluded: description of the src/ directory tree.
     Claude discovers layout by reading; a prose copy would
     go stale and pay rent in every session. -->
<!-- Excluded: "how to add a new command" walkthrough.
     Multi-step procedure needed only sometimes — belongs in
     a skill loaded on demand, not in every session. -->

That second note points at a real boundary: recurring procedures belong in skills, and rules needed only for part of the codebase belong in path-scoped rules — M10 maps that whole territory.

4 · Workflow: generate, then treat it like code

Don't start from a blank page. /init analyzes your codebase and generates a starter file (build commands, test instructions, discovered conventions) — you saw this in Week 1. The value you add is what /init can't discover: the gotchas, the decisions, the non-negotiables. From there, the docs' maintenance stance is: “Treat CLAUDE.md like code: review it when things go wrong, prune it regularly, and test changes by observing whether Claude's behavior actually shifts.”

Two debugging rules of thumb from the Memory docs: if Claude keeps violating a rule the file contains, the file is probably too long and the rule is getting lost; if Claude asks questions the file answers, the phrasing is probably ambiguous. Either way the fix is usually deletion or sharpening — almost never addition. Check what actually loaded with /context, and open the file quickly with /memory.

Self-check

1. Which candidate line best earns its place in a lean CLAUDE.md?

C. It passes the pruning test — nothing in the code reveals that hidden environmental dependency, so removing the line would cause mistakes (failing tests with confusing errors). A is a self-evident practice, B is discoverable in one glance at the repo, and D is the classic bloat pattern: layout Claude learns faster by reading than by being told, going stale from day one.

2. Claude keeps using npm despite your CLAUDE.md rule “use pnpm” — the rule is in a 240-line file. Per the docs, the most likely diagnosis is:

B. “If Claude keeps doing something you don't want despite having a rule against it, the file is probably too long and the rule is getting lost.” The counterintuitive fix is deletion — of other lines. A is half-true (it's context, not enforcement) but fatalistic: adherence responds strongly to length and specificity. C adds noise. D overreaches — though for a truly must-never-fail rule, a hook is the enforcement-grade tool, which is exactly why CLAUDE.md keeps only rules where advisory is enough.

3. A2 requires justifying two deliberate exclusions. Which justification shows the intended reasoning?

D. It names the candidate, applies the pruning test (removal causes no mistakes — the source of truth is readable), and adds the cost argument (staleness + per-session rent). A excludes something that usually passes the test — build/test commands are the docs' first-listed inclusion; “usually guesses” is exactly the recurring-mistake risk the file exists to prevent. B doesn't reason at all, and C treats the line budget as the goal rather than a symptom of good selection.

4. Why does the guidance say to phrase rules like “Run npm test before committing” rather than “Test your changes”?

A. CLAUDE.md is context, not configuration — how you write it is the only lever you have over how reliably it's followed, and “specific, concise, well-structured instructions work best.” There is no parser, no sentence filter, and no exemption: the file is plain markdown, all of it loaded, all of it paying rent.
