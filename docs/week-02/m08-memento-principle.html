Agentic Coding with Claude Week 2 · Objective M08

The Memento Principle: /clear over /compact

Two ways out of a bloated session: summarize it, or write down what matters and start empty. This lesson makes the case for the second — and teaches you to articulate exactly what the first costs you.

M08 · Context Bloom: Apply Assessed by: Quiz 2 · Assignment A2

Objective: Apply the memento principle: prefer /clear plus externalized state over /compact, and articulate the tradeoff between a deterministic empty state and a noisy summary.

1 · The two escape hatches

M07 established that sessions must be reset. Claude Code gives you two very different resets:

  • /clear — ends the conversation history entirely. In the dictionary's words: “The next message begins with an empty session and an empty context window.” (Your CLAUDE.md and auto memory still load — those live on disk, not in the conversation.)
  • /compact — replaces the conversation with an AI-written summary. The dictionary is precise about what this really is: “a handoff done in-memory: the previous session's history is summarised, and the summary seeds a fresh session.” Claude Code also runs this automatically as you approach the context limit, so a full window never ends your session.

Both free space. The difference is what the next session starts from — and that difference is the whole lesson.

2 · What a summary actually is

When /compact runs, a model reads your conversation and writes a structured summary — your requests, key decisions, files touched, errors fixed, pending work. Everything else is gone: full tool outputs, the exact code Claude read, intermediate reasoning. Three properties follow:

  1. It's lossy. The summary keeps what the summarizer guessed was important. If the crucial detail was an offhand constraint you mentioned once, it may not survive.
  2. It's noisy. Whatever polluted your session — failed approaches, dead ends, the kitchen-sink detours — gets summarized too. A summary of a muddy session is condensed mud. The pollution you were escaping rides along into the “fresh” session.
  3. It's nondeterministic. You didn't write it, you can't fully predict it, and compacting the same session twice wouldn't produce the same text. In the dictionary's terms it is a secondary source — an account of the work, not the work.

Not everything even makes it into the summarizer's hands equally: project-root CLAUDE.md, unscoped rules, and auto memory are re-injected from disk after compaction, but path-scoped rules and nested CLAUDE.md files are lost until a matching file is read again. What survives is an engineering contingency, not a guarantee.

3 · The memento principle

Pocock's alternative — the reason this course prefers /clear — is named for the film Memento, whose protagonist cannot form new memories and survives by externalizing them: notes, photographs, tattoos. He doesn't trust his fading recollection; he trusts what he wrote down while he still knew.

Your agent is in exactly his position. It will remember nothing tomorrow. So don't ask a lossy summarizer to guess what mattered — while the session is still smart, write the state that matters into files, then /clear without fear:

  • Durable conventions → CLAUDE.md (M09)
  • The goal and its constraints → a spec or PRD file (Week 3 builds this into a discipline)
  • The task list and progress → a plan file, or GitHub issues (Week 4)
  • Hard-won discoveries — “the flaky test is timing-dependent, root cause in queue.ts” → a notes file, before you clear, not after

The next session then starts from a deterministic empty state: a fresh window seeded only by files you wrote and can read, edit, and version-control. Anthropic's own best-practices guide teaches the same move for large features: have Claude interview you, write the result to SPEC.md, then “start a fresh session to execute it. The new session has clean context focused entirely on implementation, and you have a written spec to reference.” Externalized state plus a clean window — the memento principle in official clothing.

The habit, in one loop

Work in the smart zone → externalize anything worth keeping (spec, plan, notes, CLAUDE.md) → /clear → re-seed the next session from the files. Repeat. Each session gets the model at its sharpest, and your project's memory lives in git instead of in a summary you never read.

4 · Articulating the tradeoff honestly

A2 asks you to articulate this tradeoff, not recite a slogan — so be fair to /compact:

/clear + externalized state/compact
Next session starts fromAn empty window + files you wroteAn AI-written summary you didn't
Deterministic?Yes — same files, same start, repeatableNo — lossy, nondeterministic, unreviewed by default
Pollution carried forwardNoneSummarized along with everything else
Effort requiredYou must write state down before clearingNone — one command, or automatic
Mid-task continuityLost unless it's in the filesPreserved, approximately

That last row is real: mid-task, with rich unexternalized working state, a directed compact can be the pragmatic move — /compact takes instructions (/compact Focus on the API changes), and the rewind menu's summarize from here / up to here options (M11) compact only part of the conversation. Auto-compact also keeps long autonomous runs alive when nobody is there to clear. The principle isn't “compaction is forbidden” — it's that needing compaction is usually the symptom of state you failed to externalize. The better your memento habit, the rarer your compacts.

Hands-on checklist

In your Week 1 repo, practice the full loop once (this is also A2 preparation):

Self-check

1. Your session is 70% full and muddled after three abandoned approaches to a bug — but along the way you finally found the root cause. The memento move is:

B. Externalize first, then clear: the one finding worth keeping goes into a file you control; the three failed approaches — pure attention pollution — go nowhere. A summarizes the mud along with the gold; C loses everything, because /clear empties the conversation and nothing “remembers” across it except files; D spends your remaining smart-zone budget dragging dead weight.

2. “A deterministic empty state” beats “a noisy summary” because:

D. Determinism is about knowing and controlling the starting state: files you wrote, versioned in git, same result every time. A is true but incidental — it's not the argument. B is false (the official docs recommend directed compaction in several situations), and C isn't the failure mode — summaries are small; the problem is what they lose and what they keep.

3. After an auto-compact, Claude keeps following your project conventions but has forgotten a rule that lives in a nested CLAUDE.md deep in a subdirectory. Why?

A. What survives compaction depends on how it was loaded: project-root CLAUDE.md, unscoped rules, and auto memory come back from disk automatically; nested CLAUDE.md files and path-scoped rules return only when a matching file is next read. Nothing on disk is ever deleted (B). C is false — nested files load on demand — and D isn't how survival is decided.

4. Which situation is the best case for reaching for compaction rather than /clear?

C. That's the honest edge of the tradeoff — mid-task continuity is the one thing /compact buys that /clear doesn't, and a directed compact (“/compact Focus on…”) mitigates the lossiness. A is the textbook /clear case (task boundary, no state worth carrying); B is the case where compaction is worst — it condenses the mud; D wants a file in git, not a summary in a context window.

5. A teammate objects: “/clear throws away work; /compact at least keeps it.” The memento-principle reply is:

B. The objection assumes the conversation is the only home for state — that's the assumption the memento principle removes. Once the goal, plan, and discoveries live in files, the conversation is disposable by design. C overstates (a summary does persist something, just lossily and noisily); D describes resuming a session, which brings back the whole polluted context, not a fix for it.
