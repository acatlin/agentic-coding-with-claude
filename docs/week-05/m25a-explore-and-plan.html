Agentic Coding with Claude Week 5 · Objective M25

Explore and Plan Before You Code

Weeks 3 and 4 got you a spec and a backlog. This week you execute — and the first discipline of execution is refusing to let the agent type code on line one. This lesson covers the first half of the recommended four-phase workflow: have Claude read the relevant code, then produce a plan you approve before a single file changes.

M25 · Execution Bloom: Apply Assessed by: Assignment A5

Objective: Apply the Explore–Plan–Code–Commit workflow: have Claude read relevant code, produce a plan, then implement only after plan approval.

1 · Why not just start coding?

Anthropic's best-practices guide states the failure mode in one line: “Letting Claude jump straight to coding can produce code that solves the wrong problem.” An agent that starts implementing immediately commits to its first interpretation of your request — before it has seen how your codebase actually handles sessions, or state, or errors. The fix is structural, not motivational: separate research and planning from implementation, so the wrong interpretation gets caught while it's still words in a plan, not code in your files.

The recommended workflow has four named phases — Explore, Plan, Code, Commit. This page covers the two phases that happen before any edit; M25b covers the two after.

2 · The mechanism: plan mode

The workflow's first two phases run in plan mode, which you met in Week 2 (M11a). In plan mode, “Claude reads files, runs shell commands to explore, and writes a plan, but does not edit your source.” Edits stay blocked until you approve the plan — the mode is the enforcement, not your vigilance.

Three ways in:

Entering plan mode
claude --permission-mode plan    # start the session in plan mode
Shift+Tab                        # cycle default → acceptEdits → plan mid-session
/plan <prompt>                   # plan mode for a single prompt

The status bar shows ⏸ plan mode on while it's active. Pressing Shift+Tab again leaves plan mode without approving anything.

3 · Phase 1: Explore

In the Explore phase you point Claude at the code the task touches and have it build understanding — reading files and answering questions, changing nothing. The best-practices guide's example, for adding OAuth to an app:

Explore prompt (plan mode)
read /src/auth and understand how we handle sessions and login.
also look at how we manage environment variables for secrets.

Notice what the prompt does: it names the directory, and it names the concerns (sessions, login, secrets) the coming change depends on. You are steering the exploration toward what the plan will need, not asking for a general tour.

Exploration costs context — delegate it

Every file Claude reads lands in the context window you learned to protect in Week 2. For broad exploration, the docs recommend subagents: “use a subagent to investigate how our auth system handles token refresh.” The subagent reads files in its own context window and reports a summary — only the findings occupy yours, leaving the smart zone for planning and implementation.

4 · Phase 2: Plan, then approve

With the exploration done, ask for a concrete plan:

Plan prompt (plan mode)
I want to add Google OAuth. What files need to change?
What's the session flow? Create a plan.

When the plan is ready, Claude presents it and asks how to proceed. This dialog is the workflow's gate — the objective's phrase “implement only after plan approval” lives here. Your options:

  • Yes, and use auto mode (or Yes, auto-accept edits) — approve and let implementation run with few or no per-edit prompts.
  • Yes, manually approve edits — approve the plan but review each edit individually.
  • No, keep planning — stay in plan mode and tell Claude what to change. This is where you push back: a vague step, a file it shouldn't touch, a missing test.
  • No, refine with Ultraplan — send the plan to a browser-based review surface (more on this in M29).

Approving a plan exits plan mode and switches the session to the mode the option describes — the approval is also the mode change. And before approving, you can press Ctrl+G to open the plan in your own text editor and edit it directly; your edited version is what Claude proceeds from.

Treat the plan the way Week 3 taught you to treat a PRD: it's a destination marker you actually read. A plan you approve without reading is a gate that isn't there — and Assignment A5 asks you to report where you intervened, which starts with what you changed at this gate.

5 · When to skip all this

The docs are explicit that planning is overhead, and overhead needs to pay for itself: “For tasks where the scope is clear and the fix is small (like fixing a typo, adding a log line, or renaming a variable) ask Claude to do it directly.” Planning earns its cost “when you're uncertain about the approach, when the change modifies multiple files, or when you're unfamiliar with the code being modified.”

The one-sentence test

“If you could describe the diff in one sentence, skip the plan.” Your Week 4 tracer-bullet issues will mostly fail that test — they cross every layer, which is exactly why A5 requires the plan gate for each one.

Hands-on: run Explore and Plan on a real issue

Do this against one of your A5 issues (don't implement yet — that's M25b):

Self-check

1. A classmate types “add Google OAuth to the app” as their first prompt in default mode and lets Claude run. Twenty minutes later the diff reimplements a session store the codebase already had. Which workflow failure is this?

C. The code may even be internally correct — the problem is it solves the wrong problem, because nothing forced Claude to read the existing session handling first. A and B mislocate the failure downstream of where it happened. D understates it: a wrong-problem diff costs a full redo, which is why the fix is structural (explore, then plan, then a gate) rather than “review harder afterward.”

2. During Explore, you want Claude to survey a large unfamiliar subsystem, but you're worried about the reads flooding your context window. The docs-endorsed move is:

B. This is the subagent pattern the docs recommend precisely for investigation — exploration happens off-context, findings come back small. A is the Week 2 lesson in reverse: raw file contents crowd the smart zone where planning quality lives. C trades context pollution for the wrong-problem failure. D fights the symptom while the cause (reading in-context) continues.

3. Claude presents its plan. Step 4 says “update the relevant components.” You're not sure which components it means. What does the workflow say to do?

D. The plan gate exists to catch wrong interpretations while they're still words; a step you can't verify is one you can't have approved in any meaningful sense. B moves the catch to mid-implementation, where redirecting costs more (M29's topic). A is the gate-that-isn't-there. C throws away good exploration context for no reason — “keep planning” iterates in place.

4. Which task is the best candidate for skipping the Explore–Plan phases entirely?

A. It passes the one-sentence test — you just described the diff. B crosses every layer by construction (that's what a tracer bullet is), C is a multi-file architectural change, and D fails the “unfamiliar code” criterion; all three are the documented cases where planning pays. The point of the exception is calibration: the workflow is a tool with a cost, not a ritual.
