Choose the Right Permission Mode

Every session runs in a permission mode, whether you chose one or not. The mode decides how often Claude pauses to ask — and therefore whether you're reviewing each action as it happens or reviewing the result afterward. This lesson teaches the modes as a dial, and the skill the quiz and A5 actually test: matching the mode to the task and defending the choice.

M26 · Execution Bloom: Apply Assessed by: Quiz 5 · Assignment A5

Objective: Configure permission modes appropriately for a task (default approvals vs. accept-edits vs. plan-only) and justify the choice.

1 · What a permission mode is

When Claude wants to edit a file, run a shell command, or make a network request, it pauses and asks. Permission modes control how often that pause happens. The docs frame the trade-off exactly as this course's Week 1 “whose decision” lesson would predict: “Pick more oversight for sensitive work, or fewer interruptions when you trust the direction.”

The failure mode of maximum oversight is worth naming, because it's the docs naming it: with a prompt for everything, “after the tenth approval you're not really reviewing anymore, you're just clicking through.” A mode that asks too often doesn't give you more control — it gives you the feeling of control while training you to stop reading.

2 · The modes

What each mode lets run without asking, from the permission-modes doc:

ModeWhat runs without askingBest for
default (labeled Manual)Reads onlyGetting started, sensitive work
acceptEditsReads, file edits, and common filesystem commands (mkdir, touch, mv, cp…)Iterating on code you're reviewing
planReads only — and Claude won't edit at all until you approve a planExploring a codebase before changing it
autoEverything, with background safety checks by a classifier modelLong tasks, reducing prompt fatigue
dontAskOnly pre-approved tools; everything else is auto-deniedLocked-down CI and scripts
bypassPermissionsEverything, no checksIsolated containers and VMs only

Switching is a keystroke: Shift+Tab cycles default → acceptEdits → plan mid-session, and the status bar always shows where you are (⏸ plan mode on, ⏵⏵ accept edits on). You can also start a session in a mode (claude --permission-mode acceptEdits) or set a project default in .claude/settings.json.

Distinctions the quiz cares about:

  • default vs. acceptEdits is when you review edits: inline as each happens, or “in your editor or via git diff after the fact.” Accept-edits only auto-approves inside your working directory; other Bash commands still prompt.
  • plan is not just “default with reads” — it's a hard gate. Edits stay blocked until a plan is approved, which is why M25a's workflow runs on it.
  • auto is not bypass. Auto sends actions through a classifier that blocks scope escalation, unrecognized infrastructure, and hostile-content-driven actions — and it treats boundaries you state in conversation (“don't push”) as block signals. But the docs are blunt: it “reduces permission prompts but does not guarantee safety.”
  • bypassPermissions is for sandboxes. “Only use this mode in isolated environments like containers, VMs… where Claude Code cannot damage your host system.” It offers “no protection against prompt injection.”
Protected paths — the floor under every mode

In every mode except bypassPermissions, writes to a small set of protected paths (.git, .claude, shell rc files, .mcp.json…) are never auto-approved — guarding repository state and Claude's own configuration even when you've loosened everything else. Allow-rules in settings don't override this.

3 · Applying it: mode follows task and trust

The Apply skill is a two-question decision: What's the worst uncaught action this task could take? and Where will I actually review — inline, at the diff, or at the PR? Worked through the week's own workflow:

  • Explore/Plan phases (M25a): plan — you want reading and thinking, with edits structurally impossible until the plan gate.
  • Implementing an approved plan you'll diff-review: acceptEdits — per-edit prompts add nothing when your real review is the diff; the plan already bounded the scope. (The plan-approval dialog offers exactly this switch.)
  • Touching unfamiliar, risky, or production-adjacent code: default/Manual — inline review is the point, and the prompts are few enough to stay meaningful.
  • A long, well-specified task you'll leave running: auto — prompt fatigue is the enemy, and the classifier plus your stated boundaries hold the edges. Verification proportional to how little you watched (M25b's evidence habit) picks up the slack.

“Justify the choice” means saying that out loud: “acceptEdits, because the plan is approved, the blast radius is this branch's working directory, and I review the diff before commit.” A mode you can't justify is a mode you picked by habit.

Hands-on: drive the dial

Self-check

1. You've approved a solid plan for a tracer-bullet issue in your own small project, and you'll review the full diff before committing. You choose acceptEdits over default. Which justification is the strong one?

C. It names the risk bound (plan + working directory), the review point (the diff), and the cost of the alternative (approvals you'd stop reading — the documented tenth-approval effect). A justifies any loosening of anything. B is false: default prompts for edits, it doesn't block them. D describes auto mode, which isn't the mode chosen — a justification has to match the configuration.

2. What actually distinguishes plan mode from default mode, given that both auto-approve “reads only”?

A. The table rows look identical because both list “reads only,” but the semantics differ: default asks per edit; plan mode “tells Claude to research and propose changes without making them” — the approval object is a plan, not an edit. C is backwards (reading is what plan mode is for). D is false: Shift+Tab or /plan enters it any time.

3. A classmate runs an overnight unattended task on their laptop with bypassPermissions “because auto mode kept pausing to ask.” Evaluate.

D. Auto's pauses were the safety mechanism working — the fix for excessive blocks is trusting infrastructure or narrowing the task, not deleting the checker. B is exactly the distinction the docs draw (classifier vs. nothing). C is false: bypassPermissions is the one mode where protected-path writes go through. A confuses “no human present” with “no checks needed” — the opposite of verification-proportional-to-supervision.

4. In auto mode you tell Claude: “don't push until I've reviewed.” Later in the session, Claude judges the work review-ready and tries to push. What does the documented behavior say happens?

B. Near-verbatim from the docs: “A boundary stays in force until you lift it in a later message. Claude's own judgment that a condition was met does not lift it.” That's the design point — the human's stated boundary outranks the agent's self-assessment. A and C both let the agent grade its own gate. D invents a mechanism; blocks fall back to prompting only after repeated denials.
