Failing Tests First: The Feedback Loop Is the Ceiling

Plan-first told the agent where to go; test-first tells it when it has arrived. This lesson teaches the red–green loop as an agent-direction technique — write one failing test, watch it fail, implement to green — and the argument underneath it: the quality of an agent's output is capped by the quality of the feedback it can read.

M27 · TDD Bloom: Apply Assessed by: Assignment A5

Objective: Direct Claude to write failing tests first, confirm they fail, then implement to green — and explain why feedback-loop quality is the ceiling on agent output.

1 · The ceiling argument

Start with the claim in the objective's second half, because it explains everything in the first half. Anthropic's best-practices guide: “Claude stops when the work looks done. Without a check it can run, ‘looks done’ is the only signal available, and you become the verification loop: every mistake waits for you to notice it.” Give Claude a check that returns pass or fail, and “the loop closes on its own.”

Put that as a general law: an agent iterates against whatever feedback signal it has. If the signal is “the code looks plausible,” output quality tops out at plausible. If the signal is a real test suite exercising real behavior, quality tops out at passes real tests. The agent cannot outperform its own feedback — which is why Pocock, whose TDD material grounds this lesson, treats disciplined red–green as the most consistent lever on agent output quality: the test is what gives the agent eyes. Improve the loop and every session inherits the improvement; leave it weak and no amount of prompting compensates.

2 · The loop: red, green, refactor — one behavior at a time

Pocock's /tdd skill runs the classic cycle with the phases stated as hard rules:

  • RED: “Write ONE test that fails. Just one.”
  • GREEN: “Write minimal code to pass that test only. Nothing speculative.”
  • REFACTOR: “After all tests pass, clean up duplications and simplify.”

Then repeat for the next behavior, “each cycle informed by what the last one taught you.” In a session, directing this looks like:

Directing the loop (one cycle)
1. "Write one failing test: creating a deck persists it across reload.
    Run it and show me the failure."          ← red, with evidence
2. Read the failure. Is it failing for the RIGHT reason
   (missing behavior), not a typo or import error?
3. "Now write the minimal code to make that test pass.
    Run the suite and show the output."       ← green, with evidence
4. "Any duplication to clean up? Keep the suite green."  ← refactor
5. Next behavior → back to 1.
Why “confirm they fail” is in the objective

A test you never saw fail proves nothing — it might pass against an empty implementation, test nothing, or not run at all. The failing run is the test's own test. And it's what makes the loop agent-proof: “if a test fails first, the LLM can't fake it, it has to write real implementation.” Skipping the red run reopens the door that TDD closed.

3 · Why not write all the tests up front?

The tempting shortcut — have Claude generate the full test file first, then implement — is the one the skill explicitly rejects. Tests “written in bulk test imagined behavior, not observed behavior”: they are “tests of imagined behaviour — they check the shape of things and go numb to real changes.” You'll recognize the vocabulary from Week 4: batching tests is horizontal slicing, and the fix is the same — vertical slices, “one test, then just enough code to pass it, then the next test.” Because you just wrote the code, you know exactly which behavior matters next; in the skill's phrase, “you never outrun your headlights.”

Two rules from the skill keep the signal honest:

  • Test through the public API. A good test “exercises real code paths through the public API, so renaming an internal function never breaks it.” The tell for the bad kind: your test breaks when you refactor, but the behavior hasn't changed — that test was coupled to implementation, and its red is a lie.
  • Expected values come from an independent source of truth — “a known-good literal, a worked example, the spec.” If the agent computes the expected value with the same logic being tested, the test can only confirm the code agrees with itself.
The ceiling, restated with the pieces in place

Weak loop: tests written in bulk from imagined behavior, expected values derived from the implementation, red runs never observed — the agent iterates happily against a signal that can't say no. Strong loop: one real failing test at a time, public-API paths, independent expected values — every green transition is information. Same model, same prompts; the difference in output quality is entirely the loop. That's what “the ceiling” means, and why A5 grades the process.

4 · Where this sits in the week's workflow

The pieces you've built compose in order: the plan says which behaviors the issue needs; each behavior becomes a red–green cycle; the green suite is the evidence M25b taught you to demand before commit. A5 encodes the sequence for both of your issues: “have Claude plan first and approve the plan; require failing tests before implementation; confirm green; open one PR per issue.” Pocock's phase-gated tdd skill (in mattpocock/skills, which you met in Week 4) packages the loop so the agent can't skip a gate — you're welcome to use it for A5, but be able to run the loop by hand first.

Self-check

1. Claude writes the test and the implementation in one turn, then reports the test passes. The suite is green. What's wrong?

B. The red run is the test's own test: it proves the test detects the behavior's absence. Without it, green is compatible with a vacuous test — the signal can't say no, which caps the loop's value at zero. A trusts a signal that was never calibrated. C inverts the discipline. D judges by effort, not by what the evidence establishes.

2. To “save cycles,” a classmate has Claude generate all fourteen tests for the issue up front, then implement until everything passes. Per Pocock's material, the predictable cost is:

D. This is the horizontal-slicing failure, in the skill's own words. The loop's value isn't “tests exist” — it's that each cycle's discovery feeds the next test, which batching forfeits by fixing all fourteen before the first discovery. A mistakes test-first-in-time for TDD's actual unit (one behavior per cycle). B and C are real but trivial next to a test suite that can't detect real regressions.

3. A test computes its expected value by calling the same date-formatting helper the code under test uses, and asserts they match. The suite is green. What's the diagnosis?

A. The skill's rule exists for this case: “expected values come from an independent source of truth — a known-good literal, a worked example, the spec.” A self-referential expected value makes the feedback signal circular — green means “consistent,” not “correct.” B applies a production-code virtue where it destroys the test's independence. D isn't a diagnostic criterion at all.

4. A classmate asks: “Why does everyone say feedback-loop quality is the ceiling on agent output? Surely a smarter model just writes better code.” The course's answer:

C. This is the best-practices logic: Claude stops when the work looks done, and the check is what upgrades “looks done” into “is done.” A stronger model raises what's reachable under the ceiling; only a better signal raises the ceiling. A gets the dependency backwards — autonomy increases the need for checks. D is false (builds, linters, screenshots, fixtures all count — tests are just the sharpest for logic), and B dissolves the one claim the objective asks you to be able to defend.
