Agentic Coding with Claude Week 5 · Objective M25

Code Against the Plan, Then Commit

You approved a plan in M25a. Now the second half of the workflow: implementation that verifies itself against the plan as it goes, evidence instead of assertions, and a commit-and-PR step that hands the work to review. The plan you approved is the contract; this lesson is about holding the implementation to it.

M25 · Execution Bloom: Apply Assessed by: Assignment A5

Objective: Apply the Explore–Plan–Code–Commit workflow: have Claude read relevant code, produce a plan, then implement only after plan approval.

1 · Phase 3: Code — implement, verifying against the plan

Approving the plan exits plan mode; Claude can now edit. The best-practices guide's example prompt for this phase does three things at once:

Implement prompt (default or accept-edits mode)
implement the OAuth flow from your plan. write tests for the
callback handler, run the test suite and fix any failures.

It anchors the work to the approved plan (“from your plan”), it demands tests, and it demands the suite actually run. That last part is the load-bearing one. The guide's central warning: “Claude stops when the work looks done. Without a check it can run, ‘looks done’ is the only signal available, and you become the verification loop.” Give Claude a check — a test suite, a build exit code, a linter, a script that diffs output against a fixture — and “the loop closes on its own. Claude does the work, runs the check, reads the result, and iterates until the check passes.”

This is Week 3's M18 (verifiable targets) operating at the implementation phase — and it's the bridge to M27, where the check gets written before the code.

Demand evidence, not assertions

“Have Claude show evidence rather than asserting success: the test output, the command it ran and what it returned, or a screenshot of the result. Reviewing evidence is faster than re-running the verification yourself.” “Done” in the transcript is a claim; a pasted test run is a fact.

2 · An optional gate before commit: adversarial review

For work you didn't watch closely, the best-practices guide recommends one more check before you call it done: a subagent that reviews the diff in a fresh context, seeing “only the diff and the criteria you give it, not the reasoning that produced the change.”

Review-against-plan prompt (from the best-practices guide, adapted)
Use a subagent to review the diff against your approved plan. Check that
every requirement is implemented, the listed edge cases have tests, and
nothing outside the task's scope changed. Report gaps, not style preferences.

One calibration warning from the same guide: a reviewer told to find gaps “will usually report some, even when the work is sound.” Chase only findings that affect correctness or the stated requirements — the rest is optional polish, and over-reacting to it produces the over-engineered defensive code you'll learn to spot in Week 6.

3 · Phase 4: Commit — and open the PR

The final phase is one prompt: “commit with a descriptive message and open a PR.” Claude drafts the commit message and PR description from the work it just did — and per the common-workflows doc, you can refine it: “enhance the PR description with more context about the security improvements.” The docs' one instruction for this phase: “Review Claude's generated PR before submitting.”

Two mechanics worth knowing:

  • The session is linked to the PR. “When you create a PR using gh pr create, the session is automatically linked to that PR.” Later, claude --from-pr 1234 reopens the session picker filtered to that PR's sessions — review feedback in Week 6 lands back in a session that remembers the work.
  • The PR closes the issue. Week 4's closing keywords (“Closes #7”) belong in the PR description, so merging completes the issue → branch → PR → auto-close loop.
Where the workflow meets Assignment A5

Self-check

1. Claude reports: “I've implemented the OAuth flow and everything works.” No test output appears in the transcript. Per the best-practices guide, what's the right response?

B. “Works” without evidence is exactly the “looks done” signal the guide warns about. C works but forfeits the point of the pattern — “reviewing evidence is faster than re-running the verification yourself.” D substitutes a feeling for a fact; a confident assertion is still an assertion. A makes you the verification loop's missing half without noticing.

2. Your adversarial review subagent returns six findings: one missed acceptance criterion, and five suggestions to add defensive checks “for robustness.” What does the guide say to do?

D. The guide's calibration is precise: flag and fix what affects “correctness or the stated requirements,” treat the rest as optional. A is how review-driven over-engineering happens — extra abstraction and tests for cases that can't occur. B throws away the one real finding with the noise. C is reviewer-shopping; it optimizes for agreement, not correctness.

3. Why does the Commit phase belong in the workflow at all, rather than being an afterthought you handle manually later?

A. Each mechanic — descriptive message, “Closes #N,” the automatic gh pr create session link — connects this session's work to the systems (review, the board, future sessions) that consume it. B is false: checkpointing tracks Claude's edits independently of git. C is unfounded. D deletes the phase that makes the other three visible to anyone but you.

4. Mid-implementation, Claude finishes step 3 of its five-step plan and then starts refactoring an unrelated utilities file “while we're here.” The plan-anchored response is:

C. “Nothing outside the task's scope changed” is literally one of the review-prompt criteria — scope drift now is a review finding later. A buys an unreviewable mixed diff (Week 4's one-issue-per-PR lesson in reverse). B over-corrects; a redirect preserves three good steps (interrupting well is M29/M30). D makes the plan follow the drift, inverting the contract.
