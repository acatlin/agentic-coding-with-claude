One Issue, One Fresh Session, One PR

Week 4 ended by arguing this rule; this week you run it. Each issue gets its own fresh session, seeded with exactly two things — your CLAUDE.md and the issue body — and produces exactly one PR. This lesson turns the rule into a repeatable operating procedure and shows what each constraint is buying you.

M28 · Execution Bloom: Apply Assessed by: Assignment A5

Objective: Run each issue in its own fresh session with CLAUDE.md and the issue body as context; open one PR per issue.

1 · The procedure

The course workflow for executing one issue, end to end:

One issue, start to finish
1. Fresh session in the repo:  claude --permission-mode plan
     → CLAUDE.md loads automatically; context is otherwise empty
2. Seed the task:  "Implement issue #7. Read it with: gh issue view 7"
     (or paste the issue body; @-reference any files it names)
3. Branch per issue (Week 4 convention), e.g. 7-deck-persistence
4. Explore → Plan → approve          (M25a)
5. Red → green per behavior          (M27)
6. Evidence, then commit + PR with "Closes #7"   (M25b)
7. End the session. Next issue = new session at step 1.

Every piece of this is one you've already learned; the rule's job is to keep the pieces from bleeding into each other. Notice what the seed contains: CLAUDE.md carries the durable conventions (“Claude reads it at the start of every conversation”), and the issue body carries the task — which is why Week 4's M20 made you write issue bodies that can stand alone as a work order, with acceptance criteria an agent can verify. If the session needs a third thing to succeed, that's feedback: the issue body was underspecified.

2 · Why fresh, again — now with the execution-side evidence

Week 2 gave you the context argument (the smart zone; the fresh-vs-polluted experiment); Week 4's M24 argued the rule from the reviewer's chair. The best-practices guide adds the execution-side version. Its documented failure pattern: “The kitchen sink session. You start with one task, then ask Claude something unrelated, then go back to the first task. Context is full of irrelevant information.” The fix is the habit A5 enforces: “/clear between unrelated tasks” — and two issues are unrelated tasks by construction, because Week 4's decomposition made each one independently completable.

The same guide makes the positive case when describing spec-driven work: once the spec exists, “start a fresh session to execute it. The new session has clean context focused entirely on implementation, and you have a written spec to reference.” Your issue body is that spec, scaled to one tracer bullet.

Fresh session, not fresh conversation-in-the-same-window

Running /clear achieves the reset; so does closing the terminal and starting over. What matters is what M12's experiment measured: the issue starts with a context containing CLAUDE.md, the issue, and nothing else — no leftover plan from the previous issue, no failed approaches, no unrelated file reads.

3 · Why one PR per issue

The PR inherits the issue's scope, and everything downstream depends on that inheritance:

  • Review stays possible. A one-issue diff is a reviewable unit (Week 4's argument). Week 6 makes you review a partner's Week 5 PR against a checklist — a two-issue PR would make their job structurally harder.
  • The board stays truthful. “Closes #7” in the PR auto-closes exactly the issue the work implemented. A PR that closes two issues can only be merged or reverted as a pair.
  • The session-PR link stays meaningful. gh pr create links the session to the PR, and claude --from-pr finds it later — one session, one issue, one PR means the resumed context is exactly the relevant one.

A5 grades the shape directly: two issues, each in its own fresh session, one PR each, “both PR links with a note on where you intervened.”

Running both issues at once?

If issue #2 doesn't depend on issue #1 (check your blocked-by edges from M22), the two fresh sessions can run in parallel — the docs' worktree feature exists for exactly this: claude --worktree feature-auth gives each session an isolated checkout “so concurrent edits don't collide.” Optional for A5; the required part is only that each issue's session is fresh and its own.

Hands-on: A5, issue one

This checklist is the assignment's first half — run it for real:

Self-check

1. Finishing issue #4's PR, a classmate thinks: “The session already understands the codebase — starting over for #5 wastes all that warmed-up context.” What's the course's answer?

C. This is the kitchen-sink pattern wearing a productivity costume. The Week 2 insight applies: context that helped task A is noise for task B, and noise degrades output. The real lesson in the intuition: anything from #4 that would genuinely help every future issue is a CLAUDE.md candidate — promote it to the durable layer instead of smuggling it in as session residue. B compresses the pollution; it doesn't remove it.

2. Ten minutes into a fresh session, Claude asks which storage backend the app uses — the issue body doesn't say, and it matters for the implementation. Beyond answering, what should you do?

B. “If the session needs a third thing to succeed, the issue body was underspecified” — the question is free feedback on your spec layer, and the fix belongs in the artifact, not just the chat. A answers this session and re-pays the cost every future session. D fixes underspecification with overload — the smart-zone trade in the wrong direction. C mistakes a fixable gap for a fatal one.

3. While implementing issue #6, Claude notices and fixes a real bug that belongs to issue #8, so the PR now closes both. Why does the course still count this as a miss?

D. Each cost lands on a different consumer: the reviewer (Week 6), the board's truthfulness, and #8 itself — whose acceptance criteria were never the running session's target. The right move mid-run was the M25b redirect: stop, keep the fix out of scope, note it on #8. A counts outputs; the workflow grades units. B and C invent rules the course doesn't have.

4. Your two A5 issues have no blocking edge between them. What's the docs-supported way to run both at once — and the constraint that survives?

A. Worktrees exist so parallel sessions get isolated checkouts and “concurrent edits don't collide.” The rule constrains the mapping (issue↔session↔PR), not the calendar. B is the kitchen-sink session with extra steps — alternating pollutes both tasks' contexts. C invites the collisions worktrees prevent. D over-reads the rule; independent slices running in parallel is what the M22 frontier was for.
