Course-Correcting a Running Agent

Between “prompt sent” and “PR opened,” you are not a spectator. Claude Code gives you a small set of mid-run controls — interrupt, comment, rewind, undo — that operate while the work is still happening. This lesson maps each control to what it changes, and draws the line the quiz will test: how correcting during a run differs from reviewing after it.

M30 · Foundations Bloom: Understand Assessed by: Quiz 5

Objective: Describe how course-corrections during a run (interrupt, comment, rewind) differ from post-hoc review.

1 · The premise: tight feedback loops beat perfect first prompts

The best-practices guide's framing: “The best results come from tight feedback loops. Though Claude occasionally solves problems perfectly on the first attempt, correcting it quickly generally produces better solutions faster.” Course-correction tools exist because your prompt was a bet, and the run is where you find out how the bet is going. The guide's headline advice — “Correct Claude as soon as you notice it going off track” — only works if you know what each correction tool actually does.

2 · The mid-run toolkit

Four controls, from the documentation, ordered by how much they undo:

  • Esc — interrupt. “Stop the current response or tool call mid-turn so you can redirect. Claude keeps the work done so far.” Nothing is reverted; the session's context is preserved. You interrupt to add steering: “stop — use the existing session store in src/auth, don't write a new one.”
  • Comment / redirect. The message you type after interrupting (or between turns) is the correction itself. Because context is preserved, Claude carries your redirect plus everything it already learned.
  • Esc+Esc or /rewind — go back in time. Opens the rewind menu. “Every user prompt creates a checkpoint” — Claude Code snapshots files before each one — and from the menu you can restore code and conversation, restore conversation only (keep current code), or restore code only (keep the conversation). Rewind is for when the last N minutes were a wrong branch: not steering, unwinding. You met this in Week 2 (M11b).
  • “Undo that” — ask Claude to revert. The lightest-weight rollback: have Claude reverse its own last change as a normal instruction, when the miss is small and recent.

And the escape hatch when correction itself stops working: /clear. The guide's rule of thumb — “If you've corrected Claude more than twice on the same issue in one session, the context is cluttered with failed approaches… A clean session with a better prompt almost always outperforms a long session with accumulated corrections.” A third correction is evidence the problem is upstream, in the prompt or spec.

What rewind does not cover

Checkpoints track Claude's file-editing tools only: “files modified by bash commands” (an rm, a mv, a script run) “cannot be undone through rewind,” and external or concurrent-session changes aren't captured either. The docs' summary: checkpoints are “local undo,” git is “permanent history” — one more reason the branch-per-issue habit stays load-bearing.

3 · During-the-run vs. after-the-run

Post-hoc review — reading the finished diff, running /code-review, the whole of Week 6 — differs from course-correction on every axis that matters:

Course-correction (during)Post-hoc review (after)
ObjectThe trajectory: what Claude is doing and about to doThe artifact: the finished diff
TimingMid-turn, while context is liveAfter the turn or PR, context optional
What it can changeThe path — before more work builds on a wrong stepThe verdict — accept, request changes, reject
Cost of a catchCheap: the wrong step is minutes old, context is preserved for the redirectDearer: rework re-enters a session (or a fresh one) after the fact
What it catches bestWrong direction: misread spec, wrong file, wrong approachWrong details: edge cases, drift, the failure modes checklists find
Blind spotYou see fragments in motion — easy to misjudge unfinished workWhatever compounded before you looked

The two are complements, not competitors. Course-correction keeps the run pointed at the right problem; review judges whether the result is right. The plan gate (M25a) is course-correction at its cheapest — redirecting words instead of code — and Week 6's diff review is post-hoc at its most systematic. When to reach for which mid-run is a judgment call, and it's the next lesson's whole subject.

Self-check

1. You press Esc while Claude is mid-edit in the wrong file. What state is the session in?

B. Per the docs: “Stop the current response or tool call mid-turn so you can redirect. Claude keeps the work done so far.” Interrupt is steering, not undo — that's what makes it cheap. A describes rewind, a different tool for a different situation. C confuses interrupt with exit; D invents per-file semantics neither tool has.

2. Twenty minutes of session history turn out to rest on a wrong architectural assumption from your third prompt. The code changes since then are all wrong, but the conversation taught you a lot you want to keep. Which control fits?

D. This is exactly why the rewind menu separates its options: “restore code” reverts file changes while keeping the conversation. A steers the future but leaves twenty minutes of wrong code on disk for Claude to build on. B is sized for one small recent change, not a twenty-minute branch. C throws away the conversation you explicitly wanted to keep — the strongest tool, used one notch too hard.

3. During a session, Claude ran rm old-fixtures/ -r via bash. Later you rewind code to an earlier checkpoint, expecting the directory back. What happens, and why?

A. The limitation is verbatim in the checkpointing doc, with rm as its own example. Knowing a tool's coverage boundary is part of understanding the tool — misplaced confidence in rewind is how a “recoverable” mistake becomes a real one. B and D assume coverage that doesn't exist; C invents a guard the tool doesn't have.

4. A classmate says: “I don't bother steering mid-run — I just review the diff hard at the end. Review catches everything correction would, one pass, no interruptions.” What does the during/after distinction say?

C. The asymmetry is the objective's core: correction changes the trajectory while changing it is cheap; review judges the artifact after costs have compounded. But the converse also holds — unfinished work in motion hides the edge-case and drift failures that diff review (Week 6) is built to find. A and B each keep one tool and inherit its blind spot. D dissolves a distinction with concrete, costable consequences.
