PRD as Destination, Not Compiler Input

Two mental models compete over what a spec is for. One treats the PRD as source code for a compiler made of agents; the other treats it as a marker on the map you're navigating toward. Which model you hold decides how much polish the document deserves — and how you feel when it goes stale.

M16 · Specification Bloom: Understand Assessed by: Quiz 3

Objective: Explain ‘PRD as destination’: why over-polishing specs wastes effort and why doc rot is expected and acceptable.

1 · Two models of a spec

Compiler-input model (“specs-to-code”)Destination model (this course)
What the spec isThe durable source of truth; code is a generated artifactA marker recording where you agreed to go; code is the product
What you maintain foreverThe spec — change it and regenerate the codeThe code (and its tests) — the spec served its purpose
Required polishBlueprint-grade: complete, precise, perpetually currentDestination-grade: unambiguous about the decisions, done
When spec and code divergeAn emergency — the truth is corruptedExpected — the journey taught you things the marker didn't know

The specs-to-code movement — tools and workflows promising that if the spec is good enough, the code becomes a disposable compilation artifact — is the model Pocock explicitly argues against in his AI Engineer 2026 workshop. His counter is that understanding and managing the code remains essential: the code is where reality lives, where tests run, where the next feature starts. The PRD's actual job is narrower and shorter-lived. It gets its value at exactly two moments:

  1. Alignment capture — the moment you write it, it freezes the grill session's shared design concept into a page the meeting can critique and the build can start from.
  2. Ambiguity resolution during the build — “a PRD serves as the destination document that agents can reference throughout implementation.” When the agent hits a design question mid-build, it returns to the PRD for an authoritative answer instead of inventing one — M13's silent-decision failure mode, blocked by a document.

After the feature ships, both moments are behind you. Nothing still depends on the document.

2 · Why over-polishing wastes effort

The workshop's blunt version: “over-optimizing the PRD may not yield significant value.” Here's the reasoning, not just the verdict. A document's worth-polishing threshold is set by what reads it and what that reader does differently. The PRD's readers are your Meeting 3 critics and a build agent — and once the document is unambiguous about the decisions, neither reader behaves differently because you added diagrams, tightened prose, or enumerated edge cases the grill never surfaced. The build produces the same code. Polish past unambiguity is effort with no downstream effect.

Compare where effort does have downstream effect, minute for minute:

  • Before the PRD: alignment. Ten to fifteen minutes of grill questions prevent hours of wrongly-aimed autonomous work. That leverage lives in the conversation, not the document's finish.
  • After the PRD: verification. The workshop's advice is to shift the surplus effort toward QA — acceptance criteria that are actually checkable, and (Week 5) tests that enforce them. A sharper acceptance criterion changes what the agent builds; a prettier goals section doesn't.

Even the format is officially unsacred: the PRD “should encapsulate design concepts effectively without strict adherence to any particular format.” The course fixes a five-section template for peer-critique comparability, not because the sections are magic. And the 1–2 page limit is the anti-over-polish mechanism built right into the assignment: you cannot blueprint in two pages.

The tell

You're over-polishing when your edits stop changing decisions and start changing presentation. “Rewrote story 4 because it contradicted non-goal 2” is work. “Reordered sections and added a mermaid diagram” is, per this objective, waste — quiz-grade waste.

3 · Doc rot: expected, acceptable, designed-for

Doc rot is what happens next, and the destination model changes its meaning entirely. During and after the build, the code moves — implementation discovers constraints the grill missed, Meeting 3 feedback reshapes a story, Week 5's tests pin behaviors the PRD stated loosely. The document stays where it was. Under the compiler model this divergence is corruption of the source of truth, demanding constant re-synchronization. Under the destination model it's evidence of arrival: markers are supposed to be behind you eventually.

Three practices follow:

  • Design for rot. Build the PRD from the slowest-rotting material: decisions and behaviors. This is why M15 banned file paths and code snippets — Pocock's template excludes them precisely because they “may end up being outdated very quickly.” A PRD that says “storage is local, offline-first” rots in years; one that says “see src/store/localStore.js” rots in days.
  • Don't maintain it — retire it. Nobody updates a trail marker after the hike. The workshop's guidance goes further than tolerance: once an old PRD has become irrelevant or misleading, don't keep it in the repo — delete or archive it. A stale-but-present document is worse than an absent one, because future sessions (and future agents) will read it as if it were true.
  • Know what must NOT rot. The tolerance is scoped, and the quiz will probe the boundary. Your tests must track the code (they are its current truth). Week 2's durable state — CLAUDE.md, the project glossary — must stay accurate because it's loaded into every future session. The difference is consumption: rot is acceptable in a per-feature document whose job ends, unacceptable in standing context that keeps getting read.
Rot is not a license to skip

“It'll be stale in a month” argues against polishing, never against writing. The PRD earns its keep during the weeks it's live — alignment captured, critique enabled, mid-build ambiguity answered — and A3, Meeting 3, and the Week 5 build all consume yours during exactly that window. Expected rot means the document is allowed to die after serving; it was never optional to have served.

Self-check

1. A teammate wants to spend the rest of the week on the PRD: architecture diagrams, exhaustive edge-case tables, polished prose — “so the agent build can't go wrong.” The destination model's answer:

C. “Over-optimizing the PRD may not yield significant value” — the reader (agent or critic) does nothing differently once the decisions are unambiguous, so extra polish has no downstream effect; sharper checkable criteria do. B swaps one error for another: vagueness reopens M13's silent decisions. The de-risking instinct in A is right but aimed at the wrong artifact — verification, not documentation, is what catches a build going wrong.

2. Three weeks after your feature ships, you notice PRD.md describes a scheduling behavior the app no longer has. Under the destination model, the right response is:

B. The marker is behind you; the code (with its tests) is the current truth. The one active duty is hygiene: a misleading document left in the repo will be read as true by future agents, so the workshop's guidance is to remove or archive it rather than maintain it. A is the compiler model's perpetual-sync burden; C and D both subordinate working code to a dead document — the exact inversion the destination model rejects.

3. Which pairing correctly matches an artifact to its rot policy?

D. The dividing line is consumption: rot tolerance is for per-feature documents whose reading life ends, not for standing context. CLAUDE.md is loaded into every session (Week 2) and tests run against every change — staleness there actively corrupts future work, whereas a retired PRD just needs to be archived out of the way. A inverts both policies; C is the compiler model generalized into a full-time documentation job.

4. Mid-build, the agent hits a question the PRD does answer (“what happens when no cards are due?”) — and consults PRD.md instead of picking a behavior. Which claim about the destination model does this illustrate?

A. “Agents can reference [it] throughout implementation” — this is the destination doing its live job, converting a would-be silent decision (M13) into a lookup. B misreads “destination” as “write-only”; the marker guides the whole journey and only then retires. D is the opposite case: rot is when the document no longer matches reality, not when it answers correctly.
