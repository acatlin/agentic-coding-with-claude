Why Grill First: Alignment Before Implementation

The most expensive failure in agentic coding isn't bad code — it's good code that solves the wrong problem. This lesson explains the grill session: an interview, run by the agent, that makes sure you're both building the same thing before either of you commits to it.

M13 · Specification Bloom: Understand Assessed by: Quiz 3

Objective: Explain why alignment precedes implementation: what a grill session is and what failure modes it prevents.

1 · The misalignment problem

Anthropic's best-practices guide states the risk in one sentence: “Letting Claude jump straight to coding can produce code that solves the wrong problem.” Week 2 was about protecting execution — keeping the session in the smart zone so the agent works well. This week starts from a harsher observation: perfect execution is worthless when the target is wrong. Matt Pocock's skills repository names four recurring failure modes of agentic work, and misalignment — the agent building something other than what you meant — is the first, with a specific prescribed fix: use grilling sessions to align before starting.

Why does misalignment happen to careful people? Because a one-paragraph feature request contains dozens of unstated decisions, and every unstated decision still gets made — by the agent, mid-implementation, silently. You saw a miniature version in Week 1's transcript analysis: the spec said nothing about trailing newlines, so Claude decided, and the human only caught it in the diff. Scale that from one newline to a whole feature — data model, edge cases, what happens on failure — and you get software that is plausible, working, and wrong.

2 · What a grill session is

A grill session inverts the usual direction of conversation: instead of you prompting the agent to produce, the agent interviews you — “relentlessly,” in Pocock's word — until the two of you share the design. His /grill-me skill, which went viral and which Assignment A3 asks you to install or emulate, is short enough to quote whole:

The grilling skill, verbatim (mattpocock/skills)
Interview me relentlessly about every aspect of this plan until we
reach a shared understanding. Walk down each branch of the design
tree, resolving dependencies between decisions one-by-one. For each
question, provide your recommended answer.

Ask the questions one at a time, waiting for feedback on each
question before continuing. Asking multiple questions at once is
bewildering.

If a *fact* can be found by exploring the codebase, look it up
rather than asking me. The *decisions*, though, are mine — put each
one to me and wait for my answer.

Do not enact the plan until I confirm we have reached a shared
understanding.

Every line encodes a rule of the session:

  • One question at a time. Batched questions are “bewildering” — you skim, you answer shallowly, decisions slip through unexamined.
  • A recommended answer with each question. Often you just accept it; when you override it, that override is exactly the information the agent could never have guessed. Pocock added this line specifically to speed sessions up.
  • Facts from the codebase, decisions from you. The agent should never ask you what the code already answers — but it must never decide for you what only you can choose.
  • A stop condition. No implementation until you confirm shared understanding — the session ends by agreement, not exhaustion.

Real sessions are long: accounts from Pocock's AI Engineer 2026 workshop put them at 40–100 questions, and he describes typical sessions as roughly 45 minutes of “detailed, in-depth conversation.” The output is not primarily a document — it's “a rich conversation full of context, your actual ideas, and answers to difficult questions,” which he calls a shared design concept: you and the agent genuinely holding the same picture of what's being built. (Next lesson you'll write it down; that's a separate step.)

Rubber ducking, upgraded

Pocock frames grilling as the modern form of rubber-duck debugging: engineers have always discovered requirements by explaining plans out loud. The upgrade is that this duck asks follow-up questions — “fantastic in the early phase of coding, because you need to discover all of the requirements by talking them through.” He uses it beyond code, too (his example: deciding what course to build next).

3 · The failure modes it prevents

Failure mode without grillingHow the grill session prevents it
Solving the wrong problem — plausible, working, unwanted softwareMisunderstandings surface as questions before code exists, where a wrong assumption costs one sentence to correct (the same economics as plan mode in M11)
Invisible agent decisions — unstated choices resolved silently mid-build“The decisions, though, are mine”: each design choice is put to you explicitly and answered on the record
Wasted autonomous work — hours of agent time down a wrong branchWorkshop framing: 10–15 minutes of alignment questions prevent hours of wasted autonomous work — and the more you plan to walk away (Week 8), the more this matters
Late-discovered requirements — “oh, I also needed…” after the buildWalking “each branch of the design tree” forces areas you hadn't considered — auth, failure cases, edges — into the open early

4 · Calibration: when to grill

Grilling is not a virtue ritual, any more than plan mode was. The documented sweet spots: before writing a PRD, before asking for a feature implementation, before finalizing data models or API specs, and whenever multiple design choices interconnect. For a task whose diff you could describe in one sentence, 80 questions is pure overhead — the one-sentence test from Week 2 applies here unchanged.

Anthropic's best-practices guide teaches the same move in official form: “For larger features, have Claude interview you first” — a minimal prompt asking Claude to interview you in detail, digging into “the hard parts I might not have considered,” then writing a spec. Two vendors, one conclusion: for non-trivial work, the interview comes first.

The tool moved; the idea didn't

Pocock iterates fast: he has since said he now prefers /grill-with-docs (which also builds a domain model from your codebase) when aligning a plan with an existing codebase, keeping plain /grill-me for pure plan interviews. A3 targets the interview discipline itself — and your course project starts from an empty repo — so the plain grill is the right tool this week. Expect the skill names to keep changing; the alignment-before-implementation principle is the part to master.

Self-check

1. A student prompts “build me a flashcard app with spaced repetition,” and twenty minutes later Claude delivers a polished, working app — that schedules reviews completely differently from what the student had in mind. Which failure mode is this, in the course's terms?

C. The app works, so execution (and context) did their job — the failure happened before the first token of code, when “spaced repetition” left the scheduling algorithm, among dozens of other decisions, for the agent to resolve alone. A and B are execution failures; D (M16's topic) concerns documents aging after alignment, not alignment never happening.

2. Mid-grill, Claude asks: “Should decks be shareable between users? Recommended: no — single-user keeps v1 simple.” You realize you'd never thought about sharing at all. What just happened?

B. Surfacing decisions you hadn't considered is the point of walking every branch of the design tree — this is the “discover all of the requirements by talking them through” effect. The recommended answer (C) is required by the skill, not forbidden. D inverts the fact/decision rule: whether decks should be shareable is a decision, and it's yours.

3. Grilling a feature for an existing repo, Claude asks: “What database does the project currently use?” Per the skill text, what's wrong?

D. “If a fact can be found by exploring the codebase, look it up rather than asking me.” Questions spent on lookupable facts waste the human's attention, which the session exists to concentrate on genuine decisions — which database a greenfield project should use would be a fair question; what it currently uses never is.

4. Which task is the weakest candidate for a grill session first?

A. Its diff fits in one sentence — the same test that says skip plan mode says skip the grill; interviewing yourself for 45 minutes about a rename is overhead with nothing to align. B, C, and D are the documented sweet spots: data models, interconnected design choices, and pre-PRD alignment — and D is where misalignment is costliest, since nobody is watching an AFK build go wrong.
