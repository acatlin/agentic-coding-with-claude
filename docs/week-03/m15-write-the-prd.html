Agentic Coding with Claude Week 3 · Objective M15

Write the PRD

The grill session left you and the agent aligned; now you write it down. One to two pages that mark the destination — goals, non-goals, user stories, constraints, acceptance criteria. This is the document A3 grades and your peers critique at Meeting 3.

M15 · Specification Bloom: Create Assessed by: Assignment A3

Objective: Write a 1–2 page PRD treating it as a destination marker (not a compiler input): goals, non-goals, user stories, constraints, acceptance criteria.

1 · Synthesis, not a second interview

Write the PRD immediately after the grill session, in the same context. The reasoning is Week 2's plan-mode logic again: the conversation you just had is the well-contexted state a good synthesis needs. Pocock's PRD-writing skill (he calls the document a spec — “you may know this document as a PRD”) is explicit about the division of labor:

From Pocock's to-spec skill

“It does not interview you again. By the time you reach for it, the alignment work is done — to-spec synthesises what is already known rather than asking a fresh round of questions.”

So the workflow is: grill (M14) → “now write the PRD from what we've agreed, using this template” → you edit. Having Claude draft it is legitimate — every idea in the draft is yours, because it came out of your grill answers. What you can't delegate is the read-through: the Create objective grades a document you own line by line, and Meeting 3 will ask you to defend it. If the draft contains a decision you don't recognize, that's a misalignment that survived the stop test — fix it now.

2 · The five sections

A3 fixes the template: goals, non-goals, user stories, constraints, acceptance criteria. (Pocock's own template differs cosmetically — problem statement, solution, stories, implementation and testing decisions, out of scope — the course template covers the same ground and gives Meeting 3 comparable documents to critique.)

SectionWhat belongs thereRunning example (flashcard app)
GoalsWhat the project does and for whom, from the user's perspective. Three to six bullets; each one something the finished product will demonstrably do.“A student can create decks of question/answer cards and review them on a spaced schedule.”
Non-goalsWhat you are deliberately not building — the scope fence. Every “should we…? → no” from the grill lands here. As load-bearing as goals: this section is what an agent (or a peer critic) checks scope against.“No accounts, no sharing, no mobile app in v1.”
User storiesNumbered, in the fixed form “As an <actor>, I want <feature>, so that <benefit>”. Pocock's template asks for an extensive list covering all aspects — each story a concrete, independently verifiable behavior.“3. As a student, I want cards I marked hard to reappear sooner, so that my weak spots get more practice.”
ConstraintsThe technical and process decisions the grill settled: stack, storage, offline behavior, performance floors. Decisions, not layout — no file paths or code snippets, which “may end up being outdated very quickly.”“Plain HTML/JS, no build step; data in a local JSON file; must work offline.”
Acceptance criteriaCheckable statements a stranger — or an agent — can verify pass/fail without asking you anything. These are verifiable targets (M18) written in prose; in Week 5 they become tests.“Reviewing a card and grading it ‘hard’ reschedules it within 1 day; ‘easy’ within 4+ days.”
The stranger test

For every acceptance criterion ask: could someone who has never spoken to me decide, yes or no, whether this holds? “The app should be fast and intuitive” fails the test — fast at what, measured how? “A deck of 100 cards loads and renders in under a second” passes. Unverifiable criteria are the #1 flaw your peers will be hunting at Meeting 3 (M17 trains them to).

3 · Destination-grade, not blueprint-grade

The objective's parenthetical — destination marker, not compiler input — sets the quality bar, and it's lower than your instincts say. From Pocock's AI Engineer workshop: the PRD is “the destination document that agents can reference throughout implementation”; “over-optimizing the PRD may not yield significant value”; and the format is flexible — it “should encapsulate design concepts effectively without strict adherence to any particular format.” The full argument is the next lesson; for the writing session, three practical consequences:

  • The 1–2 page limit is a feature. It forces you to record decisions, not designs. If a section is ballooning, you're writing implementation, which belongs to the agent (Week 1's division of labor).
  • Stop at “unambiguous,” not at “impressive.” The document is done when a fresh agent reading it would make your decisions, not its own — additional polish past that point changes nothing about the build. Anthropic's spec guidance agrees on where the effort goes: the useful spec is self-contained, states what's out of scope, and ends with verification — precision of content, not production values.
  • Write for two readers. A peer critic at Meeting 3 and an agent in Week 5 both need to act on it without you in the room. Neither can use your good intentions; both can use a numbered story with a checkable criterion.

Here's what destination-grade looks like — a condensed excerpt (not a full PRD) of the running example:

PRD.md — excerpt
# StudyStack — PRD (v1)

## Goals
- A student can create, edit, and delete decks of question/answer cards.
- Review sessions show due cards one at a time; grading a card
  (again / hard / easy) reschedules it on a spaced schedule.
- All data survives closing the browser.

## Non-goals (v1)
- No accounts, no sharing, no sync between devices.
- No images or audio on cards. Text only.
- No statistics dashboard beyond a due-card count.

## User stories
1. As a student, I want to create a deck with a name, so that I can
   organize cards by course.
2. As a student, I want to add a card by typing a question and an
   answer, so that capture is fast during lectures.
3. As a student, I want cards I graded "again" to reappear in the
   same session, so that I don't end on a failure.
   …

## Constraints
- Plain HTML/CSS/JS, no build step, no backend; storage is
  localStorage. Must work offline after first load.

## Acceptance criteria
- Creating a deck and reloading the page: the deck is still there.
- Grading a card "easy" removes it from today's queue; "again"
  re-queues it in the current session.
- With no cards due, the review screen says so instead of erroring.

4 · Commit it

A3's last step: the PRD lives in your course-project repo as PRD.md, committed. This isn't ceremony — committed at the repo root is where the Week 5 agent will read it from, and version history means Meeting 3 feedback becomes a visible revision rather than a mystery edit. Bring the document (screen or paper) to the meeting for the peer critique.

Hands-on checklist

The second half of Assignment A3:

Self-check

1. You type “now write the PRD” and Claude replies: “Great! First, a few questions: who is the target user? What's the core feature?” What went wrong?

B. “It does not interview you again… the alignment work is done.” Re-asking answered questions signals the draft isn't building on the session — maybe you cleared the context (losing the asset M14 told you to protect), maybe the prompt didn't point at the decisions. Redirect: “synthesize from the decisions we resolved above.” C is false — drafting is delegable; owning the content isn't.

2. Which acceptance criterion passes the stranger test?

D. A stranger can perform it: grade a card “again,” keep reviewing, observe whether it returns — yes or no, no interview with the author required. A and C are judgments with no measurement; B names the right area but delegates the entire definition of “correctly” back to whoever's asking — which, mid-build, will be the agent, deciding silently (M13's failure mode, now in document form).

3. Your constraints section is growing: you're adding the module layout, function names, and a code snippet for the scheduler. Why does the course (and Pocock's template) say to stop?

A. The template's stated reason: paths and snippets “may end up being outdated very quickly.” A destination document should be built of the slowest-rotting material available — decisions — so it stays trustworthy for its short working life (M16 takes this further). It's also the division of labor: how the scheduler is arranged is execution, and pinning it in the PRD spends your pages on the agent's job.

4. A classmate's PRD has goals, stories, constraints, and criteria — but no non-goals section, “because we listed what we ARE building, so everything else is obviously out.” What's the strongest counter?

C. Goals bound what must exist; they don't bound what an eager implementer adds — a flashcard app's goals are perfectly consistent with an unrequested stats dashboard. The scope fence has to be explicit because the agent fills silence with plausibility (M13), and “plausible extra features” is scope creep in its natural form — which is also why M17 makes an empty non-goals section a critique flag.
