Agentic Coding with Claude Week 3 · Objective M18

Give the Agent a Target

“Claude stops when the work looks done.” The highest-leverage upgrade to any prompt is making “done” checkable — a test, a build, an expected output, a screenshot to compare. This lesson is why a verifiable target beats open-ended prompting, in the official docs' own words.

M18 · Foundations Bloom: Understand Assessed by: Quiz 3

Objective: Describe how giving Claude a verifiable target (tests, mock, expected output) improves results over open-ended prompting.

1 · The looks-done problem

Anthropic's best-practices guide states the mechanism precisely:

From the official docs

“Claude stops when the work looks done. Without a check it can run, ‘looks done’ is the only signal available, and you become the verification loop: every mistake waits for you to notice it.”

Unpack the two halves. “Looks done” is a plausibility judgment — the same faculty that produces confident wrong answers in a polluted context (Week 2) is grading the work. And “you become the verification loop” is the cost model: every defect travels all the way to your review before anyone notices, and every correction costs a full human round-trip. You've already watched it happen: in Week 1's transcript, Claude announced “Done — both tests pass” while the behavior was wrong — the tests passed because they encoded the agent's own assumption.

2 · Closing the loop

The fix: “Give Claude something that produces a pass or fail, and the loop closes on its own. Claude does the work, runs the check, reads the result, and iterates until the check passes.” The check is anything that returns a signal the agent can read in the conversation:

  • a test suite (including example cases pasted right into the prompt),
  • a build or linter exit code,
  • a mock or expected output — a script that diffs the program's output against a fixture,
  • a browser screenshot compared against a design, for UI work.

The official before/after examples are worth internalizing verbatim — the quiz asks for exactly this transformation:

StrategyOpen-endedWith a verifiable target
Provide verification criteria“implement a function that validates email addresses”“write a validateEmail function. example test cases: user@example.com is true, invalid is false, user@.com is false. run the tests after implementing”
Verify UI changes visually“make the dashboard look better”“[paste screenshot] implement this design. take a screenshot of the result and compare it to the original. list differences and fix them”
Address root causes“the build is failing”“the build fails with this error: [paste error]. fix it and verify the build succeeds. address the root cause, don't suppress the error”

3 · Why it beats open-ended prompting

Same task, two regimes:

Open-ended promptVerifiable target
Definition of done“Looks done” — the agent's plausibility judgmentThe check passes — unambiguous, external to the agent
Where errors surfaceAt your review, or in productionInside the session, as a failing check the agent reads
Cost of an iterationA human round-trip per correctionThe agent iterates alone until the signal is green
Ambiguity resolutionSilent agent choice (M13's failure mode)Pinned in advance by the target's examples
Supervision required“A session you watch”“One you walk away from”

The docs compress the stakes into one line: a runnable check is “the difference between a session you watch and one you walk away from.” Pocock's workshop makes the same point from the TDD side: give the agent a failing test and it will “produce code that provably solves the problem you specified” — the test is an unambiguous success definition, eliminating the interpretation gap that open-ended prompts leave open. Notice the validateEmail example doing double duty: those three test cases don't just verify, they decide — user@.com is false settles a boundary question that an open-ended prompt would have left to the agent (three test cases are also a tiny grill session).

One discipline completes the loop: evidence over assertion. “Have Claude show evidence rather than asserting success: the test output, the command it ran and what it returned, or a screenshot of the result.” Reviewing evidence is faster than re-running verification yourself — and it's the only version that works for sessions you weren't watching. When an agent claims success without evidence, the Week 1 transcript is your reminder of what that claim is worth.

4 · From acceptance criteria to targets

This objective sits in Week 3 for a reason: your PRD's acceptance criteria are verifiable targets written in prose. “Grading a card ‘easy’ removes it from today's queue” is one mechanical step from a test; “the app should work correctly” can never become one — which is why M17 treats uncheckable criteria as the highest-value critique flag. The pipeline this course builds, week by week: grill answers → acceptance criteria (this week) → failing tests the agent codes against (Week 5, test-first execution) → checks that gate long-running sessions (Week 8). For now, the habit that starts paying immediately: every prompt that asks for behavior names the check that verifies it.

Escalation ladder (names only, for later weeks)

The docs describe four levels of gating once a check exists: ask for it in the prompt (works today, no setup) → set it as a /goal condition re-checked every turn → enforce it with a Stop hook that blocks the session from ending until the check passes → have a fresh subagent try to refute the result. Each trades setup for attention. Weeks 8–9 climb this ladder; this week you only need rung one.

Self-check

1. Which prompt gives the agent a verifiable target?

C. It supplies pass/fail examples and orders the check run — the validateEmail pattern. B gestures at verification but defines neither “works” nor the check, so “looks done” still decides; D delegates the target's content to the agent, which means the tests will encode the agent's assumptions — exactly what the Week 1 transcript showed going wrong. Note C's third case is also a decision: rejecting 'tomorrow' was pinned by you, not resolved silently.

2. Claude reports: “Refactor complete — everything passes.” The docs-endorsed next move is:

B. Evidence over assertion: “the test output, the command it ran and what it returned.” Reviewing evidence is explicitly framed as faster than re-running verification yourself (C's cost, paid on every task, is the you-as-verification-loop regime the target was supposed to end) — and A trusts the exact “looks done” signal this lesson exists to replace. D just requests a second assertion.

3. “Make the dashboard look better” keeps producing designs you don't want. Applying this lesson, the fix is:

D. The official UI pattern: visual work gets a visual target, and the compare-and-fix instruction closes the loop the same way a test suite does for logic. C is more open-endedness, not less — adjectives are the ambiguity M17 taught you to flag; A and B keep you as the verification loop, just louder or five times per iteration.

4. Why does the course insist your PRD's acceptance criteria pass the stranger test this week, when tests aren't written until Week 5?

A. The pipeline runs criteria → tests → gates, and its quality is set at the first stage: “grading ‘easy’ removes the card from today's queue” converts to a failing test in one step, while “works correctly” converts to nothing in any number of steps. Vague-now-precise-later (D) means re-deriving the decisions in Week 5 — without the grill session context that produced them.
