Agentic Coding with Claude Week 4 · Objective M21

Tracer-Bullet Decomposition

Your PRD marks the destination. This lesson is about cutting it into 6–10 issues that are each a thin, complete path through every layer of the system — tracer bullets that light up whether your architecture actually works, one demoable slice at a time.

M21 · Planning Bloom: Create Assessed by: Assignment A4

Objective: Decompose a PRD into 6–10 vertical-slice ‘tracer bullet’ issues that each cross every layer and emit observable feedback.

1 · Two ways to slice a build

Imagine your flashcards app as a stack of layers: data storage, application logic, UI, tests. There are two ways to decompose the PRD into work items:

  • Horizontal slices ship one layer at a time: “build the storage layer,” “build all the endpoints,” “build the UI.” Each item sounds clean — and nothing works until every layer lands. Integration problems hide until the end, when they're most expensive.
  • Vertical slices ship one narrow path through all the layers at once: “user can add a card and see it persist.” Tiny scope, but storage, logic, and UI each exist in miniature, connected — so the moment the slice is done, you can demo it.

Pocock's term for a vertical slice — borrowed, as he notes, from The Pragmatic Programmer — is the tracer bullet: a small, end-to-end slice of functionality that spans all system layers simultaneously. Tracer rounds glow in flight so a gunner can see where fire is actually landing; a tracer-bullet issue does the same for your architecture. Build one tiny vertical slice, validate it, then expand outward.

2 · Why this controls AI slop

Left unconstrained, an agent handed a whole PRD “tends toward producing enormous complete solutions without validation” — hundreds of horizontal-layer files that have never once run together. Pocock's tracer-bullets essay frames the fix as forced feedback: build small, test immediately, gather feedback, then expand. Instead of discovering architectural problems after massive code generation, you “test assumptions continuously, identify issues early, and proceed with confidence that foundational approaches actually work.”

The objective's phrase “emit observable feedback” is the operative requirement. Every slice must end in something you can observe: a behavior you can click through, a test that passes, output you can inspect. That's Week 3's M18 (give the agent a target) fused with Week 2's smart zone: a slice small enough to verify is also small enough to fit one fresh session.

His worked example

Building a “Reveal in File System” feature, Pocock didn't wire it into every surface at once: he created the backend endpoint and connected it to one UI location first, validated the core path worked, and only then expanded to the three other UI locations. First slice proves the path; later slices widen it.

3 · The slicing rules

The course's decomposition rules come from Pocock's to-tickets skill, verbatim:

Vertical-slice rules (mattpocock/skills, to-tickets)
- Each slice cuts a narrow but COMPLETE path through every layer
  (schema, API, UI, tests) — vertical, NOT a horizontal slice of
  one layer
- A completed slice is demoable or verifiable on its own
- Each slice is sized to fit in a single fresh context window
- Any prefactoring should be done first

Plus A4's sizing constraint: 6–10 issues for your whole PRD. That number is doing real work. Fewer than six and your slices are probably multi-session epics; more than ten and you're probably slicing thinner than the overhead of a branch-PR-review cycle per issue is worth. Each issue then gets the M20 anatomy: what-to-build, acceptance criteria, blocked-by.

Two terms from the rules deserve expansion:

  • Prefactoring — if the codebase needs reshaping to make the feature slices easy, that reshaping is its own slice, ordered first: “make the change easy, then make the easy change.”
  • The wide-refactor exception — one mechanical change with a blast radius across the whole codebase (rename a column, retype a shared symbol) can't land green as any vertical slice. The skill sequences it as expand–contract: add the new form beside the old, migrate call sites in batches (each batch its own issue), then delete the old form once no caller remains. You likely won't need this for a fresh course project — but recognize it, because “this work resists vertical slicing” is a real category, not a rule violation.

4 · Worked decomposition

The Week 3 sample PRD — a flashcards app with decks, card-flipping, and spaced repetition, no accounts, localStorage — decomposed into seven tracer bullets:

A 7-issue backlog (each issue would carry full M20 anatomy)
1. Walking skeleton: app shell renders, one hard-coded deck displays,
   deployed and openable in a browser            [blocked by: none]
2. User can create a deck and it survives reload (localStorage
   round-trip)                                   [blocked by: 1]
3. User can add cards to a deck and see them listed
                                                 [blocked by: 2]
4. User can flip a card to reveal its answer     [blocked by: 3]
5. User can mark a card right/wrong and the result is stored
                                                 [blocked by: 4]
6. Review scheduling: "due today" queue derived from results
                                                 [blocked by: 5]
7. User can delete decks and cards               [blocked by: 3]

Check it against the rules: every issue crosses data, logic, and UI (even #1 — a deployed shell is the thinnest possible full path, sometimes called a walking skeleton); every issue ends in something observable (“survives reload,” “see them listed”); none is a layer (“build the storage module” appears nowhere — storage gets built inside #2, to exactly the depth #2 needs, then deepened by #5). And #7 hangs off #3, not #6 — its blocked-by edge records that deletion doesn't care about scheduling, which is what will let slices run in parallel in M22.

Decomposition exercise + A4 start

This is the Create-level heart of Assignment A4 — do the drill, then the real thing:

Self-check

1. A classmate's backlog: 1. Design the database schema. 2. Build all API endpoints. 3. Build the UI. 4. Write tests. 5. Integration. What's the core critique?

B. This is the textbook horizontal decomposition: clean-sounding items, zero observable feedback until the end, and an “integration” issue that is really “discover all the problems at once.” A misses the point — renumbering layers doesn't make them vertical. C is a real instinct but inside each slice, not as a global layer of its own. D confuses the system's structure with the work's structure — the whole insight is that they shouldn't match.

2. Which issue best satisfies “emits observable feedback”?

C. The round-trip is directly observable and even self-verifying: export, re-import, compare. You (or the agent) can check “done” without trusting anyone's assertion. A, B, and D all describe internal virtues with no defined observation that would distinguish done from not-done — as targets they'd reduce to “looks done,” the signal M18 taught you never to rely on.

3. Your first drafted slice is “User can study a full session: flip cards, mark answers, see tomorrow's queue.” It crosses every layer and is demoable. What rule does it likely still break?

A. Vertical is necessary, not sufficient — the third rule is sizing, and this slice bundles flipping, scoring, and scheduling, each independently demoable. The fix is to split it vertically (flip / mark / queue, as in the worked backlog), never by layer (D re-creates the horizontal problem at smaller scale). B forgets the context-window rule that makes slices agent-runnable at all.

4. You need to rename a core field used across the whole codebase — hundreds of call sites. Forcing it into one vertical slice would mean a giant red diff that can't land green. The course-endorsed move is:

D. This is the wide-refactor exception, sequenced exactly as the to-tickets skill prescribes: expand, migrate in blast-radius-sized batches, contract — the old form's continued existence is what keeps every intermediate state green. A produces the unlandable mega-diff the exception exists for; C hides a refactor inside unrelated diffs, sabotaging review; B pretends the need away.
