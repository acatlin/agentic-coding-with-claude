One Issue per Session

Everything this week built converges on a single working rule: one issue, one fresh session, one branch, one PR. This lesson explains why the rule holds — one argument from the context window, one from the reviewer's chair.

M24 · Judgment Bloom: Understand Assessed by: Quiz 4

Objective: Explain why one-issue-per-session keeps the agent in the smart zone and simplifies review.

1 · The rule

When implementation starts in Week 5, each unit of work runs as: pick one ready-for-agent issue from the frontier → start a fresh session → agent works the issue on its own branch → PR with Closes #N → review → merge → issue closes → next issue, next fresh session.

Nothing in the tooling forces this. You could feed an agent three issues in one conversation, or let a session that finished issue #4 “just continue” into #5. The rule says don't — for two independent reasons, either of which would justify it alone.

2 · Argument one: the smart zone

Week 2 established the mechanics: the context window holds everything — every message, file read, and command output — it fills fast, and “LLM performance degrades as context fills.” The smart zone is the early, sharp stretch of a session; the dumb zone is what a full window buys you.

Now recall the third vertical-slice rule from M21: each slice is “sized to fit in a single fresh context window.” That sizing rule and this session rule are two halves of one design:

  • Decomposition sizes the work to the window — an issue is, by construction, an amount of work an agent can complete while still sharp.
  • One-issue-per-session spends a whole window on it — the issue gets the smart zone it was sized for. Start issue #5 in issue #4's session and #5 runs in whatever degraded remainder #4 left behind; its careful sizing bought nothing.

There's a contamination argument stacked on the budget argument. Anthropic's best-practices guide names the “kitchen sink session” as a leading failure pattern — one task, then an unrelated one, then back, until “context is full of irrelevant information” — and its standing fix is /clear between tasks, because “a clean session with a better prompt almost always outperforms a long session with accumulated corrections.” Your backlog makes the guidance mechanical: issue boundaries are the task boundaries, decided once at decomposition time instead of re-judged mid-flow. And the fresh session isn't flying blind — the durable context (Week 2's memento principle) is all in the repo: PRD.md, CLAUDE.md, the merged code of every prior slice, and the issue itself.

3 · Argument two: review

Follow the chain: one issue per session means one branch per issue means the PR's diff contains exactly one issue's work. That transforms the reviewer's job:

  • The diff has a spec. The linked issue's acceptance criteria are the review checklist — walk the checkboxes against the diff. A three-issue PR makes you review against a mental union of specs, and scope creep hides in the seams between them.
  • “Out of scope” becomes decidable. With one issue linked, any changed file that serves no criterion is a flag you can raise mechanically. Anthropic's model review prompt in the best-practices guide checks precisely this: “every requirement is implemented, the listed edge cases have tests, and nothing outside the task's scope changed.”
  • Verdicts stay clean. A one-issue PR can be approved, rejected, or sent back as a unit. A bundle where slice A is right and slice B is wrong forces either partial-merge surgery or rejecting good work.
  • The record stays legible. Merge closes exactly the issue that was done (M20's closing keywords); the PR-linked session (M19b) holds the reasoning for exactly this slice. Six weeks later, “why does this code exist” has a one-issue answer.
Same principle, next scale

The best-practices guide notes that “a fresh context improves code review since Claude won't be biased toward code it just wrote” — its adversarial-review pattern gives a reviewer subagent only the diff and the criteria, not the conversation that produced the change. One-issue-per-session is that same isolation principle applied to production rather than review: bounded context in, bounded reviewable artifact out. Week 6 builds the review half; Week 8 scales the production half to parallel sessions — which only works because no two frontier issues ever share a session.

4 · What the rule is not

Understanding a rule includes knowing its edges:

  • It's not “one prompt per session.” Within the issue, iterate freely — run the checks, fix failures, course-correct. The boundary is the issue, not the message count.
  • It's not “never talk about two issues.” Planning sessions (like the one that drafted your backlog in M23) legitimately hold the whole board. The rule governs implementation sessions, where the work product is a diff.
  • It's not free. Each fresh session re-reads project context, and each issue pays branch-PR-review overhead. That cost is real — it's why M21 caps decomposition at 6–10 issues rather than fifty. The rule and the sizing rule justify each other; adopt them together or the math stops working.

Self-check

1. A session just finished issue #4; tests pass, PR opened. The agent still “knows” the codebase intimately, and issue #5 touches the same files. Restarting fresh feels wasteful. What does the restart actually buy?

C. The felt waste is mostly illusion: what the old session “knows” splits into durable context (in the repo — the memento principle means the fresh session recovers it) and conversational residue (file dumps, dead ends, #4's corrections — which is contamination, not knowledge). Slices were sized to a fresh window; spending a used one on #5 silently breaks that contract. B names a real but minor risk; it isn't the argument.

2. To “save review overhead,” a classmate has one session implement issues #5, #6, and #7, then opens a single PR: “Closes #5, closes #6, closes #7.” The keywords are correct. What did the bundling actually cost?

B. Review cost isn't proportional to PR count; it's proportional to how hard each verdict is. Three specs sharing one diff means no changed line has an unambiguous owner, and approve/reject stops being a unit decision. (C is false — M20 showed multi-issue closing syntax works fine; the mechanics allow bundling, which is exactly why the discipline has to come from the workflow.) The classmate saved two PR creations and spent the decidability of the review.

3. Which pair correctly names the two independent arguments for one-issue-per-session?

D. One argument faces the agent (context budget + contamination), one faces the human (a decidable review unit) — and they're independent: even a hypothetical never-degrading agent would leave the review argument fully intact, and even with no reviewer the smart-zone argument would stand. A's factors are side effects, not reasons; B inverts causality (the tooling permits bundling; the workflow forbids it).

4. The rule exists because of a matching rule from M21. Which one, and why do they only work together?

A. The two rules are one contract seen from both sides: decomposition promises “this fits a fresh window,” the session discipline promises “it gets a fresh window.” Each is the other's reason. B and C are real rules solving different problems (ordering, reference integrity); D's count is a consequence of per-issue overhead, not a session-scheduling constraint.
