Branches and Merges

A branch is a contained area of the repository where work can happen without touching anything else — which is exactly what delegated agent work needs. This lesson covers what branches are, how merges bring work back, and why this course adopts a branch-per-issue convention.

M19 · GitHub Bloom: Apply Assessed by: Quiz 4 · Assignment A4

Objective: Explain branches, merges, and the pull request lifecycle; create a feature branch and open a PR. Part 1 of 2 — branches & merges

1 · What a branch is

GitHub's docs put it in one sentence: branches “let you develop features, fix bugs, or safely experiment with new ideas in a contained area of your repository.” A branch is not a copy of the project — it's a movable label pointing at a line of commits. Two branches can share almost all their history and differ only in the commits made since they diverged.

Every repository starts with one default branch, commonly named main. It's the branch you see when you visit the repository, the branch checked out when someone clones it, and — unless you choose otherwise — “the base branch for new pull requests and code commits.” Treat main as the version of the project that is always presentable: everything on it has been reviewed and merged deliberately.

Branches you create for a specific piece of work are called feature branches (GitHub also says topic branches). They're typically created from the default branch, carry a handful of commits, and are deleted once merged. Creating one requires write access to the repository — which you have on your own course-project repo.

Why agents make this matter more

In Week 1 you worked directly on main, and for a solo first repo that was fine. But from this week, an agent is doing multi-file work on your behalf — and work you didn't type yourself deserves a containment area. A feature branch means the agent's changes exist somewhere reviewable before they exist on the version that counts. If the work goes wrong, you delete a branch, not untangle main.

2 · Creating and switching branches

The two commands you need, runnable yourself or delegatable to Claude:

Create a branch and switch to it
git switch -c 12-deck-import     # create branch "12-deck-import" and switch to it
git switch main                  # go back to main
git branch                       # list branches; * marks the current one

This course's convention — adopted formally in Assignment A4 — is branch-per-issue: one branch for each GitHub issue, named after it (issue number plus a short slug, like 12-deck-import). The branch name then answers “what is this work?” by pointing at the issue that specifies it. You'll see in M20 how the issue carries the acceptance criteria; the branch is where the agent works toward them.

3 · Merging: bringing work back

A merge takes the commits from one branch and incorporates them into another. On GitHub you'll almost always merge through a pull request (next lesson), and when you do, GitHub offers three merge methods:

MethodWhat it doesWhen it fits
Merge commitPreserves every commit from the branch and adds an explicit merge pointWhen individual commits have meaningful context worth keeping
Squash and mergeCombines all the branch's commits into a single commit on the base branchWhen the PR is one logical change with many work-in-progress commits — a common shape for agent branches
Rebase and mergeReplays each commit onto the base branch with no merge commit, for a linear historyWhen the team wants linear history and the commits are already well organized

For the course project, squash and merge is a sensible default: an agent working an issue tends to produce several incremental commits, but what main needs to record is “issue #12 was implemented.” One issue, one branch, one squashed commit — history that reads like your backlog.

GitHub's own workflow guidance is to merge the base branch into long-lived topic branches frequently, and — better — to merge pull requests promptly, keeping contributions small. Small, quickly-merged branches barely have time to drift from main, which is how you avoid most merge conflicts before they exist. The vertical-slice issues you'll write in M21 are sized to make exactly this possible.

4 · Protected branches

GitHub lets repository administrators mark critical branches as protected. Protection rules can block force pushes, prevent the branch from being deleted, require status checks to pass, require reviews before merging, or require signed commits. On a team repo, protecting main is what makes “everything on main was reviewed” a guarantee instead of a habit. For a solo course repo you don't need protection rules — but you should know they exist, because they're the enforcement layer for every convention this week teaches by discipline.

Ten minutes in your course-project repo — the mechanics you'll use for every A4 issue:

Self-check

1. You ask Claude to implement issue #7 and it proposes committing directly to main. The best reason to redirect it to a feature branch first is:

C. That's the containment argument, and it's why branch-per-issue is this course's convention for agent work. A is false — commits are equally permanent everywhere; branches differ in what points at them, not in mutability. B is nonsense (branches share history). D is false — you can push to main on your own repo, which is exactly why the discipline has to come from you.

2. An agent branch for one vertical-slice issue has nine commits: “wip”, “fix test”, “actually fix test”… The work is done and reviewed. Which merge method best fits, and why?

B. GitHub's docs recommend squash for exactly this shape: a PR representing one logical change with many small fixup commits. The branch still exists in the PR record if you ever need the blow-by-blow. A and C preserve nine commits nobody will want to read individually; D abandons version control's whole value — attribution, review linkage, and revertability.

3. Why does GitHub's guidance to “merge pull requests promptly” matter more in an agentic workflow than a solo-human one?

D. This week's whole workflow (6–10 vertical-slice issues, branch per issue) multiplies the number of concurrent branches — so drift you'd tolerate on one long-lived solo branch becomes compounding conflict risk across the backlog. Prompt merging keeps every slice's diff small against a fresh main. B is false (agents can resolve conflicts, it just burns time and review attention); C confuses git state with conversation state.

4. On a shared team repo, what makes protected branches different from the branch-per-issue convention you're practicing?

A. Per GitHub's docs, protection rules block force pushes, prevent deletion, and can require status checks and reviews before merging — the platform enforces them regardless of anyone's discipline. That's the same advisory-versus-guaranteed distinction Week 2 drew between CLAUDE.md instructions and hooks. B is wrong (merging via reviewed PRs is exactly what protection channels you into); C is invented.
