File Issues with Claude + gh CLI

You've decomposed and sequenced the backlog; now it goes on GitHub. This is a delegation task with a review gate in the middle: Claude drafts the issues, you check every draft for accuracy, then the gh CLI files them — blockers first, so the edges reference real numbers.

M23 · Tooling Bloom: Apply Assessed by: Assignment A4

Objective: Use Claude Code with the gh CLI to draft and file issues, and review the drafts for accuracy before submitting.

1 · Why gh is the right tool

Anthropic's best-practices guidance is direct: “If you use GitHub, install the gh CLI. Claude knows how to use it for creating issues, opening pull requests, and reading comments.” Two properties make CLIs the preferred bridge to external services:

  • Context efficiency. The docs call CLI tools “the most context-efficient way to interact with external services” — a gh issue create round-trip costs a few lines of your session's context, where a browser-automation detour would cost screenshots and page dumps.
  • Authentication. Without gh, Claude falls back to GitHub's API, where “unauthenticated requests often hit rate limits.” gh auth login (once, interactive — run it yourself in a terminal, not through Claude) gives every future session authenticated access.
The gh commands underneath
gh auth status                          # confirm you're logged in
gh label create ready-for-agent         # one-time label setup
gh issue create --title "..." --body "..." --label ready-for-agent
gh issue list                           # verify what got filed
gh issue view 12                        # read one back, with body

You won't usually type the create lines yourself — Claude will. But you should recognize them, because reviewing delegated work means being able to read what the delegate is about to run.

2 · The workflow: draft → review → file

Starting from your M21/M22 drafts (or fresh from the PRD), in a fresh session in your project repo:

Step 1 — draft
Read PRD.md. Here are my 8 sequenced issue drafts with blocked-by
edges: [paste]. Format each as a GitHub issue: title naming the
end-to-end behavior, a "What to build" section from the user's
perspective, acceptance criteria as checkboxes, and a "Blocked by"
section. Show me all 8 drafts. Do NOT file anything yet.

The explicit “do not file yet” matters: it inserts you between drafting and publishing. This is the same pattern the to-tickets skill builds in — before publishing, it “quizzes you on the breakdown (granularity, blocking edges, what to merge or split)” — and the same review-before-submit rule the PR docs gave you in M19b. Publishing is the irreversible-ish step (issues can be edited, but agents may have already grabbed them); review happens before it.

Step 2 — review every draft (the A4 requirement)

A4's wording is “reviewing every draft before submission,” and the review is for accuracy against your PRD and your decomposition — not prose style. Concretely, per draft:

  • Traceability: does every behavior claimed trace to the PRD? Drafting is where an agent most naturally embellishes — a “reasonable” extra criterion here is scope creep with your name on it.
  • Criteria fidelity: are the acceptance criteria yours — the ones your decomposition settled — not weakened (“works correctly”) or silently strengthened?
  • Edge fidelity: do the blocked-by edges match your sequencing? An added edge serializes your frontier; a dropped one lets an agent start work whose foundation doesn't exist.
  • Anatomy check (M20): behavior-named title, user-perspective body, no file paths or speculative code.
Step 3 — file, blockers first
Drafts 1–8 are approved with my edits. File them with gh in
dependency order (blockers first) so each "Blocked by" section can
reference the actual issue numbers of the already-filed blockers.
Apply the ready-for-agent label. Then run gh issue list and show me
the result.

The ordering rule comes straight from to-tickets: publish “in dependency order (blockers first) so each ticket's blocking edges can reference real identifiers.” File #1 first and issue #2's body can truly say “Blocked by #1”; file them in reverse and every edge is a dangling promise. The final gh issue list is your verification step — evidence, not assertion, that the board matches the plan.

Accountability, restated

Week 1's M05 gave you the rule: the human owns what ships under their name. Filed issues are published artifacts in your repo — each one is a claim about what your project will do, and next week they become instructions to implementation agents. An unreviewed inaccurate issue doesn't just sit there; it gets built. The review gate is where a five-second read prevents a session of misdirected work.

Hands-on checklist

Self-check

1. Reviewing Claude's draft for your “flip a card” issue, you find a criterion you never specified: “[ ] Flip animation runs at 60fps.” It's a perfectly reasonable idea. What do you do, and why?

B. This is exactly what the accuracy review exists to catch: plausible additions are the failure mode, because implausible ones would catch themselves. The issue isn't whether 60fps is good — it's whose decision it is. Adopting it deliberately is fine; letting it through unread is scope creep that a Week 5 session will faithfully build. C files an ambiguity (what does an agent do with “optional”?); D overcorrects — the gate caught it, which means the gate works.

2. Why does the workflow file issues blockers-first rather than in whatever order the drafts sit?

C. The to-tickets rule verbatim: publish in dependency order “so each ticket's blocking edges can reference real identifiers.” Dangling references (“blocked by the storage issue, whatever number that becomes”) are precisely the kind of loose end that breaks when an agent — which follows references literally — reads the board. A is false (GitHub happily accepts any #N in text); D confuses filing order with board layout, which is configurable anyway.

3. A classmate skips gh: “Claude can just open Chrome and click through the New Issue form ten times.” It would work. What's the strongest case for gh anyway?

A. Both halves come from the best-practices doc: CLI tools are “the most context-efficient way to interact with external services,” and evidence-over-assertion is the standing verification rule. Ten form fills burn context (Week 2 taught you what that costs the rest of the session) and leave nothing greppable to verify against your plan. B and C are false; D ignores that “would work” and “works efficiently and verifiably” are different bars.

4. Claude reports: “All 8 issues filed successfully with labels and blocking references.” Per this course's habits, what — if anything — remains?

D. Draft review checked the content; this checks the execution — a partial failure (a create that errored, a label that didn't exist, an edge referencing the wrong number) can hide behind a confident summary. “Have Claude show evidence rather than asserting success” is the best-practices rule, and it's cheap here: one list command against an 8-line plan. B duplicates completed review at the wrong altitude; A conflates the two gates this lesson deliberately keeps separate.
