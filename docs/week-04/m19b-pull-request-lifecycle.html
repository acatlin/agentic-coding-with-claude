The Pull Request Lifecycle

A pull request is a proposal to merge changes, wrapped in a place to discuss them. For agentic work it's something more: the checkpoint where AI-produced code waits, in full view, for a human decision. This lesson walks the lifecycle from branch to merge — then you open one for real.

M19 · GitHub Bloom: Apply Assessed by: Quiz 4 · Assignment A4

Objective: Explain branches, merges, and the pull request lifecycle; create a feature branch and open a PR. Part 2 of 2 — the PR lifecycle

1 · What a pull request is

GitHub's definition: pull requests are “proposals to merge code changes into a project,” and they're the platform's central collaboration tool — they let teams “discuss and review changes before merging them,” which “helps teams work together, catch issues early, and maintain code quality.”

Notice what a PR adds on top of a plain merge: a pause with a conversation in it. The branch (M19a) contains the work; the PR is the request for that work to be judged. In this course's terms, the PR is where the review activity — one of the three activities the whole course is built on — actually happens. Weeks 6 and 9 live inside this pause; this week you just need the mechanics.

2 · The lifecycle, step by step

  1. Branch and commit. Work happens on a feature branch — in our convention, the branch for one issue.
  2. Open the PR. You propose merging the feature branch (the head branch) into main (the base branch), with a title and description. The description is where you tell the reviewer — often future-you — what this change is and which issue it resolves (M20 covers the linking keywords).
  3. Discussion and review. Reviewers comment, request changes, or approve. Crucially, new commits pushed to the head branch automatically update the open PR — you respond to review feedback by pushing fixes, not by opening a new PR.
  4. Checks run. Automated validations (tests, linters, CI) report pass/fail on the PR itself.
  5. Merge. Someone with write access merges — merge commit, squash, or rebase (M19a) — and the head branch can be deleted. If the PR's description linked an issue with a closing keyword, merging into the default branch closes that issue automatically.

The PR page organizes all of this into tabs — GitHub currently shows five: Conversation (description, timeline, comments, reviews), Commits (how the branch evolved), Checks (automated validations), Files changed (the diffs — where review actually happens), and Findings (code-scanning results). A merge-status indicator shows what's still blocking: missing approvals, failing checks.

Draft PRs

A PR can be opened as a draft: it “cannot be merged,” and code owners aren't automatically asked to review it. Drafts let you share work-in-progress without requesting formal review, then mark it “ready for review” when it's time. In agentic work this maps neatly onto “the agent thinks it's done, but I haven't looked yet” — open as draft, review the diff yourself, then promote.

3 · Opening a PR with Claude

Claude Code creates pull requests on request — the Common Workflows docs show it as a one-liner: after the work is committed, say create a pr. Claude uses the gh CLI underneath (gh pr create), writes a summary of the changes as the description, and you can refine it: “enhance the PR description with more context about the security improvements.”

The step-by-step version (from code.claude.com Common Workflows)
> summarize the changes I've made to the authentication module
> create a pr
> enhance the PR description with more context about the security improvements

Two facts worth keeping:

  • Sessions link to PRs. When a PR is created via gh pr create, the Claude Code session is automatically linked to it — later, claude --from-pr 1234 reopens the session picker filtered to sessions tied to that PR. The conversation that produced the code stays findable from the code.
  • Review before submitting. The docs' own tip: “Review Claude's generated PR before submitting.” The description is a claim about what the diff does — and you are accountable for claims made under your username. This is the same review-the-draft rule M23 applies to issues.

4 · Two development models

GitHub describes two ways PRs get used. In the shared repository model, everyone has push access to one repository, creates branches in it, and uses PRs for review before merging — this is your course project, and most professional team repos. In the fork and pull model, contributors fork the repository, work in their own copy, and submit PRs without needing upstream permissions — this is how open source accepts changes from strangers. Same lifecycle, different trust boundary. Knowing which model a repo runs tells you whether a branch or a fork is your starting move.

Self-check

1. A reviewer requests changes on your open PR. Claude has made the fixes on the same feature branch. What happens next?

B. A PR tracks its head branch, so pushed commits appear in the existing PR — the Commits tab shows the branch's evolution over time precisely because this is the normal flow. A fragments one logical change across two proposals; C is unnecessary ritual; D merges before review completes, defeating the pause the PR exists to create.

2. Claude finishes an issue overnight and you want the work visible on GitHub — but you haven't read the diff yet. The mechanism that best fits this exact state is:

C. Draft status encodes “exists, not yet endorsed” — exactly the gap between agent-done and human-reviewed. A treats passing checks as equivalent to review (Week 3's M18 told you a check is a floor, not a verdict). B hides the work and loses the PR's diff view for your own reading; D manufactures a false endorsement, which is worse than no review signal at all.

3. Six weeks from now you're staring at a merged PR wondering why the code does something odd. Which fact from this lesson helps most?

D. The session link is the answer to “what was the agent thinking?” — the reasoning, alternatives considered, and instructions given all live in the linked conversation. A is wrong (Findings shows code-scanning results, not rationale); B is wrong (squash condenses commits — though the PR page itself still shows them); C names a real UI element but approval names don't explain oddities.

4. You want to contribute a fix to an open-source library you have no permissions on. Applying this lesson's models:

A. GitHub's fork and pull model exists exactly so open-source projects can accept proposals from contributors without granting write access. B inverts the trust model (access is the thing you don't have and don't need); D is what the missing write access prevents; C abandons the review, checks, and discussion machinery the PR provides for free.
