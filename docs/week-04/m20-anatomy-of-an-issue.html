Issues as Units of Agent Work

GitHub built issues for humans to plan, discuss, and track work. This course uses them for something sharper: an issue is the unit of work you hand an agent — a title, a body with acceptance criteria, labels that say what state it's in, and a link that closes it when the PR merges.

M20 · GitHub Bloom: Apply Assessed by: Assignment A4

Objective: Explain GitHub issues as units of agent work: titles, bodies with acceptance criteria, labels, and linking issues to PRs.

1 · What GitHub gives you

Per GitHub's docs, issues are flexible tracking tools to “plan, discuss, and track work” — bug reports, feature requests, ideas, team discussion. They can be created from the web interface, GitHub Desktop, the gh CLI, the APIs, or mobile; even from a comment on another issue or a specific line of code. Around each issue, GitHub provides organizing machinery:

  • Labels, issue types, and milestones to classify and group work;
  • Assignees to make responsibility explicit;
  • Sub-issues to break large tasks into a hierarchy, and dependencies to record which issues block which;
  • Cross-references: #number links issues anywhere in comments and PRs, and task lists (- [ ] checkboxes) track steps inside a body;
  • Templates and issue forms so a repo can require structure from whoever files.

All of that predates AI agents. What's changed is who reads the issue next.

2 · The issue as an agent's work order

When the next reader of an issue is an agent in a fresh session, the issue body is the specification for that session — the agent has your PRD (committed in Week 3) plus this issue, and nothing else. That raises the bar on each part:

PartFor a human teamFor an agent
TitleFindable in a listNames the end-to-end behavior delivered — “User can import a deck from CSV,” not “CSV work”
BodyEnough context to discussWhat to build, described as behavior from the user's perspective — not a layer-by-layer implementation list
Acceptance criteriaDefinition of done, often implicitA checklist of verifiable statements — the target (Week 3's M18) the agent iterates toward and the reviewer checks against
LabelsCategorizationWorkflow state — e.g. a ready-for-agent label meaning “spec complete; safe to hand to a session”
LinksRelated discussionBlocking edges to other issues, and the closing link from the PR that implements it

Pocock's to-tickets skill — the source of this course's issue format — writes each issue with exactly three sections: What to build (“the end-to-end behaviour this ticket makes work, from the user's perspective — not layer-by-layer implementation”), Acceptance criteria (checkboxes), and Blocked by (references to blocking issues, or “None — can start immediately”). Issues that are fully specified get the ready-for-agent label: “agent-grabbable by construction.”

Keep file paths out

The same skill warns: avoid specific file paths and code snippets in issue bodies — “they go stale fast.” The agent will explore the codebase fresh when it picks the issue up; paths written at decomposition time are guesses that rot. The documented exception: a snippet from a prototype that encodes a decision more precisely than prose can (a schema, a state machine) — trimmed to the decision-rich parts.

3 · A worked example

An agent-ready issue for the Week 3 flashcards project
Title: User can flip a card to reveal its answer

## What to build
From a deck's study view, clicking a card flips it to show the
answer side; clicking again flips it back. This is the app's core
interaction and must work before any scheduling logic matters.

## Acceptance criteria
- [ ] Clicking a face-down card reveals its answer within 100 ms
- [ ] Clicking a revealed card returns it to the question side
- [ ] Flip state is per-card: flipping one card does not affect others

## Blocked by
- #3 (Deck view renders cards from stored data)

Labels: ready-for-agent

Read it as the agent will: every sentence either describes observable behavior or points at a dependency. Nothing prescribes how — no component names, no file paths — because the how is the agent's job and the PRD plus codebase supply the context. Compare this against the “stranger test” your Week 3 acceptance criteria had to pass: same test, smaller scope.

4 · Linking issues to PRs

The loop closes with GitHub's linking mechanism. Put a closing keyword plus the issue number in a PR's description — the supported keywords are close, closes, closed, fix, fixes, fixed, resolve, resolves, and resolved — and the issue is linked to the PR:

In the PR description
Closes #12

When a PR with a linked issue merges into the default branch, the linked issue is automatically closed. (PRs targeting other branches don't trigger this.) Cross-repo linking uses Fixes owner/repo#100; to close several issues, repeat the full syntax for each — Resolves #10, resolves #123. You can also link manually from the Development section in the PR or issue sidebar, up to ten issues per PR.

Now assemble the whole convention this week has been building: one issue → one branch named for it → one PR whose description says Closes #N → merge closes the issue. The board updates itself; your backlog's state is derived from what actually merged, not from anyone remembering to tidy it.

Hands-on checklist

Run the full issue-to-close loop once in your course-project repo, by hand (M23 delegates it to Claude — do it manually first so you can review what Claude later drafts):

Self-check

1. Which issue body is closest to agent-ready?

C. End-to-end behavior from the user's perspective, verifiable criteria, explicit blocking state. A specifies the how — file paths and function names that will be stale guesses by the time an agent picks it up; the agent should explore fresh. B's spec lives somewhere the fresh session can't see. D is a horizontal layer with unbounded scope — M21's subject.

2. Your PR description says “This closes the import issue (#12).” The PR merges to main — but issue #12 stays open. Most likely explanation?

B. The linking syntax is KEYWORD immediately followed by the issue reference — Closes #12. Words in between break the pattern, so GitHub never linked the pair, and merge had nothing to auto-close. (You can still link manually from the sidebar's Development section.) A and C are false; D is wrong — descriptions are a standard place for closing keywords.

3. Why does the course's issue format ban file paths but its acceptance criteria demand millisecond-level specifics like “reveals the answer within 100 ms”? Isn't that inconsistent precision?

D. Precision about what and silence about how are one principle, not a contradiction: outcomes are checked at review time and don't rot; implementation details written at decomposition time are guesses that do. This is Week 3's destination-not-blueprint rule (M16) applied at issue scale. B is false — agents read files by path constantly; the problem is the paths being wrong later.

4. What does a ready-for-agent label assert that an unlabeled, well-written issue doesn't?

A. The label turns a judgment (“I believe this is fully specified”) into a visible, filterable state — you or an agent can pull the frontier of startable work with one query. An unlabeled issue might be equally good, but nothing asserts it. B inverts who writes vs. who reads; C confuses readiness with priority — a ready issue can still be sequenced last (M22).
