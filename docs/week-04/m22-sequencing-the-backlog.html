Sequence the Backlog to De-Risk

Ten well-cut issues can still be a bad backlog if they're ordered wrong. Sequencing is a judgment call: which slice, done first, would teach you the most about whether your design survives contact with reality? This lesson gives you the criteria — and practice making the call.

M22 · Planning Bloom: Evaluate Assessed by: Assignment A4

Objective: Sequence and prioritize an issue backlog (Kanban) so that early slices de-risk the architecture.

1 · The backlog as a Kanban board

In Pocock's seven-phase workflow, the Kanban board is phase 5 — between the PRD and execution. His board is not a fancy tool requirement; it's a shape of information: a list of tickets, each declaring which tickets block it. GitHub issues with blocked-by lines (M20/M21) already are one; GitHub Projects, Linear, or three columns of sticky notes just render it.

The blocking edges are what make a backlog more useful than a plan document. The to-tickets skill publishes tickets blockers-first, so every edge can reference a real issue number, and then defines the key concept:

The frontier (mattpocock/skills, to-tickets)
Work the frontier: any ticket whose blockers are all done.
For a purely linear chain that means top to bottom.

The frontier is the set of issues startable right now. A linear chain has a one-issue frontier — fine for working solo. A backlog whose independent slices carry no false edges has a wide frontier — which is what lets multiple agent sessions work in parallel later (Week 8's subject). Sequencing decisions you make now determine how wide that frontier can get.

2 · What “de-risk” means

Not all slices carry equal information. The tracer-bullets essay's core promise is that early slices let you “identify issues early, and proceed with confidence that foundational approaches actually work” — if the early slices are the ones that test foundational approaches. A slice de-risks the architecture to the degree that its failure would force a redesign. So the sequencing question for every issue is:

“If this slice fails, how much of the design do I have to change?” — schedule the biggest answers first.

  • Riskiest assumptions first. Unproven integrations, novel data models, the API you've never used — put a thin slice through each one early. If localStorage can't hold your data shape, you want to know in slice 2, not slice 9.
  • The walking skeleton is always slice 1. Deployed shell, thinnest full path, observable in a browser. Everything else builds on a proven pipeline.
  • Prefactoring before the slices it enables — by construction, it's a blocker.
  • Low-information slices last. Polish, secondary CRUD, cosmetic features: if their failure changes nothing about the design, they can't de-risk anything — their position in the sequence is a free choice, so spend it late.
The demo-first temptation

The tempting order is the impressive order: build the flashiest feature first. But demo value and information value are different currencies. A gorgeous card-flip animation on slice 1 proves nothing about whether review scheduling — the app's actual reason to exist — can be computed from your data model. Sequence by what failure would teach, not by what success would show.

3 · Priority is not readiness, and neither is order

Three properties of an issue that are easy to conflate, kept straight:

PropertyQuestion it answersWhere it lives
ReadinessIs this specified well enough to start?ready-for-agent label (M20)
BlockednessMay this start now?Blocked-by edges; the frontier is where they're all resolved
PriorityOf the startable issues, which should go first?Your sequencing judgment — the subject of this lesson

Blocking edges are facts about the work; priority is a decision about risk. Keep them separate: don't add a fake blocked-by edge to encode “I want this done first” — false edges narrow the frontier and serialize work that could have run in parallel. Order within the frontier is what you control by choosing what to pick up (or hand to an agent) next.

One more consequence of sequencing well, from the seven-phases workflow: the board is not write-once. QA (phase 7) generates new tickets, and “you'll iterate through phases 5–7 multiple times.” A backlog whose early slices de-risked the design absorbs new tickets cheaply — they're additions, not rework.

4 · Worked judgment: sequencing the flashcards backlog

Take M21's seven issues. The blocking edges force a partial order (1 before 2 before 3; 7 after 3) — but they leave real decisions open. Two of them, worked:

Decision 1: After #3 (add cards), the frontier holds #4 (flip) and #7 (delete). Which first? Flip. Its failure would implicate the card data model and per-card state handling — foundations five other slices sit on. Deletion's failure would implicate almost nothing. Same frontier, different information value.

Decision 2: A classmate proposes moving #6 (review scheduling) to slice 2, arguing it's the riskiest logic. The instinct is right — scheduling is the app's riskiest assumption — but the move breaks a genuine dependency: scheduling consumes marked-answer data that doesn't exist until #5. The de-risking move that respects the edges: keep the chain but thin every intermediate slice to the minimum that feeds #6, so the riskiest logic is reached in the fewest possible sessions. Sequencing operates within the dependency structure, not against it.

Self-check

Evaluate-level: each question asks you to judge a sequencing decision, then compare your rationale with the model answer.

1. A course project's PRD hinges on calling an external dictionary API the student has never used. Their backlog defers the API slice to #8 of 9, “because the UI issues are easier to start with.” Judge the sequencing.

C. “If this slice fails, how much design changes?” — for the API slice the answer is “most of it,” which is precisely the profile that belongs early. Easy-first (A) optimizes comfort, not information; the momentum it builds is on a foundation that hasn't been checked. D is refuted by the asymmetry: total effort is order-independent, but the cost of discovering a failure grows with everything built on top of it.

2. To make sure the walking skeleton is done first, a student adds “Blocked by #1” to all eight other issues — including three that don't touch anything #1 produces. Judge it.

B. Priority is a decision; edges are facts — encoding one as the other corrupts the frontier, the very thing that lets independent slices run in parallel (and that Week 8 will exploit with multiple sessions). The skeleton being genuinely foundational means most issues legitimately depend on it — but the three that don't are exactly the parallelism the false edges destroy (D's “harmless” is measurably false). Order-within-the-frontier was already yours to choose.

3. Two startable slices: (i) a settings page for font size — cosmetic, nothing depends on it; (ii) “due today” queue computation — novel logic that three later slices consume. The student picks the settings page “to bank an easy win before the hard part.” Judge it.

D. Both slices are startable; only one is informative. The de-risking criterion says schedule by what failure would teach — the queue's failure forces redesign, the settings page's failure forces nothing. B mistakes visibility for risk (the page is visible but its design is settled); A's pacing argument prices the student's mood but not the compounding cost of deferred discovery. Low-information slices are exactly the ones whose position is free — so spend them where they cost nothing: last.

4. Mid-project, QA on slice 4 reveals card data needs a schema change. A new ticket is filed. The student is discouraged: “my sequencing failed — the plan changed.” Evaluate.

A. The success criterion for sequencing isn't “the plan survived unchanged” — it's “surprises arrived while they were cheap.” Pocock's workflow explicitly loops: QA generates tickets, and you iterate through Kanban–execution–QA multiple times. B describes a board that produced no learning (or was never tested); C protects the artifact by discarding its purpose; D is exactly backwards — the cost of a schema change is proportional to how much has been built on the schema, which is what the ordering minimized.
