Six Failure Modes of AI-Generated Code

M32 told you when to look at a diff; this lesson tells you what to look for. AI-generated code doesn't fail randomly — it fails in recognizable, recurring patterns, each with a cause rooted in how models generate code. Learn the six and diff review stops being “read everything and hope” and becomes a targeted scan. This taxonomy is the checklist A6 asks you to review your partner's PR against.

M33 · Review Bloom: Analyze Assessed by: Assignment A6

Objective: Identify common AI code failure modes in a real diff: hallucinated imports, redundant logic, architectural drift, security gaps, style drift, happy-path-only error handling.

1 · The taxonomy

Each failure mode below has three parts worth memorizing together: what it is, why models produce it (the cause tells you where to look), and how to spot it in a diff.

Failure modeWhat it isWhy it happensSpot it by
1 · Hallucinated imports“Referencing packages or modules that don't exist in your project”Models generate plausible-seeming dependency references without verifying they're availableScanning every import line in the diff; verifying each package against package.json / the dependency manifest
2 · Redundant logic“Duplicating existing functionality instead of reusing it” — e.g. a new auth handler when one existsThe agent lacks full-codebase context and regenerates solutions already presentComparing each new function against the existing codebase; watching for near-duplicate patterns
3 · Architectural drift“Adding new patterns instead of following existing ones, or restructuring code in ways that conflict” — e.g. introducing MVC into a functional codebaseModels default to common patterns rather than learning project-specific conventionsReviewing file organization, module structure, and design patterns against the project's established ones
4 · Security gaps“Hardcoded credentials, missing input validation, or overly permissive permissions”Models prioritize functionality over security practiceClose reading of anything touching authentication, credential handling, or permission scoping
5 · Style drift“Ignoring your project's conventions for naming, error handling, or file structure” — e.g. camelCase in a snake_case projectTraining data reflects countless styles; without project guidelines, the agent picks oneChecking naming, error patterns, and structure against neighboring existing code
6 · Happy-path-only error handling“Happy-path code that crashes on edge cases” — e.g. processing an array without validating length or typeModels focus on the success scenario, not the failure conditionsHunting for missing try/catch, null checks, and edge-case handling around every new code path
The causes cluster into two families

Modes 2, 3, and 5 share a cause: the agent doesn't know your project — it regenerates, restructures, or restyles because your conventions weren't in its context. (This is why Week 2's CLAUDE.md and Week 5's explore-first discipline are preventions for this family.) Modes 1, 4, and 6 share a different cause: plausibility over verification — the code looks like working code because looking-like is what generation optimizes. Prevention can shrink the first family; only review catches the second reliably.

2 · Where to aim first

The source's review guidance concentrates attention on “areas where agents are most likely to make mistakes: imports, error handling, and anything that touches security or authentication” — modes 1, 6, and 4, the plausibility family. Those are also the three that hurt most in production, which is not a coincidence: they're the ones neither the compiler nor a casual skim reliably flags.

The condensed checklist, verbatim from the source — this is the “failure-mode checklist” A6 names:

The A6 review checklist

“Check that imports resolve, new code doesn't duplicate existing functionality, credentials aren't hardcoded, error handling covers failure cases, style matches your project, tests still pass, and the agent only changed what was asked.”

Note the last two items reach outside this lesson's taxonomy: “tests still pass” is M32's checkpoint 4, and “only changed what was asked” is checkpoint 1's scope bound. The checklist is the whole week in one sentence.

3 · Worked analysis: one diff, several failures

Your flashcards project (Python, snake_case, has utils/validation.py with a working validate_deck(), credentials via environment variables). You asked Claude to “add a JSON deck-import endpoint.” The diff includes this new file:

import_deck.py (new file, excerpt) 
import json
import flask
from deck_tools import parse_manifest        # ← line A

API_KEY = "sk-flashcards-prod-8f3ab2"        # ← line B

def importDeck(request):                     # ← line C
    payload = json.loads(request.data)       # ← line D
    for card in payload["cards"]:            # ← line D
        front, back = card["front"], card["back"]
        save_card(front, back)

def check_deck_format(payload):              # ← line E
    return "cards" in payload and isinstance(payload["cards"], list)

Walk the taxonomy against the marked lines:

  • Line A — hallucinated import. There is no deck_tools package in this project or its manifest. The import reads plausibly — that's the failure mode's signature. Verification is mechanical: is it in the dependency manifest or the repo? No → finding.
  • Line B — security gap. A hardcoded credential, in a project whose convention is environment variables. Two findings in one line, actually: the secret itself, and the ignored convention.
  • Line C — style drift. importDeck is camelCase in a snake_case codebase. Trivial to fix, but its real value to a reviewer is as a signal: the agent wasn't imitating neighboring code, so read the rest with that prior.
  • Line D — happy-path-only. json.loads on raw request data with no error handling, then unguarded key access. Malformed JSON or a missing "cards" key crashes the endpoint. The success path is complete; the failure paths don't exist.
  • Line E — redundant logic. check_deck_format() re-implements a narrower version of the existing validate_deck(). It's not wrong — it's duplicate, and it will drift from the real validator over time. You only catch this one by knowing (or grepping) the codebase; nothing in the diff itself is suspicious.

One diff, five of six modes. What it's missing is architectural drift — and noticing an absence is part of the skill: this file follows the project's flat-module structure, so mode 3 gets a clean pass. A finding list that says “checked, clean” for some modes is more trustworthy than one that only lists hits.

Self-check

Analyze-level: each question gives you diff evidence; classify the failure mode or judge the reasoning.

1. A diff adds from retry_helpers import with_backoff. Your reviewer instinct fires. What's the fastest conclusive check, and which failure mode is it testing for?

C. Hallucinated imports are the one mode whose detection needs zero judgment: the package is in the manifest/repo or it isn't. That's why “scan the import lines” is the recommended first pass — maximum findings per minute. D often would catch it, but slower and only if tests exercise that module; the point of the taxonomy is knowing the direct check. A and B test real modes but not the one an import line raises.

2. In a codebase of plain functions and modules, the agent's diff introduces a BaseImporter abstract class, a factory, and a registry/ directory to support the one importer you asked for. Every line is individually well-written. Classify it.

A. This is the mode's textbook form — “adding new patterns instead of following existing ones” — and its documented cause: models default to patterns common in training data (OO scaffolding) over project-specific convention. It's caught by reviewing structure against the project, never by line-reading, which is why it slips past reviewers who only read hunks. B requires an existing counterpart; there isn't one. C trivializes it — this is structure, not naming. D is the trap: quality of lines and fitness of shape are independent axes.

3. Why does the taxonomy's cause column matter to a reviewer, not just the failure list itself?

D. The cause column converts a flat checklist into a strategy: it explains the source's advice to concentrate on imports, error handling, and security/auth (the plausibility family — nothing upstream reliably stops them), and it connects Week 2 and Week 5's disciplines to review as prevention for the context family. B draws the wrong inference — causes tell you what review must still catch, not what review can skip. C isn't a claim the taxonomy makes; human code fails too, just in different patterns.

4. From the worked diff: a classmate's review flagged lines A–D but called check_deck_format() (line E) “fine — it's correct and tested.” What did their review process miss, structurally?

A is the seduction, B the answer. Redundant logic is the mode whose evidence lives outside the diff: nothing in check_deck_format() is suspicious; the finding is the existence of validate_deck() elsewhere. The documented detection — “compare new functions against the existing codebase” — is a different physical action than reading hunks, which is why a hunks-only reviewer misses it every time while feeling thorough. C: linters check style, not duplication-of-intent. D misclassifies; validation weakness would be mode 6's territory, and the deeper issue is the duplicate will drift from the maintained validator.
