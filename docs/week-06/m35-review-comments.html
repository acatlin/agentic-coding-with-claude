Writing Actionable PR Review Comments

In A6, your review comments have an unusual property: they will be executed. Your partner reads them, then directs Claude to address them — so every comment you write is, in effect, a prompt with an extra hop in the middle. A comment that would make a human colleague walk over and ask “what did you mean?” makes an agent guess. This lesson gives you a five-part shape for comments that survive the hop, and the drafting discipline to produce them.

M35 · Collaboration Bloom: Create Assessed by: Assignment A6 (pair)

Objective: Write structured, actionable PR review comments that a peer (or an agent) can act on without further clarification.

1 · The standard: no round trip

“Actionable without further clarification” is a testable property. Hand the comment to someone who wasn't in your head, and ask: can they (a) find the code you mean, (b) understand what's wrong with it, (c) know what change would satisfy you, and (d) tell how urgent it is? Fail any one and you've bought a round trip — a clarifying question, a wrong guess, or in the agent case, a plausible-looking fix to a problem you didn't raise.

You've now studied two reviewers that meet this standard, and both are templates. The machine reviewer's findings (M34) each carry a location, a severity, a one-sentence issue, and expandable reasoning — that anatomy is why they're act-on-able. And the review platforms you've used anchor feedback the same way: comments attached “to specific lines,” submitted as one batch, feeding an agent-iteration loop. Your handwritten comments should clear the bar the machine's do.

2 · The five-part comment

Every substantive comment in your A6 review carries these parts. Small comments compress parts into single clauses; none get dropped.

PartWhat it doesFailure when missing
1 · AnchorPins the comment to the exact place: inline on the line(s) in the PR, naming the file/function if the fix spans more than the anchored line“There's a bug in the validation somewhere” — the reader searches, and may fix a different spot
2 · ObservationStates what the code does, as a checkable claim — cite the behavior, not your reaction to it“This looks wrong” — nothing to verify, nothing to dispute
3 · ConsequenceSays why it matters: the input or state that produces the wrong outcomeThe author can't rank it, so it's triaged by tone instead of impact
4 · RequestNames the change that would resolve the comment — concrete enough to act on, without dictating every keystroke“Please improve this” — the fix satisfies the fixer, not the reviewer
5 · SeverityBlocking or not — borrow M34's vocabulary: 🔴 must fix before merge, 🟡 nit, non-blockingNits get treated as blockers and blockers as nits; the merge decision loses its evidence

Assembled, on a line from M33's worked diff:

inline comment on import_deck.py, line 9 (the json.loads call) 
🔴 [blocking] `json.loads(request.data)` is called with no error handling,
and line 10 accesses `payload["cards"]` unguarded. A request with malformed
JSON, or valid JSON without a "cards" key, crashes the endpoint with a 500.
Wrap the parse and key access, and return a 400 with a message for both
cases — `validate_deck()` in utils/validation.py already covers the
missing-key check, so reuse it rather than re-checking here.

Anchor (inline, plus the spillover line named), observation (no error handling — checkable), consequence (the exact inputs that 500), request (400s for both cases, via the existing validator), severity (🔴 blocking). An agent can act on this without asking anything; so could a colleague in another timezone. Note the request also heads off an M33 failure mode — pointing at validate_deck() prevents the fix from introducing redundant logic.

Hold yourself to the machine's verification bar

The Code Review docs let repos demand that “behavior claims need a file:line citation in the source, not an inference from naming” — because uncited claims cost “the author a round trip.” Apply that bar to your own part 2: if you can't cite the line that misbehaves, you have a suspicion, not an observation. Say so honestly: “Question (non-blocking): does save_card handle duplicates? I couldn't find where that's checked.” A labeled question is actionable; an unverified accusation isn't.

3 · Review-level structure

A set of good comments still needs an envelope — the author decides what to do first from the review's shape:

  • Batch, don't dribble. Submit comments as one review, not one-at-a-time — the platform-review model you saw in M33's source (“submit batched feedback for agent iteration”) exists because the author's fix session wants the whole list at once. For A6 this is literal: your partner will paste your review into one Claude session.
  • Open with the tally. The docs' summary-shape advice: lead with the shape of the work — “2 blocking, 3 nits” — and when there's nothing blocking, say that first. The author wants to know the shape before the details.
  • Cap your nits. A review that's 80% style comments buries its one real finding. State the pattern once (“naming: camelCase in several new functions — matching snake_case throughout; flagging the first instance only”) instead of anchoring twelve copies.
  • Verified vs. relayed. If a comment originates from your /code-review run, you verified it before forwarding (M34's protocol) — so present it as yours. Anything you didn't verify stays a labeled question.

4 · Create it: drafting exercise

Your review of a partner's PR surfaced three raw findings. Draft each as a five-part comment before revealing the model version — this is the Create-level work, and the reveal is a comparison, not an answer key.

  1. The new exportDeck function builds its CSV by string concatenation; card text containing commas or quotes will produce a corrupt file. There's a csv module in the standard library.
  2. The PR renames deck.py to deck_manager.py. Nothing in the task required it; ten files' imports changed as a result.
  3. You suspect the new import endpoint doesn't check deck size limits, but you haven't found where limits are enforced for other endpoints either.

Compare your drafts: which set below handles all three correctly?

A. (1) has all five parts: anchor, checkable observation, the exact corrupting inputs, a concrete request, severity. (2) is the M32 move — an unrequested rename is a scope finding; its existence is the observation, and “revert or split” is the standard request for it. (3) is the verification bar honestly applied: you have a suspicion, so it ships as a labeled non-blocking question — which is still actionable, because “answer this” is an action. B fails part 2 twice (“seems fragile” is a reaction) and violates the bar in (3) by asserting “missing” without a citation. C erases the severity signal — when everything blocks, nothing does. D over-corrects: the fix for an unverified claim is labeling, not silence; your unasked question becomes the author's production incident.

Self-check

1. Which comment is actionable without clarification, per the four-question test?

C. Findable (line 42, retry()), checkable (retries on every exception — go look), consequential (the 10s user-facing delay), requested (narrow to TransientError, with an in-repo precedent cited), ranked (🟡 non-blocking). B looks precise but fails question (b) — no observation or consequence means the author can't tell why, so they can't generalize the fix or push back. A fails (a), (c), (d) at once. D anchors nothing and requests archaeology; “read my branch” is a round trip by construction.

2. Why does the “agent will execute this” framing raise the bar on part 4 (the request) specifically?

B. The failure mode shifts from “question” to “confident wrong fix” — Week 6's whole lesson in miniature, since you then have to review the misfix. That's also why part 2's checkable observation matters doubly: it's the acceptance criterion the author's agent session can verify against. A overshoots — the request names the destination (“return 400 for both cases”), not every keystroke; dictating the patch wastes the agent's ability to fit the fix to the codebase. D confuses actionable structure with a file format.

3. Your review found one likely-serious concurrency issue and fourteen naming inconsistencies. How do you structure the submitted review?

D. Three review-level rules at once: batch (one review, one fix session), tally first (the author learns “one thing really matters” before reading anything), cap nits (state the pattern once — fourteen anchors for one pattern is noise wearing precision's clothes). A is technically complete and practically unreadable — the 🔴 is comment nine of fifteen. B discards true information; non-blocking ≠ not worth saying. C inverts the priorities: the blocking finding is precisely the one that can't wait, and “once I'm certain” is what the labeled-question form is for.

4. Your partner responds to your 🔴 CSV comment: their Claude session swapped in the csv module, tests pass, thread resolved. A6 grades your role as reviewer. What's left for you to do?

A. A well-formed comment contains its own acceptance test — the consequence clause names the inputs that failed, so verifying means checking exactly those. This closes A6's loop: review given, response received, resolution verified — and the fix is itself AI-generated code, so M33's taxonomy applies to it too (the classic miss: the fix handles commas but the new csv import shadows an existing symbol). B abandons the reviewer role one step early; the machine reviewer you studied auto-resolves threads only after re-checking the push, and you're held to the same standard. C over-scopes — the delta since your review is the review surface. D verifies a claim nobody disputed; green tests were never the observation.
