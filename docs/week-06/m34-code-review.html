Running /code-review and Interpreting Findings

You've now built the manual review skill: checkpoints, diff views, a failure-mode checklist. This lesson adds the machine reviewer — /code-review runs a multi-agent analysis of your diff and reports findings with severity tags. The Apply skill has two halves: invoking it on the right scope, and reading its output the way you'd read any reviewer's — findings are claims to evaluate, not orders to follow.

M34 · Review Bloom: Apply Assessed by: Assignment A6

Objective: Use /code-review on a PR and interpret multi-agent findings, distinguishing verified issues from noise.

1 · Running it locally

In any Claude Code session, /code-review reviews a diff right in your terminal — no setup, no GitHub App. It reports correctness bugs plus reuse, simplification, and efficiency cleanups. What it reviews:

  • Default scope: your branch's commits ahead of its upstream, plus any uncommitted changes in the working tree — i.e., “the work on this branch,” which is usually what you want before opening a PR.
  • Explicit targets: pass a file path, a PR number (the A6 case: reviewing your partner's PR), a branch name, or a ref range like main...my-feature — the ref-range form reviews the committed diff that a PR from my-feature into main would contain.
  • Output flags: --comment posts the findings as inline PR comments; --fix applies the findings to your working tree after the review.

Two knobs shape the findings themselves. The command follows your CLAUDE.md like any session, so your project conventions inform what gets flagged. And effort level trades coverage against confidence: lower effort levels “return fewer, higher-confidence findings, while high through max give broader coverage and may include uncertain findings.” That sentence matters for interpretation — at high effort, the tool is telling you some findings may be noise.

One name, two products

The same docs page describes the managed Code Review service: a GitHub App (Team/Enterprise research preview) that reviews PRs automatically and posts inline comments. The mechanism below is shared; setup, @claude review triggers, and REVIEW.md customization belong to the managed service. For A6 you'll use the local command — but read the finding format from the managed docs, because that's the interpretive skill both share. (There's also /code-review ultra, which runs a deeper cloud review; it needs claude.ai auth and falls back to a local review when unavailable.)

2 · What “multi-agent” buys: the verification step

When a review runs, “multiple agents analyze the diff and surrounding code in parallel… Each agent looks for a different class of issue, then a verification step checks candidates against actual code behavior to filter out false positives. The results are deduplicated, ranked by severity,” and reported. Read that pipeline as three review properties:

  • Specialized finders — parallel agents each hunting one class of issue (compare your M33 taxonomy: a per-mode scan, mechanized).
  • Adversarial verification — candidate findings are checked against what the code actually does before being reported. A finding that survives is a verified claim, not a pattern match.
  • Dedup and ranking — so one root cause doesn't arrive as five comments, and severity ordering tells you where to start.

Findings come tagged with severity, and the tags are load-bearing:

MarkerSeverityMeaning
🔴ImportantA bug that should be fixed before merging
🟡NitA minor issue, worth fixing but not blocking
🟣Pre-existingA bug that exists in the codebase but was not introduced by this PR

Each finding also carries a collapsible extended reasoning section — “why Claude flagged the issue and how it verified the problem.” And deliberately, findings “don't approve or block your PR”: the managed check run always completes neutral. The reviewer advises; the human merges. That's Week 1's whose-decision line, drawn in infrastructure.

3 · Interpreting findings: verified vs. noise

“Distinguishing verified issues from noise” is a reading protocol, not a vibe. For each finding, in order:

  1. Severity first. Triage 🔴 before 🟡; treat 🟣 as information about the codebase, not about this PR (it's real, but it isn't your partner's diff's fault — file an issue rather than blocking the merge).
  2. Open the extended reasoning. The verification evidence is the difference between “a model said so” and “here's the code path that breaks.” A finding whose reasoning cites specific lines and traces actual behavior is verified; one that reasons from a name or a resemblance (“this function is probably called with null”) is a candidate the verifier should have caught — treat it as noise until you confirm it yourself.
  3. Reproduce the Important ones. For a 🔴, the reasoning should let you construct the failing case. If you can't, that's your answer.
  4. Close the loop. On the managed service every comment carries 👍/👎 buttons — rate findings (👍 useful, 👎 wrong or noisy; reactions tune the reviewer and don't trigger re-reviews). To act on a finding, fix the code and push — replying to the comment doesn't prompt Claude to respond.
Two symmetric failure postures

Accept-everything turns the tool into an authority and imports its false positives straight into your codebase — worst with --fix, which applies findings you haven't read. Dismiss-as-noise spends the tool's $0 of your attention and misses the verified 🔴 sitting in position one. The protocol above is the middle: the machine proposes with evidence; you judge on the evidence. Your M33 taxonomy is the judging instrument — most findings land in a mode you can now evaluate.

4 · Where it sits in your A6 workflow

For the pair exercise: run /code-review <PR#> on your partner's PR after your own manual pass against the M33 checklist, not before. Two reasons. Anchoring: read the machine's findings first and you'll review its list instead of the diff. Coverage: the modes the machine hunts overlap yours but aren't identical — your knowledge of the project's intent (checkpoint 1's scope, the PRD) is evidence no reviewing agent has. Then merge the two lists: machine findings you verified, plus your own findings, become the structured comments M35 teaches you to write.

Self-check

1. Your partner's PR is #14. You want the machine review before writing your comments, without touching your own working tree's state. Which invocation matches the documented behavior?

B. The default scope is defined relative to your branch and working tree; someone else's PR needs the explicit target (PR number, branch, or ref range). C does the opposite of “without touching state” — --fix applies findings to your working tree, and for A6 the whole point is that the author directs the fixes after reading your comments. D confuses the two products: the GitHub App is the managed service; the local command needs no installation.

2. A review returns a 🟣 finding: a race condition in session.ts, in code your partner's PR never touched. How does a good reviewer handle it?

C. The severity taxonomy encodes attribution, not just size: 🟣 answers “whose diff caused this?” with “not this one.” Blocking your partner's merge on it punishes them for the machine's wider gaze (A), while discarding it throws away a verified finding (B) — the Week 4 move is a tracer-bullet issue in the backlog. D treats the reviewer as a slot machine; the tag reflects git history, which re-rolling won't change.

3. Two 🔴 findings. #1's reasoning: “parseExpiry returns 0 on malformed input (line 88), and line 142's refresh path treats 0 as ‘never expires’ — a token with a corrupted expiry stays valid forever.” #2's reasoning: “deleteAccount is a destructive operation and likely lacks confirmation-flow handling.” Apply the protocol.

A. Same severity tag, opposite evidentiary standing — which is exactly why the protocol says open the reasoning rather than trust the marker. #1 has the shape verification produces: lines, mechanism, consequence. #2 has the shape a finder produces before verification — an inference from naming that the pipeline's verifier is supposed to filter and sometimes doesn't. B forwards the machine's noise under your name (M35 will hold you to better). C invents a priority rule from topic rather than evidence. D over-corrects: the reasoning sections exist precisely so review effort can be targeted.

4. A teammate: “I run /code-review at max effort and auto-apply with --fix. More findings caught, zero review time — strictly better.” What does the documentation say that breaks this?

D. Each flag is fine alone: max effort with a human filter is broad coverage; --fix on a low-effort, high-confidence run is a reasonable shortcut. Together they wire the noisiest output directly to your files. The deeper error is the metric — “zero review time” counts the reviewing you skipped as savings, when unreviewed machine edits are unreviewed edits (M36: verification proportional to supervision — auto-applied fixes got none). B and C are invented restrictions.
