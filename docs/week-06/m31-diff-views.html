Agentic Coding with Claude Week 6 · Objective M31

Reading /diff: Full-Diff and Per-Turn Views

Week 5 ended every workflow the same way: review the diff before you commit. This lesson gives that step its instrument. /diff opens an interactive panel inside Claude Code with two views of what changed — everything uncommitted, and each conversation turn's changes replayed one at a time. The habit this week builds: after every task, /diff before commit, both views, no exceptions.

M31 · Review Bloom: Apply Assessed by: Assignment A6

Objective: Run /diff after a task and read both the full-diff and per-turn views before committing.

1 · Why a diff panel inside the session

You already know git diff HEAD from the terminal. What /diff adds is where and when the review happens: an interactive panel right in the session, at the moment the task finishes — no switching to another terminal, no losing the conversation that produced the changes. After Claude modifies several files in a run, one command puts every change in front of you before you decide anything about committing.

That placement matters because the review this course keeps asking for is review against the conversation: did the changes match what you asked for in each turn? A raw git diff can't answer that — it has no idea where turn 2 ended and turn 3 began. /diff can, because it has the conversation.

2 · The two views, two sources of truth

ViewWhere the data comes fromWhat it shows
Current (full diff)Git — the result of git diff HEADAll uncommitted changes in the working directory, whoever made them
Per-turnThe conversation itself — reconstructed from each turn's tool-call recordsWhat Claude changed in each individual turn, in order

The distinction to internalize is the data source. The current view is a git perspective: it runs real git commands under the hood (with safeguards — files over 1 MB are skipped, changes beyond 400 lines are truncated, and a 5-second timeout protects the session). The per-turn view doesn't consult git at all: it rebuilds each turn's edits from the conversation's tool-call records, so it is, in the author's words, “unaffected by git state — even if you manually modified files or ran git add in between.”

Why two sources instead of one

Each view answers a question the other can't. Current: “what exactly will this commit contain?” — including your own hand edits. Per-turn: “what did the agent do at each step, and did each step match what I asked?” — even after you've edited or staged files and blurred the git picture. Full-diff review without per-turn review misses how the change was built; per-turn without full-diff misses what the commit will actually say.

3 · Driving the panel

  • Arrow keys switch between the two sources and move through the file list.
  • Enter opens the detailed diff for a file — additions in green, removals in red, with word-level highlighting so a one-token change doesn't hide inside a rewritten line.
  • Escape closes the panel and returns you to the conversation.

Reading order that works: current view first, file list only — is the set of touched files what you expected? Then per-turn, turn by turn, pairing each turn's diff with what you asked in that turn. Then back to any file whose changes surprised you, Enter, and read the lines.

4 · The habit, and what per-turn replay buys you

The source post's recommendation is one sentence long: “After every task Claude Code completes, run /diff before committing.” Not for big tasks; for every task. The author frames it as plain “good engineering,” resting on the principle this whole week orbits: “understanding the changes is what it means to truly own your code” — M36 takes that sentence apart in detail.

Per-turn replay is the underrated half of the habit. When a run took eight turns and touched five files, the full diff shows you the destination but not the route. Stepping through per-turn diffs — ideally alongside the conversation history — lets you replay the agent's approach: which turn introduced the helper, which turn's fix caused the follow-up fix, where a course correction actually landed. Next week's failure-mode hunting (M33) gets much easier when you can see which turn each suspicious change arrived in.

The habit fails silently

Nothing forces you to run /diff. Skipping it doesn't error; it just converts “I reviewed the changes” into “the changes looked done.” A6 grades the review you actually performed — your partner's PR will contain things only a deliberate diff read catches.

Hands-on: both views on a real task

Self-check

1. Mid-task, you hand-fixed a typo in utils.py and ran git add on it. The task finishes and you run /diff. What do the two views show for that typo fix?

B. This is the data-source distinction doing work. Current = git diff HEAD, which reports all uncommitted changes regardless of author or staging. Per-turn = the conversation's tool-call records, which contain only what Claude did. C is a common git misread — staged changes are still uncommitted and still in diff HEAD. A and D each collapse the two sources into one.

2. An eight-turn run produced a correct-looking full diff, but one file contains a helper you never asked for. Which /diff move most directly tells you where it came from?

C. Attribution is exactly what per-turn replay is for: it localizes a change to the turn that made it, and pairing it with the conversation shows why the agent added it. A keeps staring at the view that can't answer “when.” B looks at commits — these changes aren't committed yet. D is the anti-habit: the review point is before the commit, not after the incident.

3. A classmate says: “I skim the full diff when the task was big, and skip /diff for small tasks — a one-file change doesn't need a panel.” What does the assigned workflow actually prescribe?

D. “After every task Claude Code completes, run /diff before committing” has no size clause. The reason isn't ritual: the tasks where the diff surprises you are precisely the ones you didn't predict — you can't pre-classify them as “small.” A confuses review depth (which M36 does scale) with whether the look happens at all. C substitutes behavior verification for change review; Week 6's point is you need both.

4. In the current view you notice a 600-line generated file shows only part of its changes. What's the right interpretation?

A. The current view runs real git commands with guardrails: >1 MB files skipped, >400-line changes truncated, 5-second timeout. The panel is a review instrument, not the only one — when you hit its limits, the underlying git command has no such cap. B is exactly the unsafe assumption truncation invites. D isn't guaranteed — per-turn shows what Claude changed per turn, not a fuller render of a giant diff.
