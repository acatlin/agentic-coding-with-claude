The Four-Checkpoint Review Workflow

“Stop trying to read every line an AI agent writes. Use four checkpoints instead.” That's the whole thesis. Agents produce code faster than you can read it — so instead of one heroic line-by-line review at the end, you place four cheap gates around the run: bound the scope before, approve tool calls during, gate on the diff after, and verify with tests before merge. Each gate catches what the previous one structurally can't.

M32 · Review Bloom: Apply Assessed by: Quiz 6 · Assignment A6

Objective: Apply the four-checkpoint review workflow: scope bounds before, approval gates during, diff gate after, tests before merge.

1 · Checkpoint 1 — Scope bounds, before the run

Before the agent touches anything, declare what it may touch. Two mechanisms you already own from earlier weeks:

  • In the prompt: explicit scope directives — which files and directories are allowed, which are forbidden, and standing prohibitions like “no dependency changes, no removing exports, no edits outside src/flashcards/.”
  • In CLAUDE.md: the scope rules that apply to every run in this repo, so you stop retyping them (Week 2's lean-CLAUDE.md discipline, now doing review work).

Why this is a review checkpoint and not just prompting hygiene: declared scope converts diff reading from an open-ended search into a checklist. “If the diff shows edits outside the declared scope, that's an immediate flag” — you don't have to judge whether the out-of-scope edit is good code; its existence is the finding. Out-of-scope changes are the hardest class to catch later, because nothing in the diff marks them as unrequested except your memory of the request.

2 · Checkpoint 2 — Approval gates, during the run

This is Week 5's default permission mode reframed as a review instrument: the agent pauses before each consequential tool call — file writes, bash commands, deletions — and waits for you. Two things make the gate more than a click-through:

  • Approve with a comment. Approving a file write with “use the existing parseDate utility instead” steers the run without aborting it — mid-flight course correction with the session's context intact (M30's toolkit, applied at the gate).
  • Know your red flags. Deny-and-ask when you see: file writes outside the declared scope, package installations or git commits you didn't request, unexpected deletions or directory reorganizations.

The author's claim for this gate: “This workflow guarantees me being in the loop, fully understanding the changes, spotting issues early.” The cost is interruptions — which is why M26 taught you to loosen this gate deliberately (acceptEdits, auto) when other gates are strong. The checkpoints trade off: less supervision during the run means the diff gate and the test gate carry more weight, which is M36's proportionality argument.

Checkpoints 1+2 are why you can skip line-by-line reading

The workflow's speed doesn't come from reviewing less — it comes from having already reviewed scope and high-impact operations by the time the diff exists. What's left for the diff gate is a bounded question, not an archaeology dig.

3 · Checkpoint 3 — The diff gate, after every session

Immediately after the agent finishes: git diff HEAD — or M31's /diff panel, which renders the same git perspective in-session. The gate is a structured two-minute pass answering four questions, in order:

  1. Did the agent stay within declared scope? (Checkpoint 1 wrote this checklist for you.)
  2. Are there unexpected structural changes? New files, moved directories, reorganized modules you didn't ask for.
  3. Does the logic match expectations? Skim the substantive hunks against what you asked.
  4. Is the diff size reasonable for the task? “More than 200 lines for a ‘small fix’ is a warning sign.”

Work coarse-to-fine with git's own tools: git diff HEAD --stat first for the file-level shape, git diff HEAD -- src/auth/ to scope into a directory that concerns you, git diff HEAD --word-diff for small targeted changes. And when the gate fails, recovery is cheap because nothing is committed yet: git checkout -- <file> reverts one file, git restore . reverts everything.

4 · Checkpoint 4 — Tests, before merge

The last gate turns the test suite into the specification, not just an afterthought. The pattern:

  1. Confirm tests exist and pass before the agent session — a red baseline makes “did the agent break it” unanswerable.
  2. Run the agent against that documented expected behavior.
  3. Verify tests still pass after completion.
  4. Review test changes as carefully as implementation changes.
  5. Run the full suite, catching regressions in adjacent modules the diff gate didn't flag.

Step 4 is the one that separates this gate from “CI is green.” The documented pitfalls are all test-shaped: tests that verify the agent's implementation rather than the intended behavior; new code that arrived with no tests at all; mocked functions creating false confidence. An agent that edits a failing assertion until it passes produces the same green checkmark as an agent that fixed the bug — only reading the test diff tells you which happened. Tests are still “the fastest path to behavioral confidence,” but only if the tests themselves survived review.

Hands-on: run all four gates once

Self-check

1. You asked for a “small fix” to a date-formatting bug. The diff gate shows 340 changed lines across six files, including a new utils/dates/ directory. Tests are green. What does the workflow say?

C. The gate's questions are answerable from the --stat view alone — that's what makes it a two-minute pass. A treats the checkpoints as substitutes when they're serial: green tests don't validate scope (nothing tested the new directory's necessity). B is the line-by-line reading the workflow exists to replace — the coarse signals already made the decision. D commits changes that failed the gate, converting a free git restore into future rework.

2. Why does the workflow insist scope bounds are declared before the run, when the diff gate could catch out-of-scope edits after?

B. The pre-declaration doesn't prevent drift by magic — agents can still edit the wrong file. It changes the review economics: detection becomes mechanical (“is this path in the allowed list?”), which is why out-of-scope changes go from hardest-to-catch to easiest. A is false; the diff shows everything that changed. D inverts the design — the checkpoints are complementary, each catching what the others structurally miss.

3. During a run, Claude requests approval to run npm install left-pad. Your task was “fix the header alignment in the settings page.” Which checkpoint is firing, and what's the right move?

D. Package installs, unrequested git commits, and unexpected deletions are the named deny-and-ask triggers for the approval gate — high-impact operations whose whole point of surfacing mid-run is that they're cheapest to stop there. A and B defer to later gates that handle this class worse: a dependency change survives the diff gate looking like one lockfile hunk, and tests won't flag an extra package at all. C is the tenth-approval click-through reflex this gate degrades into when you stop reading.

4. A classmate proposes: “Checkpoint 4 makes 1–3 redundant — if the full suite passes, the code is right, so scope bounds and diff reading are ceremony.” Evaluate against the documented pitfalls.

A. A green suite certifies “the code satisfies the tests as they now exist” — and the agent may have edited the tests. The classmate's argument silently assumes the specification survived the run unmodified, which is precisely what checkpoint 4's step-4 review checks. B and D also miss that entire finding classes (scope, structure, dependencies) live outside behavior. C concedes the conclusion for the wrong reason — the argument fails even with excellent tests.
