Agentic Coding with Claude Week 1 · Objective M03 · Lesson 2 of 2

First Project: Repo + /init

Point Claude Code at a real Git repository, let it introduce itself to your project, and generate the CLAUDE.md file that every future session will read.

M03 · Setup Bloom: Apply Assessed by: Assignment A1

Objective: Install Claude Code, authenticate with a Claude Pro account, and initialize a project with /init inside a Git repository. Lesson 1 covered install + authenticate; this lesson gets you initialized in a real repo.

Why a Git repository first?

Assignment A1 says: create a GitHub repository, then run /init. The order isn't arbitrary. Claude Code is built to work with git — the docs list “your git state: current branch, uncommitted changes, and recent commit history” among the things Claude can see. A repo gives your session three things:

  • A boundary. Running claude in the repo directory scopes the agent to your project.
  • A safety net beyond checkpoints. Claude's file snapshots help mid-session, but the docs are explicit that checkpointing “isn't a replacement for git.” Commits are your durable undo.
  • An audit trail. The course grades process artifacts — commits are evidence of how you worked.

If you haven't made your GitHub repository yet, do objective M06 first and come back — this lesson assumes you have a cloned repo on disk.

Step 1 — Start a session inside the repo

$ cd path/to/your-repo
$ claude

You'll see the Claude Code prompt with the version, current model, and working directory shown above it — confirm the directory is your repo, not your home folder. (An agent pointed at the wrong directory is the Week 1 version of a context problem; Week 2 makes this a theme.)

Step 2 — Let Claude read the project

Before initializing anything, try the Quickstart's opening move — ask questions:

> what does this project do?
> explain the folder structure

Claude analyzes your files and answers. The docs make the key point: “Claude Code reads your project files as needed. You don't have to manually add context.” Even in a nearly-empty starter repo this is worth 60 seconds — you're watching the gather-context phase of the agentic loop run on your own code.

Step 3 — Run /init

> /init

What it does, per the best-practices guide: /init analyzes your codebase — detecting build systems, test frameworks, and code patterns — and generates a starter CLAUDE.md file, giving you “a solid foundation to refine.”

And what is CLAUDE.md? The docs' definition: “a special file that Claude reads at the start of every conversation,” holding things like build commands, code style, and workflow rules — “persistent context it can't infer from code alone.” Every session you start in this repo, forever, begins by reading this file.

Don't over-invest yet

In a brand-new repo, /init will produce something short — there's little to detect. That's fine. Week 2 (objective M09) is entirely about crafting a lean CLAUDE.md and justifying every line. Today's goal is just that the file exists, you've read it, and you understand when it gets loaded.

Step 4 — Review and commit it

Open the generated CLAUDE.md and read every line — Week 1's habit-in-training is review what the agent produced before you keep it. Delete anything that's wrong or obvious. Then make it permanent; you can even have Claude do the commit:

> commit CLAUDE.md with a descriptive message

The best-practices guide recommends checking CLAUDE.md into git “so your team can contribute” — in this course, so your grader can see it. You'll approve a permission prompt for the git command: that's the permission system doing its job on a state-changing action.

Sanity check

Start a fresh session (/exit, then claude again) and run /context — the best-practices guide uses this to confirm CLAUDE.md actually loaded. Seeing your file in the context listing closes the loop on what “read at the start of every conversation” means. If anything about your setup feels off, /doctor runs a configuration checkup.

Hands-on checklist

Do this in your Assignment A1 repository:

Self-check

1. What does /init actually do?

C. Per the docs, /init analyzes your codebase (build systems, test frameworks, patterns) and generates a starter CLAUDE.md to refine. Git init is git's job (A), and clearing context is /clear (D).

2. When does Claude read your CLAUDE.md?

B. The docs define it as “a special file that Claude reads at the start of every conversation” — that's what makes it the right home for persistent, always-relevant project rules, and why Week 2 cares so much about keeping it lean.

3. Why does the course insist Claude Code sessions happen inside a Git repository, even though checkpoints can already undo Claude's edits?

D. The docs warn that checkpointing only tracks Claude's file edits and “isn't a replacement for git.” Commits survive anything, cover everything, and double as the process evidence this course grades.
