GitHub Basics: Repo, Clone, Commit

Create a repository on GitHub, get it onto your machine, and make a first commit with a message a human (and an agent) can understand.

M06 · Git basics Bloom: Apply Assessed by: Assignment A1

Objective: Create a GitHub repository, clone it locally, and make a first commit with a well-formed commit message.

How much Git does this course need?

Uneven Git backgrounds are expected and fine. Week 1 requires only clone, commit, and push — exactly this page. Branches, issues, and pull requests get their own dedicated week (Week 4), and nothing beyond branch/PR/issue mechanics is ever required. If you want extra practice, GitHub's free interactive courses at skills.github.com are the recommended optional remediation.

The 60-second mental model

  • A repository (repo) is a project folder whose entire history Git tracks.
  • GitHub hosts a copy of your repo online; your laptop holds a clone — a full local copy linked to it.
  • A commit is a saved snapshot of your changes with a message explaining them.
  • Push uploads your local commits to GitHub; that's how graders (and teammates) see your work.

For this course, the repo is more than storage: it's the workspace Claude Code operates in, the safety net beyond checkpoints, and the audit trail your process is graded on.

Step 1 — Create the repository on GitHub

Following GitHub's Hello World guide (sign in at github.com first):

  1. Click the + icon in the upper-right corner of any GitHub page and select New repository.
  2. Name it something meaningful — for A1, e.g. wordcount-cli (lowercase, hyphens, no spaces).
  3. Add a brief description.
  4. Choose Public or Private — for coursework, either works as long as your grader has access; check the assignment instructions.
  5. Check Add a README file — this gives the repo an initial commit so it clones cleanly.
  6. Click Create repository.

Step 2 — Clone it to your machine

On the repository page, click the green Code button and copy the HTTPS URL. Then in your terminal:

$ git clone https://github.com/YOUR-USERNAME/wordcount-cli.git
$ cd wordcount-cli

The first time you push, GitHub will ask you to authenticate — modern Git for Windows/macOS pops up a browser login and remembers it. Now this folder is where you run claude, per objective M03.

Step 3 — Make a change and commit it

Edit README.md — add a sentence about what the project will be. Then the three-command rhythm you'll use hundreds of times:

$ git status                        # what changed?
$ git add README.md                 # stage it
$ git commit -m "Describe the wordcount CLI project in README"

What makes a commit message “well-formed”?

GitHub's guide says a commit message should “briefly describe the change” — the widely-followed conventions sharpen that:

  • Summary line ≈ 50 characters, capitalized, no trailing period.
  • Imperative mood — write what the commit does: “Add input validation,” not “Added…” or “misc fixes.”
  • Say what and why, not how — the diff already shows how.

Why agents care about your history

Commit messages aren't just for humans anymore. Claude Code reads your recent commit history as part of its project context, and the best-practices guide suggests prompts like “look through the git history and summarize how this API came to be.” A well-formed history is context you get for free in every future session — written once, read by every agent and grader after.

Step 4 — Push

$ git push

Refresh the repository page on GitHub: your commit and updated README should be there. That round trip — edit locally, commit, push, verify on GitHub — is the full Week 1 Git loop. (You can also do all of this conversationally: objective M04 showed Claude committing for you. The mechanics underneath are identical, which is why you learn them by hand once.)

Hands-on checklist

This is Assignment A1's backbone — do it for real:

Self-check

1. Your teammate edited files on their laptop, but the changes don't appear on GitHub. Assuming they committed, what step is missing?

B. Commits are local snapshots; git push is what uploads them to GitHub. Clone copies down, push sends up, status just reports.

2. Which is the best-formed commit message for a change that makes the CLI print an error for missing files?

C. Imperative, specific, about 45 characters, and meaningful without opening the diff. (B) has the right content but belongs in the message body, not a run-on summary line; (A) and (D) tell a future reader — or agent — nothing.

3. Beyond backup, why does a clean commit history particularly matter when you work with an agent?

D. The docs list git state — including recent commit history — among what Claude can access, and best practices recommends pointing Claude at history to answer “why” questions. Every well-formed message is future context for free.

4. Why did you check “Add a README file” when creating the repository?

A. A repo needs at least one commit to have a branch you can clone and build on; initializing with a README provides it (and A1 requires a README anyway). Neither GitHub nor Claude Code mandates one.
