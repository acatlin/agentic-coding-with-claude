Agentic Coding with Claude Week 1 · Objective M04

Core Session Controls

Prompting, interrupting, getting help, resetting, and — most importantly — reviewing what changed before you commit. The five controls you'll use in every session for the rest of the course.

M04 · Setup Bloom: Apply Assessed by: Assignment A1

Objective: Use core session controls: submitting a prompt, interrupting, /help, /clear, and reviewing what Claude changed before committing.

1 · Submitting a prompt

Type what you want and press Enter. The docs' advice on what to type boils down to two points:

  • Talk like you'd talk to a colleague. “Talk to Claude like you would a helpful colleague. Describe what you want to achieve, and it will help you get there.” No magic phrasing required.
  • Specific beats vague. The Quickstart's own example: not “fix the bug” but “fix the login bug where users see a blank screen after entering wrong credentials.” Precision up front saves corrections later.
  • It's a conversation, not a slot machine: when the first attempt isn't right, you refine — “That's not quite right. The issue is in the session handling.” — rather than starting over.

Two input helpers worth knowing from day one: type / to see all available commands, and press ↑ for prompt history.

2 · Interrupting

You are part of the agentic loop, and interrupting is how you exercise that. The docs give you two distinct moves:

  • Press Esc to stop. Claude halts immediately; the running tool call is canceled; context is preserved; Claude waits for your next instruction.
  • Type a correction and press Enter — without stopping. Claude reads your message as soon as the current action completes and adjusts course. Use this when the work is going mostly right and just needs steering.

There's also the bigger hammer: press Esc Esc (or run /rewind) to open the rewind menu and restore the conversation and your files to an earlier checkpoint. Week 2 practices this deliberately (objective M11); today, just know it exists — it's why you can let an agent try things without fear.

3 · /help and friends

The Quickstart's essential session commands:

CommandWhat it does
/helpShow available commands
/clearClear conversation history
/exit (or Ctrl+D twice)Exit Claude Code

And from your shell: claude starts a session, claude -c continues your most recent conversation in this directory, claude -r lets you pick an older one to resume. One more help channel the docs highlight: Claude Code can teach you how to use it — just ask it things like “how do I set up hooks?” in plain language.

4 · /clear: the reset button

Every message, every file read, and every command output accumulates in the session's context window — and the best-practices guide is blunt about the consequence: long sessions full of irrelevant context “can reduce performance and sometimes distract Claude.” The fix is a habit, not a feature: run /clear between unrelated tasks.

Preview of Week 2

The guide names the failure mode “the kitchen-sink session”: one task, then an unrelated question, then back again, until the context is mud. Week 2 builds your full mental model of this (the “smart zone”); for now, adopt the rule mechanically — new task, fresh /clear.

5 · Reviewing changes before committing

This is the control that separates using AI labor from merely consuming it. When Claude edits files in default mode, it shows you each proposed change and asks approval. But the review that matters most comes at the end, before anything is committed. Two conversational moves from the Quickstart:

> what files have I changed?
> review my changes and suggest improvements

And the classic, always available in another terminal (or via Claude running it for you):

$ git diff

Read the whole diff. Ask about any line you don't understand — “why did you change this?” is a perfectly good prompt. Only then:

> commit my changes with a descriptive message
The course policy behind the habit

This course replaces “did you write this yourself?” with “can you account for it?” — you must be able to explain any line in your repository. The diff review before every commit is where that accountability is earned. Week 6 (“Owning the Diff”) turns it into a full discipline; Assignment A1's reflection already depends on you having watched what changed.

Hands-on checklist

Run one throwaway session in your A1 repo and exercise every control:

Self-check

1. Claude is mid-task and mostly on the right track, but you realize it should target the staging config, not production. The least disruptive correct move is:

B. Typing without Esc sends a steering message Claude reads as soon as the current action completes — no work lost, course corrected. Esc (stop) is for when the direction is wrong; rewind and /clear are for when the session state is wrong.

2. You finish fixing a parser bug and want to ask Claude about an unrelated CSS layout problem. Per the best-practices guide, you should first:

C. Unrelated leftover context (parser files, test output) doesn't help the CSS task — it dilutes attention and can degrade output. “Use /clear frequently between tasks” is the guide's literal advice, and Week 2 explains the mechanism.

3. Claude reports “Done! All tests pass” after modifying four files. Per this course's working discipline, what comes before the commit?

D. Tests check behavior the tests cover; your review covers everything else — and the course's accountability policy (“can you account for it?”) requires you to be able to explain every line you commit. Review the diff, then commit.

4. What's the difference between pressing Esc once and pressing Esc twice?

A. Single Esc is the interrupt: stop now, context preserved, awaiting instructions. Double Esc (or /rewind) opens checkpoints — restore conversation, code, or both to a previous point. Exit is /exit or Ctrl+D twice.
