Tools & Permissions

The loop acts on your machine through tools — and a permission system decides which actions need your sign-off.

M01 · Foundations Bloom: Understand Assessed by: Quiz 1

Objective: Explain the agentic loop and how Claude Code uses tools to read files, run commands, and edit code. Lesson 1 covered the loop; this lesson covers the tools that power it and the permissions that keep you in control.

Tools are what make it agentic

The official docs put it plainly: “Without tools, Claude can only respond with text. With tools, Claude can act.” Every step of the agentic loop — gathering context, taking action, verifying — happens through a tool call, and every tool call returns information that feeds back into the loop.

The built-in tools fall into five categories:

CategoryWhat Claude can do with itLoop phase it usually serves
File operationsRead files, edit code, create new files, rename and reorganizeContext + action
SearchFind files by pattern, search content with regex, explore codebasesContext
ExecutionRun shell commands, start servers, run tests, use gitAction + verify
WebSearch the web, fetch documentation, look up error messagesContext
Code intelligenceSee type errors after edits, jump to definitions, find references (via plugins)Verify

Claude chooses which tools to use based on your prompt and what it learns along the way — you never call a tool yourself. Beyond these five, there are orchestration tools (spawning subagents, asking you questions); those arrive in Week 8.

What Claude can access

When you run claude in a project directory, the tools give Claude access to:

  • Your project — files in that directory and its subdirectories (elsewhere only with your permission)
  • Your terminal — any command you could run: build tools, git, package managers, scripts
  • Your git state — current branch, uncommitted changes, recent history
  • Your CLAUDE.md and memory — persistent project instructions (the star of Week 2)

This whole-project access is precisely what separates Claude Code from inline autocomplete that sees only the current file — a distinction objective M02 examines closely.

Permissions: agency with a leash

An agent that can run any command you could run needs guardrails. Claude Code's rule of thumb: reading is free, changing things needs permission. By default, Claude Code requests approval for actions that might modify your system — file edits, shell commands, and the like — while reads and searches proceed freely.

How often you're asked depends on your permission mode. Press Shift+Tab in a session to cycle through them:

ModeBehaviorWhen it fits
Manual (default)Claude asks before file edits and shell commandsLearning the tool (this course's default), unfamiliar or risky work
Accept editsFile edits and common filesystem commands auto-approve; other commands still askYou trust the direction and want fewer interruptions
PlanClaude explores and proposes a plan without editing your source filesScoping work before committing to it (formalized in Week 5)
AutoA background safety check evaluates each action and blocks risky onesLonger unattended runs (Week 8 territory)

You can also allowlist specific trusted commands (like npm test or git status) in .claude/settings.json so Claude never has to ask about those. The best-practices guide frames the trade-off honestly: default prompting is safe but tedious — “after the tenth approval you're not really reviewing anymore, you're just clicking through” — so the goal is to reduce interruptions without giving up control where it matters.

Safety net: checkpoints

Permissions control what Claude may do; checkpoints let you undo what it did. Before editing a file, Claude Code snapshots it, and pressing Esc Esc rewinds to a previous state. But checkpoints only cover Claude's file edits — actions with external side effects (databases, deployments, pushed commits) can't be rewound, which is exactly why commands get the permission prompt.

Course connection

On a Pro budget with short scoped sessions, the default Manual mode is your friend for Weeks 1–4: every prompt is a chance to see what the agent intends before it acts. Choosing permission modes deliberately is itself a mastery objective in Week 5 (M26).

Self-check

Click an answer to check it.

1. In a default-mode session, which of these will Claude Code typically do without asking you first?

B. Reading and searching gather context and don't modify your system, so they proceed freely. Edits, deletions, and shell commands can change things, so default (Manual) mode asks first.

2. You want Claude to investigate a bug and propose an approach, but you're not ready to let it touch any source files. Which permission mode fits best?

C. Plan mode is built for exactly this: Claude explores and proposes a plan without editing source files. Option D sort of works but relies on instructions instead of an enforced mode — the mode is the guarantee.

3. Claude Code asks permission before running git push, but pressing Esc-Esc can undo a file edit without asking anyone. What explains the difference?

D. Claude snapshots files before editing them, so edits can be rewound locally. Actions that affect remote systems can't be checkpointed — which is why commands with external side effects sit behind the permission prompt.

4. Your test command npm test is safe and you run it constantly, but Claude asks for approval every time. What's the intended remedy?

A. Allowlists exist precisely for trusted, frequently-used commands: approve once in settings, and Claude stops asking — while everything else still prompts. Mode changes (B) are a blunter instrument than a one-command allowlist.

5. Which statement about tools is closest to how the documentation describes Claude Code?

C. Tool choice is the model's job, made step by step: “Each tool use gives Claude new information that informs the next step. This is the agentic loop in action.” Without tools there is no agent — just text.
