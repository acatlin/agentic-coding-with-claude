Agentic Coding with Claude Week 1 · Objective M02

What Is AI Labor?

Autocomplete suggests, chat advises, agents work. Understanding the difference — and how planning and execution decisions divide between you and the agent — is the intellectual core of this course.

M02 · Foundations Bloom: Analyze Assessed by: Quiz 1 · A1 reflection

Objective: Distinguish agentic coding from chat-based code assistance and from autocomplete, and describe what “AI labor” means as a division of planning vs. execution decisions.

Three ways AI can help you code

By 2026 you've likely met all three of these. They look similar — “AI writes code” — but they occupy different positions in your workflow, and confusing them leads to using each one badly.

AutocompleteChat assistanceAgentic coding
What it sees The current file (roughly), your cursor position What you paste or describe Your whole project: files, terminal, git state
What it produces The next few tokens or lines, inline Text: explanations, snippets, advice Completed work: edited files, run tests, commits
Who acts on the world You (accept/reject each suggestion) You (apply the advice by hand) The agent, with your permission
Who verifies You, continuously You, after applying The agent first (tests, builds), then you review
Unit of interaction A keystroke A question A task

The documentation draws the same line: because Claude Code sees your whole project, it can “work across it” — search for relevant files, make coordinated edits across several of them, run tests, and commit. “This is different from inline code assistants that only see the current file.” And unlike a chatbot that answers and waits, an agentic tool “can read your files, run commands, make changes, and autonomously work through problems while you watch, redirect, or step away entirely.”

“AI labor”: a division of decisions

This course's name for what you're buying with that autonomy is AI labor: software work performed by an agentic tool under human direction. The emphasis matters. AI labor is not “the AI does everything” — it's a specific division of decisions, and Anthropic's research measured exactly where the line falls in practice.

The research post “How Claude Code is used in practice” analyzed roughly 400,000 real sessions and found a consistent split:

  • Humans make the planning decisions. Users made roughly 70% of the “what to do” determinations — which feature to build, which bug matters, what done looks like.
  • Agents make the execution decisions. Claude handled about 80% of the “how to do it” choices — which files to read, what order to work in, how to implement.
  • The leverage is real. In typical sessions Claude performed around 10 actions per user instruction, sometimes more than 100.
The slogan, and what it hides

“Humans plan, agents execute” is the study's headline — but note it describes proportions, not a wall. You still make some execution decisions (you'll interrupt and redirect), and Claude contributes to planning (it proposes approaches). The study found that the more decision authority users kept, the fewer actions Claude took per instruction (8 vs. 16) — the division is a dial you control, not a fixed contract.

Expertise is the multiplier

The study's most important finding for you as a student: what you know determines what the agent can do for you.

  • Users with domain expertise achieved 28–33% verified success on their tasks; novices, 15%.
  • Expert users triggered 2.4× more agent actions and about 5× more output per instruction — the researchers put it as: “the more domain expertise a person brings to a session, the more work Claude does per instruction.”
  • Novices abandoned troubled sessions 19% of the time versus 5–7% for others — expertise is what lets you recover when a session goes sideways.
  • Strikingly, professionals from non-software fields succeeded within 5–7 percentage points of software engineers — domain understanding mattered more than coding background.

The researchers' conclusion: “coding agents are not substituting for domain expertise — the greater understanding a worker brings to an agent, the higher quality work the agent performs.” That is why this course spends ten weeks on your skills — specification, direction, review — rather than on prompting tricks.

For your A1 reflection

Assignment A1 asks you to identify which decisions in your first session were yours and which were Claude's. The planning/execution frame is your instrument: for each decision, ask “was this a what/whether decision or a how decision — and who made it?” Objective M05 gives you a full worked example.

Analysis exercise: classify the tool

Before the self-check, test the distinction on borderline cases. For each scenario below, decide which category the interaction belongs to — then reveal the analysis.

Scenario 1. You paste a 40-line function into a web chat, ask “why does this throw on empty input?”, read the explanation, and edit your file accordingly. Which category was this?

B. The diagnostic quality doesn't matter — what matters is the division of labor. The AI saw only what you pasted, produced only words, and you performed every action and all verification. That's chat assistance, however smart it is.

Scenario 2. You type “fix the flaky test in CI” into Claude Code and answer two permission prompts while it runs the suite, reads logs, edits a test helper, and reruns to green. You then review the diff. Which category — and who made the planning decision?

C. The agent acted on the world (ran, read, edited, verified) — agentic coding. But the what came from you: you decided the flaky test was the thing worth fixing. Claude's decisions — where to look, what to change, when to rerun — were all how decisions. This is the 70/80 split in miniature.

Scenario 3. As you type for (const user of, your editor ghosts in the rest of the loop body and you press Tab. Which category?

A. Volume of code isn't the test. The interaction unit is the keystroke, the AI sees roughly the current file, and you accept or reject each suggestion in-line. No task was delegated and nothing was verified by the AI.

Self-check

1. Which finding comes from Anthropic's study of real Claude Code sessions?

D. That's the study's core division-of-labor finding. (B) is backwards — domain expertise, not coding background, drove success; (C) is off by an order of magnitude (~10 actions per instruction).

2. A teammate says: “AI labor means the agent replaces the developer.” Based on the research findings, what's the strongest correction?

B. The study concluded agentic coding “amplifies rather than replaces expertise”: the more understanding the human brings, the more and better work the agent performs. Execution gets cheaper; judgment gets more valuable.

3. What most fundamentally separates agentic coding from chat-based assistance?

C. Same models can power both. The difference is the harness: tools plus project access mean the agent performs the work and verifies it, rather than advising you while you do the work.

4. The study found managers and other non-engineers succeeded nearly as often as software engineers at coding tasks. What's the best explanation offered?

A. With execution delegated, what's left is knowing what to build and recognizing whether it's right — domain judgment and specification clarity. The study even notes managers' slight edge may reflect delegation skill. That's exactly the skill set this course trains.
