The Agentic Loop

How Claude Code actually works: a repeating cycle of gathering context, taking action, and verifying results — with you in the loop.

M01 · Foundations Bloom: Understand Assessed by: Quiz 1

Objective: Explain the agentic loop (gather context → take action → verify → repeat) and how Claude Code uses tools to read files, run commands, and edit code. This lesson covers the loop; Lesson 2 covers the tools and permissions that power it.

A different kind of assistant

You have probably used an AI chat assistant for code: you paste a snippet, it replies with text, and then you do everything else — open the file, apply the change, run the tests, read the error, paste the error back. The assistant only ever produces words. You are the one acting on the world.

Claude Code is different in exactly one fundamental way: it can act. It runs in your terminal with access to your project, and it can read your files, search your codebase, edit code, and run commands — the same things you would do by hand. Anthropic's documentation calls this an agentic coding tool: you describe an outcome, and the agent works toward it, step by step, checking its own progress as it goes.

The three phases

When you give Claude Code a task, it works through three phases, over and over, until the task is done. The official docs name them like this:

1 · Gather context → 2 · Take action → 3 · Verify results ↺

1. Gather context

Before changing anything, Claude figures out what it needs to know. It might search for files matching a pattern, read the relevant source, check your git state, or look at your CLAUDE.md project notes. You don't have to paste code in — Claude reads your project files as needed.

2. Take action

Then it acts: editing a file, creating a new one, running a build, executing a git command. Each action is performed through a tool (the subject of the next lesson), and each action's result comes back to Claude as new information.

3. Verify results

Finally, it checks its own work — rerunning the failing test, rebuilding the project, re-reading the file it just changed. If the check fails, that failure becomes new context, and the loop begins again: gather, act, verify.

Key idea

The phases blend together and repeat. A question about your codebase might need only context-gathering. A bug fix cycles through all three phases repeatedly. Claude decides what each step requires based on what it learned from the previous step — the docs describe it chaining dozens of actions together and course-correcting along the way.

Watch the loop run: “fix the failing tests”

Here is the worked example straight from the official documentation. You type one sentence — fix the failing tests — and Claude might:

  1. Run the test suite to see what's failing (action that gathers context)
  2. Read the error output (context)
  3. Search for the relevant source files (context)
  4. Read those files to understand the code (context)
  5. Edit the files to fix the issue (action)
  6. Run the tests again to verify (verify — and if they still fail, loop back to step 2)

Notice what you did not have to do: name the failing test, locate the file, describe the fix, or apply it. Each tool use gave Claude new information that informed the next step. That is the agentic loop in action.

You are part of the loop

“Agentic” does not mean “unattended.” The docs are explicit: you can interrupt at any point to steer Claude in a different direction, provide additional context, or ask it to try a different approach. Claude works autonomously but stays responsive to your input. In Week 5 you'll formalize this into a deliberate workflow (plan first, then execute); for now, just know the loop has a fourth, human, participant — you.

Model + harness = agent

One more piece of vocabulary that Quiz 1 may probe. The loop is powered by two components:

  • The model — a Claude language model (Sonnet by default in this course) does the reasoning: understanding code, deciding what needs to change, choosing the next step. When docs say “Claude decides,” it's the model deciding.
  • The tools — file operations, search, command execution, and more, which let those decisions touch the real world.

Claude Code itself is the agentic harness around the model: it provides the tools, context management, and execution environment that, in the docs' words, “turn a language model into a capable coding agent.” Same model, no harness → a chatbot. Model + harness → an agent.

Why this matters for the course

Everything in the next nine weeks is about managing this loop well: feeding it good context (Week 2), pointing it at a well-specified destination (Weeks 3–4), letting it verify against real tests (Week 5), and reviewing what it did (Week 6). If you can explain the loop, the rest of the course is refinements to it.

Self-check

In the style of Quiz 1: application and discrimination, not recall. Click an answer to check it.

1. You ask Claude Code to “make the header sticky.” Before editing anything, it searches the codebase for CSS files and reads two components. Which phase of the agentic loop is this?

C. Reading and searching before changing anything is context-gathering. Tool use happens in every phase — what distinguishes the phases is the purpose: learning (context), changing (action), or checking (verify).

2. Claude Code edits a function, then immediately reruns the test suite, sees a new failure, and reads the error output before editing again. What does this sequence best illustrate?

B. This is the loop working as designed: verify produced a failure, the failure became new context, and the cycle began again. Each tool result feeds back into the loop and informs the next decision.

3. What turns a Claude language model into a coding agent?

D. The docs call Claude Code the agentic harness around Claude: the model reasons, and the harness supplies the tools and environment that let reasoning become action. Without tools, Claude can only respond with text.

4. Which task would most likely run the agentic loop through only its first phase?

A. The docs note the loop adapts to the ask: “a question about your codebase might only need context gathering.” The other three all require edits (action) and checks (verify).
